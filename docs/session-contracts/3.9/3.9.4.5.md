## Session 3.9.4.5 — Secrets & bootstrap runbook

**Sub-version:** 3.9.4.5 (Phase 4 — Continuity & recovery; one session)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.4.5
**UX gate:** No

## 1. Objective

From zero — new machine, no local state — the operator can reconstruct a
working development environment and, if needed, production configuration,
because every secret's *location and re-obtainment path* is inventoried.

## 2. Current context and dependencies

- **Depends on 3.9.4.2** so the runbook lands in a backed-up home.

## 3. Done conditions

- One runbook covers: every env-registry key with where its value lives
  (Vercel env, the CCP developer application, Neon, Convex, Upstash, Better
  Auth secret, service secrets) and how to regenerate or re-obtain each; the
  CCP dev-app registration's role as the scope ceiling (and that its settings
  are console-state, not code); the local bring-up sequence cross-checked
  against the agent guide; and the rotation posture (which secrets can rotate
  without user impact, which force re-auth).
- No secret **values** anywhere in it.
- A dry-run walk of the dev-bootstrap half validates the doc.

## 4. In scope

The inventory/runbook + the dev-half validation.

## 5. Out of scope

- Actual rotations (report staleness findings).
- Storing values.

## 6. Hard constraints

- Arc-wide (Phase 4): secrets never enter any continuity mechanism — the
  runbook records *where* each secret lives and how to re-obtain it, never
  values; written runbook + validation result in SCRATCHPAD; operator-mode
  steps explicit.
- The env inventory derives from the typed registry in `src/lib/env.ts` (one
  home), cross-checked with the 3.9.1.3 env-example checker where it exists.

## 7. Decisions the session plan must resolve

- Runbook structure (per-service vs. per-key inventory).
- The dev dry-run method (how "from zero" is simulated without disturbing the
  live environment).

## 8. Acceptance criteria

- Every env-registry key appears with location + re-obtainment path.
- The rotation posture is stated per secret.
- The dev-bootstrap dry-run completes from the runbook alone; gaps found are
  fixed in the doc.
- A scan confirms no secret values in the runbook.

## 9. Verification

- The committed runbook (in the backed-up workspace home) + the dev dry-run
  readout in SCRATCHPAD.
- Changelog entry + APP_VERSION bump (3.9.4.5) for tracked changes; otherwise
  the non-code completion shape.

## 10. UX/operator gates

**UX gate: No.** Operator involvement: console access for verifying
re-obtainment paths.

## 11. Baseline/hotspot boundary

Neutral. No hotspot contact. No-campaign decision preserved.

## 12. Close-out behavior
