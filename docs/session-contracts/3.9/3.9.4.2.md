## Session 3.9.4.2 — Workspace continuity (mechanism + restore drill)

**Sub-version:** 3.9.4.2 (Phase 4 — Continuity & recovery; one session)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.4.2
**UX gate:** No

## 1. Objective

The gitignored workspace — `docs/`, agent guides, both skill trees,
`.agent-local/`, memory files — and the Document Archive survive machine loss,
deletion, and corruption, verifiably.

## 2. Current context and dependencies

- No dependencies. 3.9.4.5 (secrets runbook) follows this slice so its
  runbook lands in a backed-up home.
- **Exposure:** the entire document-driven lifecycle lives in gitignored paths
  on one machine whose only replication is iCloud *sync* — and sync is not
  backup: a deletion or corruption propagates.

## 3. Done conditions

- A mechanism is chosen and running (decision surfaced to Ryan with the
  trade-offs): the recommended primary is a **private git repository** for the
  workspace + archive (text-shaped, versioned, off-device,
  deletion-recoverable, and both runtimes already hold `gh` auth), with a
  scheduled one-way snapshot copy to a separate directory as belt-and-braces;
  a second live-synced iCloud folder alone is insufficient.
- Secrets and env files are categorically excluded (3.9.4.5 owns those).
- The drill: from a clean checkout/simulated-new-machine, restore the
  workspace and demonstrate `resolve_development_state.py` +
  `check_agent_drift.py` running green — timed and written up as the restore
  runbook.

## 4. In scope

The mechanism, its exclusion rules, the drill, the runbook.

## 5. Out of scope

- The application repo (already on GitHub).
- Secret values.

## 6. Hard constraints

- Arc-wide (Phase 4): drills run against disposable targets; secrets never
  enter the mechanism (exclusion rules are part of done); the drill produces a
  written runbook + a timed result in SCRATCHPAD; operator-mode steps
  explicit.
- The mechanism choice is surfaced to Ryan with trade-offs before it goes
  live (a new private repo is an outward-facing action).

## 7. Decisions the session plan must resolve

- The mechanism decision package for Ryan (private repo layout, snapshot
  cadence, exclusion list).
- The drill's simulated-new-machine method (clean directory vs. second user
  account vs. container).

## 8. Acceptance criteria

- The mechanism is live with one verified backup cycle.
- Exclusion rules provably keep secrets/env files out.
- The timed restore drill passes: resolver + drift check green from the
  restored workspace.

## 9. Verification

- The timed drill write-up + runbook committed to the workspace (and therefore
  itself now backed up).
- SCRATCHPAD records the drill result.
- Changelog entry + APP_VERSION bump (3.9.4.2) for any tracked-file changes;
  drift gate green.

## 10. UX/operator gates

**UX gate: No.** Operator involvement: mechanism approval and any account
creation are Ryan's.

## 11. Baseline/hotspot boundary

Neutral. No hotspot contact. No-campaign decision preserved.

## 12. Close-out behavior

Single-session sub-version: close-out per delivery shape (PR for tracked
