## Session 3.9.2.2.1 — Cron shell & wake classes (shell + batch-route migration)

**Sub-version:** 3.9.2.2 (Phase 2 — Primitive lifecycle; session 1 of 2, one branch)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.2.2
**UX gate:** No

## 1. Objective

A cron route becomes a declaration — name, lock key, wake class, work,
response shape — with one shell primitive owning everything the seven routes
currently repeat. This session delivers the shell, the wake-class model, the
recording policy, the schedule cross-check gate, and migration of the five
batch routes plus the sweeper.

**Verdict & evidence.** *Expand* the existing cron primitives
(`requireCronAuth`, `runCronJob`, `cronLogger`) into one route-shell seam with
a declared **wake class**. Seven cron routes (47–146 LOC) each re-assemble
logger creation, start-time capture, busy/refreshed response construction,
lock-key plumbing, and event emission around the same gate — including clone
`dup:b54bf337` (AF-009 Watch). The clone alone would not justify work (P10)
and is not the driver — the PR #159 idle-wake regression is (see the master
plan §3.9.2.2 for the three-fold `drain-esi-refresh-jobs` failure mode; ~96
Neon wakes/day on an idle deploy). The shell expansion deletes the clone as a
byproduct.

## 2. Current context and dependencies

- 3.9.2.1 preferred first (characterize the seven routes' response contracts
  on the harness where DB-backed).
- Session .2 (drain-cron idle probe) depends on this session's shell and

## 3. Done conditions

- One shell primitive composes auth, wake-class enforcement, advisory-lock
  gating, duration capture, cron telemetry with the noteworthy-gated recording
  policy, and the busy-path response; each route supplies only its declaration
  and its `work`.
- **Wake class, required per cron:** `batch` (daily jobs whose purpose is
  waking Neon) or `idle-silent` (sub-daily watchdogs/drains — a healthy no-op
  run touches zero Neon: no lock, no read, no write). The shell owns the
  ordering guarantee: auth → idle probe → advisory lock → work →
  durable-telemetry gate.
- **Recording policy:** durable telemetry rows only on failure or work-done;
  `always`-record requires a written justification in the declaration.
- **Schedule cross-check gate:** a registry-style test parses `vercel.json`
  crons; any schedule more frequent than daily must map to an `idle-silent`
  declaration or carry a justification string.
- The five batch routes and the sweeper migrate; route-specific behavior (GSC
  quota handling, SDE ingest phases and its pre-lock version gate, prices'
  deliberate lock-free mode) stays visibly route-local — the shell owns the
  *shape* and supports composed pre-lock stages rather than flattening them.
  The sweeper's declaration adopts the existing probe-free idle-silent shape
  (it already touches Neon only when noteworthy).
- `dup:b54bf337` no longer appears in a whole-version Fallow run.

## 4. In scope

The `src/lib/cron.ts` / `src/db/cron-gate.ts` seam, six of the seven
`src/app/api/cron/*/route.ts` files (all but the drain's probe work) and their
tests, the schedule cross-check gate, the restricted-import lint rule, cron
telemetry call sites.

## 5. Out of scope

- Schedule changes (`vercel.json` cadences untouched); new cron jobs.
- The drain-cron idle probe and its Redis signals (session .2).
- The deferred-queue worker's job-processing internals (its route shell
  migrates; per-job logic does not change).
- Event-ledger vocabulary changes.

## 6. Hard constraints

- **Rail (two-part):** (1) the schedule cross-check gate (`vercel.json` ↔
  declarations); (2) restricted-import lint scoped to `src/app/api/cron/**` —
  `requireCronAuth`, `withAdvisoryLock`, `directClient`, and direct
  durable-telemetry writes banned in route files; a cron route reaches them
  only through the shell.
- Full pattern per slice: migrate every consumer in the sub-version; zero
  old-path call sites at branch close; no compatibility façade.
- Protected non-goals off-limits; combine-bias; behavior-preserving migration
  (same responses, same lock keys, same schedules).
