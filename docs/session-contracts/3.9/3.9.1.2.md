## Session 3.9.1.2 — Resolver hardening

**Sub-version:** 3.9.1.2 (Phase 1 — Workflow mechanization; one session)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.1.2
**UX gate:** No

## 1. Objective

The resolver validates the new vocabulary and markers exactly, and gains
optional git awareness, without changing its directive contract.

## 2. Current context and dependencies

- Depends on 3.9.1.1 (vocabulary + marker formats must be specified first).
- The resolver (`.agent-local/resolve_development_state.py`) currently has 13
  passing fixtures in `test_development_state.py`.
- The directive contract (stages, handlers, seven fields) is frozen this
  version — hardening refines validation, never the dispatch model.

## 3. Done conditions

- Terminal-status detection is an exact match against the closed set (a cell
  containing `INCOMPLETE` or `NOT SHIPPED` is nonterminal / flagged), and
  marker values outside their vocabularies are reported as errors with the
  offending file and value.
- The resolver validates the 3.9.1.1 markers where present: a session plan
  missing `Baseline effect`, or a contract missing `UX gate`, is a
  `session-plan-needed`-style reason (legacy pre-3.9 artifacts exempt).
- A `session-ready` directive for a `UX gate: Yes` contract carries that gate
  in its `pause` field.
- An opt-in `--git` flag adds **warnings** (never errors): current branch vs.
  the expected sub-version branch pattern, dirty worktree during a plan-mode
  directive, local main behind `origin/main`. Default behavior without the
  flag is byte-identical to today.
- Fixture coverage extends to every new rule, including the exact-vocabulary
  regressions; the existing 13 fixtures still pass.

## 4. In scope

`resolve_development_state.py`, `test_development_state.py`, the manifest's
`developmentState` schema if fields change, and the doc sentences that
describe resolver behavior.

## 5. Out of scope

- New sibling scripts (3.9.1.3+).
- Changing stage names, handler names, or the directive field set (the
  dispatch contract is frozen).
- Making git checks blocking.

## 6. Hard constraints

- Arc-wide (Phase 1): Python 3 stdlib only; no new runtime dependencies;
  `--check`/`--pretty` conventions; error vs. warn distinction (snapshot-timing
  situations warn, never block); fixture tests follow the
  `test_development_state.py` pattern; no behavior change to application code.
- Standing decision: checkers report; they never mutate artifacts or
  auto-promote findings.
- Legacy exemption: pre-3.9 artifacts are treated as legacy by every new
  validation.

## 7. Decisions the session plan must resolve

- Whether marker-validation failures surface as stage-level reasons or as
  `errors[]` entries.
- The exact branch-pattern expectation (`feat/X.Y.N-*` with named exceptions
  like `codex/`-prefixed tooling branches).

## 8. Acceptance criteria

- Seeded fixture with a nonterminal-looking status cell (`INCOMPLETE`,
  `NOT SHIPPED`) is flagged, never treated as terminal.
- Seeded fixture with an out-of-vocabulary marker reports an error naming file
  and value.
- Missing `Baseline effect` / `UX gate` markers on 3.9-era artifacts produce
  the specified reason; pre-3.9 fixtures are exempt.
- `UX gate: Yes` flows into the directive's `pause`.
- Without `--git`, resolver output on the live repo is byte-identical to the
  pre-change directive (plus any new warnings only under the flag).

- Resolver run against the live repo returns the same directive as before the

- `python3 .agent-local/check_agent_drift.py` green.
