## Session 3.9.2.1 — Real-Postgres test harness promotion (Expand)

**Sub-version:** 3.9.2.1 (Phase 2 — Primitive lifecycle; one session)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.2.1
**UX gate:** No

## 1. Objective

Promote the schema-steered local-Postgres testing pattern from a
coverage-suite helper into the repo's first-class DB test primitive, so
characterization work (this arc's and every future campaign's) stops
hand-assembling setup.

**Verdict & evidence.** *Expand* `src/db/test-support/db-coverage-harness.ts`.
The 3.8.5.x campaigns built five-plus real-Postgres suites (auth migration
25-scenario coverage, retention boundaries, concurrency cases) that each
re-assemble reachability gating, schema steering, seeding, and teardown around
the same harness core.

## 2. Current context and dependencies

- No dependencies. **First in the Phase 2 arc** — later slices' migrations
  lean on cheap characterization.
- Phase 2 may start after 3.9.1.7 lands (preferred ordering), so new code is
  born under the enforced comment standard.

## 3. Done conditions

- One documented harness surface owns: DB-reachability skip-gating,
  disposable-schema lifecycle (create/steer/drop), migration application into
  the schema, seed/fixture helpers for the common row shapes (user + linked
  character + tokens; dataset rows keyed `by_user`), and teardown.
- Existing real-Postgres suites are migrated onto it with zero behavior change
  to what they assert; a new suite needs only its seeds and assertions.
- The testing-policy section of the agent guide names it as the standard.

## 4. In scope

`src/db/test-support/`, the consuming `.db.test.ts` suites, the lint rule
(rail below), agent-guide testing-policy wording.

## 5. Out of scope

- New test *coverage* (no assertion changes, no padding).
- Component/jsdom testing (separate backlog item).
- CI changes.

## 6. Hard constraints

- Arc-wide (Phase 2): full pattern per slice — expand → migrate every consumer
  → enforce by rail in the same sub-version; zero old-path call sites at
  close; no compatibility façade survives the migration.
- **Rail:** restricted-import lint scoped to `*.db.test.ts` — direct
  `postgres()` construction and connection-string literals are banned outside
  `src/db/test-support/`; a suite that hand-rolls schema steering fails lint
  with a pointer to the harness.
- Combine-bias; protected non-goals off-limits (`tree-resolver.ts`,
  `convex/engine.ts`, `src/lib/esi/`); anything campaign-shaped stops and
  reports.
- New shared code lives in `src/db/test-support/`.

## 7. Decisions the session plan must resolve

- Fixture surface breadth — which seed shapes have two-plus real consumers
  today (P4 trims the rest).
- Whether suite migration is one behavior-preserving commit per suite (P9
  default) or batched.

## 8. Acceptance criteria

- All migrated suites green with unchanged assertion counts.
- A grep shows zero suites hand-rolling schema steering outside the harness.
- The rail is demonstrated **red** on a seeded violation (a test constructing
  `postgres()` directly), green on the live tree.
- The agent guide names the harness as the DB-testing standard.

## 9. Verification

- `pnpm verify` + coverage-backed Fallow green.
- Migration grep evidence and the seeded rail-red demonstration in PR notes.
- Changelog entry + APP_VERSION bump (3.9.2.1).
