## Session 3.9.2.2.2 — Drain-cron idle probe (wake policy proven)

**Sub-version:** 3.9.2.2 (Phase 2 — Primitive lifecycle; session 2 of 2, one branch)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.2.2
**UX gate:** No

## 1. Objective

The `drain-esi-refresh-jobs` cron becomes the proving `idle-silent` consumer:
an Upstash Redis pending-work signal lets its healthy no-op runs touch zero
Neon, encoding the PR #159 lesson as an enforced property of the cron
primitive instead of a one-route predicate.

## 2. Current context and dependencies

- Depends on 3.9.2.2.1's shell (same branch).
- The live regression this fixes: the `runCronJob` advisory lock opens the
  direct Neon connection every run; `maybeAlertPublicEsiBudgetExhaustion()`
  runs a Neon telemetry window count every run (~always zero); the drain
  queries the queue even when empty — 96 wakes/day on an idle deploy.

## 3. Done conditions

- **Idle probe:** an Upstash Redis pending-work signal — enqueue paths (which
  run with Neon already awake) set/increment it; the drain writes back the
  residual due-count (or earliest `nextAttemptAt`) after each run; the probe
  reads it, absent/zero → idle. The shell runs the probe after auth and
  *before* any Neon touch (lock included); idle → one structured console line
  + JSON response, return.
- **Budget alert gated the same way:** a short-TTL "recent exhaustion" marker
  set in the Redis budget gate (where exhaustion originates) means the Neon
  window query runs only when a marker exists.
- **Two safety valves:** Redis unconfigured (local dev) → probe returns
  unknown → proceed to Neon, matching the Redis-optional convention; one
  designated daily run drains Neon-backed regardless of the signal, so a lost
  Redis flag strands a job at most ~24h.
- Both `*/15` crons are declared `idle-silent` and demonstrably touch zero
  Neon on an idle run.

## 4. In scope

The drain cron route's declaration and probe, the enqueue paths' Redis signal
writes, the exhaustion-marker write in the Redis budget gate, the write-back,
the daily Neon-backed heal slot, tests.

## 5. Out of scope

- The deferred-queue worker's per-job processing logic.
- Moving the budget-exhaustion *history* out of Neon (the marker gates the
  read; the durable record stays).
- Schedule changes; new cron jobs; event-ledger vocabulary changes.

## 6. Hard constraints

- The shell's ordering guarantee (auth → idle probe → advisory lock → work →
  durable-telemetry gate) is owned by the shell, never re-implemented
  route-side.
- The 3.9.2.2.1 rails stay green (schedule gate; restricted-import lint).
- Redis-optional local dev must keep working (probe-unknown path).

## 7. Decisions the session plan must resolve

- The Redis signal's exact shape (flag vs. count vs. next-due timestamp), with
  the thrash case (a pending retry due in hours) explicitly analyzed.
- Which run is the daily heal slot.

## 8. Acceptance criteria

- An idle-path test (or instrumented local run) demonstrates **zero Neon
  connections** for both `idle-silent` crons' no-op path.
- A pending job present in Redis → the drain proceeds and drains it; the
  write-back leaves the correct residual signal.
- Redis-unconfigured path proceeds to Neon (probe unknown).
- The daily heal slot drains Neon-backed regardless of the signal.
