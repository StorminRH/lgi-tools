## Session 3.9.2.3.1 — ESI dataset registry & gate test (declaration layer)

**Sub-version:** 3.9.2.3 (Phase 2 — Primitive lifecycle; session 1 of 2, one branch)
**Master plan:** `docs/VERSION_3_9_PLAN.md` §3.9.2.3
**UX gate:** No

## 1. Objective

One declaration point per ESI-fed dataset — placement, verified cache time,
freshness model, refresh owner — with the placement rule enforced by a gate
test. This session delivers the registry module, the gate test, and complete
entries for every existing dataset.

**Verdict & evidence.** *Create* the registry (registry-with-gate pattern:
purge contributors, table-growth precedent). Combine rationale for the Create:
the owner-sync descriptor owns personal datasets' refresh *mechanics* but
nothing owns the placement *decision*; global/cron datasets (prices, indices,
SDE, GSC, market history) and the Convex live datasets have no declaration
point at all — today the rule lives in agent-guide prose + a SELF_REVIEW
checklist, and the 3.7.5.1 owned-blueprints mis-placement is the documented
failure this class of gate prevents.

## 2. Current context and dependencies

- 3.9.2.1 preferred first (cheap characterization of gated read paths); after
  3.9.2.2 keeps cron-route references stable (soft ordering). Must land before
  3.9.2.6 (the combine target must exist).
- Session .2 (runtime freshness gate + six-gate collapse) shares this branch.
- **Structure (pinned — the `src/purge/` layering pattern):** one primitive,
  layered by import direction. The **leaf** — entry type + clock-injected
  verdict function — lives in `src/lib/`; the **entries manifest + CI
  validator** compose above the slices. Slices import only the leaf; the
  validator imports the world. Three pinned properties keep it one primitive:
  one entry per dataset, a verdict with zero configuration of its own, and the
  rail banning staleness computation anywhere else.

## 3. Done conditions

- **The entry (written once, complete from day one):** dataset name; verified
  ESI cache time with the spec path it was read from; store (`convex` |
  `neon`); shape (`global-cron` | `personal-on-view` | `live`); refresh owner
  (descriptor, cron route, or engine); freshness model (`caller-ttl` |
  `row-stale-after` | `expires-boundary`); effective TTL, defaulting to the
  verified cache time, with an override-with-rationale field where it
  deliberately diverges (e.g. prices' 24h sweep).
- **The gate test enforces:** `convex` requires cache ≤ 120s or the
  collaborative flag; `neon` + `global-cron` names its cron route; `neon` +
  `personal-on-view` names its `OwnerSyncDescriptor`; effective TTL ≥ verified
  upstream cache; no durable table mirroring ESI data is unregistered
  (schema-discovery, the table-growth mechanism).
- Non-conformant-by-history placements are fixed or explicitly waivered *in
  the registry with rationale* — visible debt, never silent blessing.
- Every ESI-fed dataset has exactly one registry entry (freshness fields
  included from the start — no second pass over the entries later).

## 4. In scope

The registry entries manifest + CI validator (composed above the slices), the
`src/lib/` leaf (entry type; the verdict function itself may land in .2),
entries for every existing dataset, the gate test, SELF_REVIEW §1 pointer
update.

## 5. Out of scope

- Moving any dataset between stores (a discovered placement bug is *reported*,
  its waiver the record — migration is its own decision).
- Changing any TTL value or refresh cadence.
- The staleness-gate collapse and trigger comparison (session .2).
- The pricing *feature* surface: the AF-005 five-concern contexts
  (`planner-contexts.tsx`) and `PricingProvider` (902 LOC,
  monitored-not-actionable) are not touched.
- Next.js `'use cache'`/`cacheLife` profiles (eve-status et al.) stay
  unwrapped — the framework owns that layer.

  verdict) land with session .2's collapse.

  off-limits; anything campaign-shaped stops and reports.
