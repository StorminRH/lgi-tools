## Session 3.10.3.1.1 — Capability telemetry, SLIs, and the idempotency inventory

**Sub-version:** 3.10.3.1
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.3.1
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.3.1 and §3.10.3.2 — capability telemetry, SLIs, and the idempotency inventory
**Internal phases:** 1. Add capability telemetry names, outcome codes, and dependency durations and define the SLI set with owners and response actions; 2. Inventory every retryable or redeliverable mutation and job and record a verdict with evidence for each; 3. Add idempotency keys, uniqueness constraints, and replay tests only for the genuinely at-risk entries
**Split triggers:** The idempotency inventory surfaces a large at-risk set whose remediation exceeds this bundle and needs a master-plan amendment

## 1. Objective

Give the platform's important requests and jobs stable capability names and
outcome codes with a small owned set of user-centered service indicators, and
judge every retryable or redeliverable mutation and job for duplicate or loss
risk. Roadmap §3.10.3.2 is delivered inside this sub-version alongside
§3.10.3.1: the two form one observability-and-robustness bundle so the same
telemetry that names an operation is also the evidence that decides whether that
operation needs idempotency protection.

## 2. Current context and dependencies

- **DEP-1:** Sub-version 3.10.2.1 has delivered the stable error codes; outcome categories recorded here reuse those codes and introduce no parallel taxonomy.
- **DEP-2:** Sub-version 3.10.2.4's retry declarations name what can redeliver; the inventory's population of retryable and redeliverable entries derives from those declarations rather than a fresh survey.
- **DEP-3:** Existing telemetry lives in the `src/data/telemetry` tables and queries; that surface is the extension target. Its query file currently exposes 25 exports against the AF-006 Watch trigger, which fires at 26.
- **DEP-4:** The `src/data/esi-refresh-jobs` scheduled-ingestion surface is a redeliverable-job source for the inventory and carries the AF-007 Watch trigger; the version plan and baseline remain authoritative for both triggers.
- **DEP-5:** The LGI-06 revocation outbox belongs to the security tranche; the inventory may cite it as a coordinated future workflow but this session does not build it.

## 3. Done conditions

- **DC-1:** Important requests and jobs record feature plus operation name, an outcome category carrying the 3.10.2.1 stable error code, total duration, dependency durations, retry/rate-limit outcome, correlation id, and app version.
- **DC-2:** No high-cardinality identifier enters any metric label.
- **DC-3:** An SLI set is defined covering page/tool success rate, mutation success rate excluding validation failures, p95 latency on critical reads and writes, ESI success/throttle rate, and job backlog and terminal failures, each carrying a named owner and a response action.
- **DC-4:** Telemetry additions extend the existing tables and queries rather than paralleling them, the telemetry query surface stays below its AF-006 export trigger, and AF-006 and AF-007 are rechecked in the same change.
- **DC-5:** An inventory enumerates every cron job, webhook-like entry, scheduled ingestion, and redeliverable user mutation, each carrying a verdict of inherently idempotent, key-protected, or accepted-risk with recorded evidence.
- **DC-6:** Every entry judged at-risk gains an idempotency key persisted with a uniqueness constraint and a replay test proving no duplicated business effect, each such constraint rejects only writes already invalid under the workflow's own rule, and a recorded "no at-risk workflow" verdict is an accepted terminal outcome.

## 4. In scope

- **IS-1:** Telemetry field additions on the existing telemetry records and tables and their supporting query surface.
- **IS-2:** The SLI definition set with owners and response actions, and the minimal readout support needed in the existing admin telemetry surface.
- **IS-3:** The idempotency inventory artifact and its per-entry verdicts and evidence.
- **IS-4:** Idempotency keys, uniqueness constraints, and replay tests for the at-risk entries only.

## 5. Out of scope

- **OOS-1:** New observability vendors or an OpenTelemetry migration, which is recorded only as a future decision.
- **OOS-2:** Alerting infrastructure beyond what already exists.
- **OOS-3:** A general outbox framework, message bus, or event-sourcing system.
- **OOS-4:** The LGI-06 revocation outbox itself, which remains in the security tranche.
- **OOS-5:** The §3.10.3.3 generated architecture map and devlog flowchart.

## 6. Hard constraints

- **HC-1:** Outcome categories reuse the 3.10.2.1 stable error codes; no second code taxonomy is created.
- **HC-2:** No high-cardinality identifier is placed in a metric label.
- **HC-3:** The existing telemetry tables and queries are extended, not paralleled; the telemetry query export count stays below the AF-006 trigger and AF-006 and AF-007 are rechecked within this change.
- **HC-4:** An idempotency key is added only where duplicate or loss risk is real, and any uniqueness constraint rejects only writes that are already invalid under the protected workflow's business rule.
- **HC-5:** No application behavior changes except the telemetry recording, the SLI readouts, and the at-risk idempotency protections the master-plan sections authorize.
- **HC-6:** The LGI-06 revocation outbox may be cited by the inventory but is not implemented here.

## 7. Decisions the session plan must resolve

- **PD-1:** Where the SLI definitions live and which artifact owns their owners and response actions.
- **PD-2:** Which existing admin telemetry panels absorb the new readouts.
- **PD-3:** The inventory home — extending the existing cron and dataset rails versus a standalone document.
- **PD-4:** The at-risk shortlist selected from the completed inventory.
- **PD-5:** The exact telemetry field set and the recording seam for dependency durations and correlation id that keeps the telemetry query export breadth below the AF-006 trigger.

## 8. Acceptance criteria

- **AC-1:** Proves DC-1 and DC-2: a forced failure on a critical mutation is traceable in the recorded telemetry to its feature and operation, its outcome category with the stable code, its total and dependency durations, its retry/rate-limit outcome, its correlation id, and the app version, and no high-cardinality identifier appears in any metric label.
- **AC-2:** Proves DC-3: each defined SLI query returns a live value and each SLI names one owner and one response action.
- **AC-3:** Proves DC-4: the telemetry changes extend the existing tables and queries and the recorded telemetry query export count stays below the AF-006 trigger, with AF-006 and AF-007 rechecked and unbroken.
- **AC-4:** Proves DC-5: the inventory enumerates every cron job, webhook-like entry, scheduled ingestion, and redeliverable user mutation, each with a verdict and its evidence.
- **AC-5:** Proves DC-6: a green replay test exists for each at-risk workflow protected by a persisted key and a uniqueness constraint that rejects only already-invalid writes, or the recorded no-at-risk verdict stands as the accepted outcome.

## 9. Verification

- **V-1:** Recorded-telemetry trace evidence from a forced critical-mutation failure showing every required field present and correctly populated.
- **V-2:** Live SLI query outputs paired with each indicator's declared owner and response action.
- **V-3:** The completed idempotency inventory with a per-entry verdict and its supporting evidence.
- **V-4:** Replay-test results on each protected at-risk workflow, or the recorded no-at-risk verdict, with each added uniqueness constraint shown to reject only already-invalid writes.
- **V-5:** The AF-006 and AF-007 Watch recheck plus the standard close-out verification gates.

## 10. UX/operator gates

- **G-1:** UX gate is No; this session changes no user-facing behavior or appearance, so no operator browser-design review is required.

## 11. Baseline/hotspot boundary

The telemetry field and query additions create temporary pressure on the
telemetry query surfaces the AF-006 and AF-007 Watch triggers guard. The
session plan must state the expected export-count effect and stay below both:
the `src/data/telemetry/queries.ts` breadth remains below its 26-export trigger
(25 at version start) and the `src/data/esi-refresh-jobs/queries.ts` breadth
remains below its trigger (13 at version start), with both rechecked in the same
change. The idempotency inventory and its at-risk protections are baseline
neutral. Preserve the strict data-only baseline and its frozen version-start
values; no metric refresh is required beyond the Watch recheck.

and §3.10.3.2, which shares this delivery row and has no separate delivery of
