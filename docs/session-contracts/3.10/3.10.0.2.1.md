## Session 3.10.0.2.1 — Canonical contract form & lifecycle-narrative retirement

**Sub-version:** 3.10.0.2 (Phase 0 — Documentation & lifecycle consolidation; session A of two, one branch)
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.0.2
**UX gate:** No

## 1. Objective

Build the canonical contract frame the later migrations move content into: the
`docs/workflows/` home exists, the contract template is a single canonical file,
the resolver validates a contract's section structure by deriving required
sections from that template's own headers, the release-consistency dispatch gate
is wired explicitly (A9), and the two docs the resolver already replaces
(`DEVELOPMENT_LIFECYCLE.md`, `SESSION_CONTRACTS.md`) are deleted.

## 2. Current context and dependencies

- Depends on 3.10.0.1 (green base). Session A of sub-version 3.10.0.2; session B
  (`3.10.0.2.2`) reformats the baseline on the same branch.
- The resolver already *is* the lifecycle; `DEVELOPMENT_LIFECYCLE.md` and
  `SESSION_CONTRACTS.md` are prose describing what the resolver enforces.
- Interacts with the v3.9 close-out audit: if that audit already wrote the
  baseline in strict data-only form, session B only adds enforcement; otherwise
  it also reformats. (This session A is the contract-form half.)

## 3. Done conditions

- `docs/workflows/` exists as the canonical procedure home.
- The contract *template* (section skeleton + marker vocabulary, extracted from
  `SESSION_CONTRACTS`) is a single canonical file.
- The resolver validates a contract's section structure by **deriving the
  required sections from the template's own headers** (extending or rewriting its
  marker/digest parser — parser changes are in scope), so a contract missing a
  required section resolves to a **repair directive** instead of dispatching the
  next stage (recorded decision A5).
- The `check_release_consistency.py --check` dispatch gate is wired as an
  explicit pre-dispatch step in the resolver's directive contract (recorded
  decision A9).
- Every `DEVELOPMENT_LIFECYCLE.md` section is confirmed to have a live home in
  the resolver/checkers; then `DEVELOPMENT_LIFECYCLE.md` and
  `SESSION_CONTRACTS.md` are deleted, and `policy-manifest.json` `canonicalGuides`
  plus any checker references are updated in the same slice.
- **Every policed live reference to the two deleted docs is repointed in the same
  slice**, so `check_doc_refs` stays green in this PR. Policed referencing files
  today are: `AGENTS.md`; the standing docs still present at this slice
  (`PR_REVIEW.md`, `SESSION_END.md`, `SESSION_PLANNING.md`, `VERSION_AUDIT.md` —
  each retired in a later slice but live now); and both runtime skill trees
  (`start-session`, `plan-version`, `plan-session`, and any other referencing
  skill). The active roadmap and active session-contracts are RECORD (exempt from
  policing after 3.10.0.1) so their references do not block, but obviously-stale
  ones — including the INDEX header's schema pointer — are tidied here too.
  References are repointed to the resolver / the new canonical template. This is a
  reference cutover only — migrating any doc's *steps* stays with its own slice
  (0.3–0.5). (Note: consolidating `start-session`'s own duplicated step content is
  **not assigned by the master plan** — see the disclosure list; this slice only
  repoints its references.)

## 4. In scope

The `docs/workflows/` directory; the contract template canonical file; the
resolver/parser modification for section-derivation and the A9 dispatch gate;
the two doc retirements; the live-reference cutover across `AGENTS.md`, the
surviving standing docs, the active roadmap, and both skill trees (including
`start-session`); the manifest/checker updates and revision bump.

## 5. Out of scope

- The strict data-only baseline + changelog form (session B, `3.10.0.2.2`).
- Migrating any workflow's *steps* (0.3–0.5).
- Any map/policy change (0.6); the AGENTS numbering *wording* (A4 lands in 0.6 —
  only the template's `X.Y.N` sub-version shape is fixed here).
- The version-audit workflow that *writes* the baseline (aligned in 0.5).

## 6. Hard constraints

- The resolver's parser is **in scope, not a constraint to design around** — it
  is modified as needed to fit the model.
- No section of a retired doc is deleted until its live home in the
  resolver/checkers is confirmed (Decisions §7), and **no live reference to a
  deleted doc survives this PR** — references are repointed, not left dangling.
- No application code changes. Touched surfaces are `.agent-local/` (resolver +
  checkers), `docs/workflows/`, `policy-manifest.json`, the agent guides/skills
  carrying the reference cutover, and the two deleted docs.
- Each Phase 0 slice keeps `check_agent_drift.py`, `check_doc_refs.py`, and the
  resolver green at merge (bootstrapping discipline).

## 7. Decisions the session plan must resolve

- Physical home of the template (`docs/workflows/` vs a `docs/templates/`
  sibling).
- How the resolver reads the template headers (runtime parse vs generated
  manifest entry).
- Confirmation that no `DEVELOPMENT_LIFECYCLE` section (runtime-todo invariant
  §2, health-feedback §8, and the rest) lacks a resolver/checker home before
  deletion.

## 8. Acceptance criteria

- A contract with a missing required section fails resolution with a **repair
  directive** (proved by a checker/resolver test).
- The release-consistency gate is an explicit, tested pre-dispatch step in the
  resolver directive contract.
- `DEVELOPMENT_LIFECYCLE.md` and `SESSION_CONTRACTS.md` no longer exist and
  nothing references them (`check_doc_refs.py` clean).
- `check_agent_drift.py` green at a bumped revision.

## 9. Verification

- `check_doc_refs.py` zero dead references to the deleted docs.
- `check_agent_drift.py` green; `pnpm verify` green.
- Changelog entry note (sub-version 3.10.0.2 ships once, at session B close-out).
