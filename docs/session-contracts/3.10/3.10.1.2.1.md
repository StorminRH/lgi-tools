**Sub-version:** 3.10.1.2
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.1.2
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.1.2 and §3.10.1.3 — shared→composition split, blocking cycles, and server-only rails
**Internal phases:** 1. Split the shared zone into a composition zone and a leaf shared zone and close the .tsx-only glob gap; 2. **Split triggers:** A cycle flip reveals a genuine dependency cycle that needs a design change rather than a mechanical reclassification; or the authoritative server-only root list is contested and needs an operator decision

## 1. Objective

Close the remaining structural loopholes on the boundary map: separate leaf
shared code from cross-feature composition into two one-way zones so the
sanctioned `shared`↔`features` edge disappears, make dependency-cycle detection
blocking now that the graph is clean, and give the browser/server boundary
mechanical enforcement whose primary in-repo gate runs in verify and lint. This
belongs in sub-version 3.10.1.2 because it is the boundary-hardening step that
completes Phase 1 on top of the full zone coverage delivered by 3.10.1.1, with
no application behavior change.

## 2. Current context and dependencies

- **DEP-1:** Sub-version 3.10.1.1 delivered the clean, fully-classified zone map covering every production source area; this bundle builds only on that confirmed-clean graph.
- **DEP-2:** The current shared zone is 34 files — 20 that import `@/features` or `@/data` (composition) and 14 leaf files; the split projects onto that live classification.
- **DEP-3:** Three of the 20 composition importers cross the zone with type-only imports (`ServerStatus.tsx`, `type-icon.tsx`, `telemetry/telemetry-payload.ts`), a borderline classification that the plan must settle.
- **DEP-4:** `circular-dependencies` and `re-export-cycle` are `warn` today; the flip to `error` requires the confirmed-clean dependency graph from DEP-1.
- **DEP-5:** `server-only` is not yet a dependency and the browser/server boundary has no mechanical guard; the server roots are `src/db`, `src/lib/env.ts`, the server auth configuration, the ESI dispatch gate, and the vendor adapter roots.
- **DEP-6:** The typed-env rail exists and stays unchanged; Convex isolate modules have no client graph.

## 3. Done conditions

- **DC-1:** The former shared zone resolves into two one-way zones — a composition zone importable only by `app`, and a leaf shared zone restricted to `ui`/`lib` — and the classification covers `.ts` as well as `.tsx` files, closing the glob gap.
- **DC-2:** Features may import leaf shared and never composition, composition is importable only by `app`, and no reverse edge from either shared zone into `features` remains; an unsanctioned feature→composition import is mechanically rejected.
- **DC-3:** `circular-dependencies` and `re-export-cycle` are `error` rather than `warn`, and the full boundary run is green with no suppression or waiver used to reach it.
- **DC-4:** Server-only markers sit at the authoritative server roots, and an ESLint restricted-import rule — the primary in-repo gate that runs inside `pnpm verify`/lint — blocks client files from importing those roots; the `server-only` package remains only a production-build backstop.
- **DC-5:** An enumeration test asserts the approved server-only root set and fails when a new server root lacks its marker or when a client file reaches one.
- **DC-6:** The single owning guide states the two-zone shared/composition rule and the server-only boundary rule, reconciled with the consolidated map owner and free of duplicated normative copy, and records the Convex isolate exemption.
- **DC-7:** No application behavior changes; only boundary configuration, markers, lint rules, tests, and guide sentences are touched, and the typed-env rail is unchanged.

## 4. In scope

- **IS-1:** Splitting the shared zone into a composition zone and a leaf shared zone, including any file moves the plan judges cheaper than pattern membership, and closing the `.tsx`-only glob gap so classification also covers `.ts` files.
- **IS-2:** Flipping `circular-dependencies` and `re-export-cycle` from `warn` to `error` after a confirmed-clean run.
- **IS-3:** Server-only marker imports at the server roots, the ESLint restricted-import rail as the primary gate, and the enumeration test over the approved root set.
- **IS-4:** Guide sentence updates for the two-zone rule, the server-only boundary rule, and the recorded Convex isolate exemption.

## 5. Out of scope

- **OOS-3:** New directory ceremony beyond what the composition/leaf classification requires.
- **OOS-4:** The full zone-coverage work owned by 3.10.1.1 and any Phase 2+ flow-contract or telemetry arc.
- **OOS-5:** Adding server-only markers to Convex isolate modules; they have no client graph and stay exempt.

## 6. Hard constraints

- **HC-1:** The primary in-repo server-only gate MUST be the ESLint restricted-import rule that runs in `pnpm verify`/lint; the `server-only` package guard fires only at the Vercel production build, which this repo never runs locally, so it is a backstop and cannot be the enforcing gate.
- **HC-2:** The cycle rules flip to `error` only after a confirmed-clean dependency-graph run; no cycle may be suppressed, ignored, or waived to achieve the flip.
- **HC-3:** Features may import leaf shared and never composition, composition is importable only by `app`, and no reverse edge into `features` may exist.
- **HC-4:** No application behavior changes and the existing typed-env rail is left unchanged.
- **HC-5:** Convex isolate modules are exempt from server-only marking, and the exemption is recorded rather than left implicit.
- **HC-6:** No Fallow waiver, baseline entry, threshold relaxation, or boundary exception is added to reach a green boundary run.

- **PD-3:** The authoritative server-only root list.
- **PD-4:** Whether the enumeration test discovers server roots from the zone map or from a declared registry, with discovery preferred to match the established endpoint-gate pattern.
