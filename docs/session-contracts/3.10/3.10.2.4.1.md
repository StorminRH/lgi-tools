## Session 3.10.2.4.1 — Vendor resilience registry

**Sub-version:** 3.10.2.4
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.2.4
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.2.4 — vendor resilience registry
**Internal phases:** 1. Record the per-integration resilience registry across every external dependency; 2. Route each external call through a declared wrapper with an explicit timeout, closing the unwrapped Redis gap; 3. Add the gate that fails on a direct vendor call outside its registered adapter and reconcile the guide sentences
**Split triggers:** An integration has no safe default timeout or retry policy and needs an operator decision; or closing an unwrapped call materially changes runtime behavior beyond resilience

## 1. Objective

Give every production external dependency one declared resilience policy: a
single registry that records each integration's wrapper, timeout, and retry and
degradation behavior; every external call site routed through its declared
wrapper with an explicit timeout; and a gate that fails a direct vendor call
made outside its registered adapter. This is the resilience closure of the
Phase 2 correctness arc, extending the integration rails that already exist for
ESI to full coverage. No application behavior changes except the timeout, retry,
and degradation routing this section explicitly mandates.

## 2. Current context and dependencies

- **DEP-1:** Phase 0 (3.10.0.x) documentation and lifecycle consolidation is merged into current `origin/main`; the section otherwise declares no in-version dependency and may plan against present repository state.
- **DEP-2:** EVE ESI already has a single wrapper (`esiFetch`/`esiUrl` under `src/lib/esi`) and an ESLint host-ban rail; this is the established pattern to extend, not rewrite, and its Redis budget model is the pattern rather than the patient.
- **DEP-3:** Upstash Redis has no single wrapper — a raw client is constructed at four production sites that share only a credential resolver — which is the known unwrapped call gap this session must close.
- **DEP-4:** EVE SSO, Better Auth, and Convex have central modules but no import rail; Vercel is a deploy platform only, with no programmatic API call surface; the GitHub API is used only in `.agent-local` Python tooling, which is outside ESLint scope.
- **DEP-5:** The integrations in scope are EVE ESI, EVE SSO/Better Auth, Convex, Upstash, Neon, Vercel APIs, and the GitHub API in tooling; each must carry a recorded resilience policy.

## 3. Done conditions

- **DC-1:** A single registry records, per integration in scope, all eight resilience fields — wrapper entry point, timeout, retryable error classes, backoff/jitter policy, rate-limit behavior, idempotency stance, degradation behavior, and telemetry fields — with each listed integration present and an integration that has no programmatic call surface recording that fact rather than a fabricated policy.
- **DC-2:** Every production external call site routes through its integration's declared wrapper with an explicit timeout, and the four unwrapped Upstash Redis construction sites resolve to one declared wrapper without any runtime behavior change beyond the added resilience policy.
- **DC-3:** Adapters translate each integration's SDK types and errors to owned types at the boundary.
- **DC-4:** A gate (test or lint) fails on a direct vendor call made outside its registered adapter, extending the existing ESI-gate and vendor-import rails to full coverage across the listed integrations.
- **DC-5:** The guide sentences that describe vendor-call resilience ownership are reconciled to the completed registry and gate, leaving no stale resilience claim.

## 4. In scope

- **IS-1:** The per-integration vendor resilience registry as the contracted artifact, complete for every listed integration.
- **IS-2:** The explicit-timeout and retry gaps the registry exposes at existing call sites, including consolidating the four unwrapped Upstash Redis construction sites behind one declared wrapper.
- **IS-3:** The out-of-adapter vendor-call gate that extends the existing ESI-gate and vendor-import lint rails to full coverage.
- **IS-4:** Boundary adapters that translate the listed integrations' SDK types and errors to owned types.
- **IS-5:** The guide sentences that state vendor-call resilience ownership.

## 5. Out of scope

- **OOS-1:** Circuit breakers or any new resilience infrastructure without a demonstrated need.
- **OOS-2:** Changing any vendor choice.
- **OOS-3:** Rewriting the ESI gate or its Redis budget model; its budget model is the pattern this session copies, not a target of change.
- **OOS-4:** Neighboring Phase 2 roadmap intent — the typed error contract (§3.10.2.1), the mutation pipeline order and same-origin enforcement (§3.10.2.2), and the data ownership, transaction, and RLS registry (§3.10.2.3).
- **OOS-5:** Implementing capability telemetry, SLIs, or new observability emission (§3.10.3.1); the registry only records each integration's telemetry fields, it does not build the telemetry pipeline.
- **OOS-6:** Application, UI, database schema, dependency, or Convex data-placement behavior changes beyond the resilience routing DC-2 mandates.

## 6. Hard constraints

- **HC-1:** No production application behavior changes except the timeout, retry, and degradation routing this section explicitly mandates; closing an unwrapped call preserves the site's existing runtime behavior other than the added resilience policy.
- **HC-2:** The existing ESI wrapper, host-ban lint rail, and Redis budget model are extended toward full coverage, never rewritten and never duplicated by a parallel wrapper.
- **HC-3:** Timeouts are recorded and enforced as explicit values at each call site; an implicit or SDK-default timeout is not an acceptable end state.
- **HC-4:** Every listed integration appears in the registry with all eight resilience fields populated, and an integration with no programmatic call surface records that absence explicitly.
- **HC-5:** The gate fails on a real seeded direct out-of-adapter vendor call; no waiver, suppression, threshold relaxation, or fuzzy exception may be added to make it pass.

## 7. Decisions the session plan must resolve

- **PD-1:** The registry form — a typed module consumed by tests versus a document paired with a census test that reads it.
- **PD-2:** The exact set of existing call sites that lack a real explicit timeout, resolved by sweeping first and fixing in-slice where the fix is Floss-sized.
- **PD-3:** The single Upstash Redis wrapper's entry point and how the four current construction sites consolidate behind it without changing their runtime behavior.
- **PD-4:** Whether the out-of-adapter gate is a test or a lint rail per integration, and how it extends the existing rails to cover integrations that have a central module but no import rail (EVE SSO, Better Auth, Convex) and the GitHub API tooling that sits outside ESLint scope.
- **PD-5:** The idempotency stance and degradation behavior recorded for each integration, and which of those, or which absent default timeout or retry policy, require an operator decision rather than a session-plan default.
- **PD-6:** Which guide sentences describe vendor-call resilience ownership and how they reconcile with the completed registry and gate.

## 8. Acceptance criteria

- **AC-1:** Proves DC-1: the registry lists every integration in scope with all eight resilience fields populated, or an explicit no-programmatic-surface note, verifiable through the registry's census or consumer check.
- **AC-2:** Proves DC-2: a repository sweep shows every production external call site routing through its declared wrapper with an explicit timeout, and the four Upstash Redis construction sites resolving to one declared wrapper with unchanged runtime behavior.
- **AC-3:** Proves DC-3: each integration's SDK types and errors are translated to owned types at the boundary rather than leaking vendor types to callers.
- **AC-4:** Proves DC-4: the gate goes red on a seeded direct out-of-adapter vendor call and green once that call routes through its registered adapter.
- **AC-5:** Proves DC-5: the reconciled guide sentences describe the completed registry and gate, and an ownership sweep finds no remaining stale vendor-call resilience claim.

## 11. Baseline/hotspot boundary

Expected baseline direction is neutral: this session adds a registry, wrappers,
and a gate without restructuring measured production surfaces. `src/lib/esi/dispatch.ts`
is a listed large file — extend it, do not restructure it. No metric refresh is
mandated; planning verifies current proximity against the live baseline and
keeps the change below the universal caps.

## 12. Close-out behavior

separate sub-version from 3.10.2.3 for the same reason: the two movements are
independent enforcement systems carrying independent production-risk changes,
and combining their diffs would be genuinely unreviewable. The execution marker
may become `Complete` only when every done condition holds. Delivery is standard
planned close-out on the sub-version branch: the single sub-version PR carries
the terminal roadmap row, the `APP_VERSION` bump to 3.10.2.4, and the published
changelog entry that absorbs the pending fragments present at its cutoff, and it
passes the standard close-out verification, the pre-PR design review, and the
current-head Greptile gate before the operator-reserved merge. No UX pause
applies. The version plan remains authoritative for sequence and delivery
status; the session plan supplies the exact branch, commands, and artifacts.
