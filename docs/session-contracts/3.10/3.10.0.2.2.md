## Session 3.10.0.2.2 — Strict data-only baseline & changelog form

**Sub-version:** 3.10.0.2 (Phase 0 — Documentation & lifecycle consolidation; session B of two, one branch)
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.0.2
**UX gate:** No

## 1. Objective

Make `CODE_HEALTH_BASELINE.md` a pure, resolver-enforced metrics record with a
frozen version-start column and a writable current column, and extract the
changelog-entry template as a canonical form.

## 2. Current context and dependencies

- Depends on session A (`3.10.0.2.1`, same branch): the `docs/workflows/` home
  and the canonical-form derivation pattern exist.
- Today's baseline carries a rolling "Previous" column, ~100 lines of prose, and
  a campaign queue — the drift that currently defeats start→current comparison
  (a session can move the starting point).

## 3. Done conditions

- `CODE_HEALTH_BASELINE.md` becomes a **pure metrics record**: every row is a
  measured metric with exactly two value columns — **version-start** (captured
  once at version adoption, frozen for the whole version) and **current** (the
  only column a session updates) — plus a derived delta.
- There are **no prose sections, no notes, no delta commentary, and no campaign
  queue**: scheduled structural work lives in the version plan / backlog, and
  findings and rationale live in the version-tagged audit report
  (`docs/version-audits/X.Y/`) that archives with the version.
- The baseline schema is a single canonical template; the resolver /
  `check_baseline_claims` is extended to enforce two invariants: it **rejects any
  content outside the schema** (an injected note or extra section fails the gate)
  and **rejects any session edit to the version-start column** (the frozen anchor
  is write-once at version adoption).
- The reformat removes the rolling "Previous" column, the prose, and the campaign
  queue.
- The changelog-entry template (from `PR_REVIEW`) is extracted as a canonical
  form.

## 4. In scope

The strict two-column baseline template + reformat of the live baseline; the
resolver/`check_baseline_claims` enforcement (schema-only + version-start
immutability); the changelog canonical form; manifest/checker updates and the
revision bump; the sub-version's single PR.

## 5. Out of scope

- The contract-template + lifecycle-doc retirement (session A).
- The version-audit workflow that *writes* the baseline (aligned to this schema
  in 0.5) — only the template + enforcement land here.
- Re-measuring any metric: the baseline is reformatted, not re-measured.

## 6. Hard constraints

- The version-start column is **write-once at version adoption**; a session may
  update only the current column (enforced, not conventional).
- No prose or free-text survives in the baseline (Phase 0 non-goal).
- No application code changes.
- Green resolver/drift/doc-ref at merge.

## 7. Decisions the session plan must resolve

- How the resolver reads the baseline schema (runtime parse vs generated
  manifest entry).
- Where and when the immutable version-start column is captured (the resolver's
  version-adoption step) and how immutability is enforced (diff against the
  adoption snapshot vs a write-lock).

## 8. Acceptance criteria

- A baseline with an injected prose note or extra section fails
  `check_baseline_claims` (test).
- A session edit to the version-start column fails the gate (test).
- The reformatted baseline holds only the two-column metric rows + derived delta;
  no campaign queue, no prose.
- The changelog-entry canonical form exists and `PR_REVIEW`/close-out reference
  it (pending its own migration in 0.3).

## 9. Verification

- `check_baseline_claims` red on a seeded prose note and on a seeded
  version-start edit; green on the reformatted baseline.
- `check_agent_drift.py` green at a bumped revision; `check_doc_refs.py` clean.
- `pnpm verify` green.

## 10. UX/operator gates

**UX gate: No.** Baseline/checker + canonical-form change; no user surface.

## 11. Baseline/hotspot boundary

The baseline is reformatted, not re-measured — metric values are preserved. No
hotspot contact; no-campaign decision stands (the campaign queue is removed as a
*structure*, and 3.10 has no open campaign to carry into it).

## 12. Close-out behavior

Session B closes sub-version 3.10.0.2: full close-out on the shared branch —
`pre-pr-design-review`, the single PR for 3.10.0.2, Greptile loop, clean merge,
production reconciliation. Roadmap row 3.10.0.2 → terminal on merge evidence.
`APP_VERSION` bump + v3.10 changelog entry for 3.10.0.2 ride this PR.
