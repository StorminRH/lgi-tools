## Session 3.10.2.2.1 — Mutation pipeline declared order and same-origin enforcement

**Sub-version:** 3.10.2.2
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.2.2
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.2.2 — mutation pipeline declared order and same-origin enforcement (backlog LGI-03)
**Internal phases:** 1. Document the mutation gate order and its threat-model rationale at the pipeline owner; 2. Review the observe-only cross-origin telemetry and confirm no legitimate cross-origin caller before enforcing; 3. Reject explicit Origin/Referer mismatches with a mapped 403 and add negative tests plus same-origin coverage discovery
**Split triggers:** The telemetry review finds a legitimate cross-origin caller, requiring an operator decision on an allowlist or a reject-log-then-enforce window before the enforcement flip

## 1. Objective

Make the mutation path's gate order a written, tested contract at its owner, and
graduate the same-origin check from observe-only logging to real enforcement so
cross-origin browser mutations begin returning 403 — completing backlog LGI-03.
This is the Phase 2 production-flow-contract slice that closes the last
observe-only security gap, and it is the entire architecture movement's only
intentional production behavior change, which is why it lives alone in
sub-version 3.10.2.2.

## 2. Current context and dependencies

- **DEP-1:** Sub-version 3.10.2.1 must be complete before this session begins; its typed failure set and RFC 9457-compatible problem mapper will emit the 403 this session introduces, rather than a new error path.
- **DEP-2:** The same-origin check is observe-only today — it records a cross-origin mutation telemetry event and always proceeds, so no cross-origin mutation is currently rejected (backlog LGI-03).
- **DEP-3:** The mutation pipeline's live order is authorize, then same-origin (observe), then parse, then handler, with rate limits owned by callers before the boundary; the intended documented order is cheap rejection, rate limit, identity, same-origin, parse, object authorization, transaction, telemetry.
- **DEP-4:** A same-origin coverage inventory already pins the full mutating-route set (17 pipeline routes plus 5 direct routes, 22 total); it is the existing rail this session extends with classification discovery rather than a parallel mechanism.
- **DEP-5:** Cron and service routes authenticate on separate caller auth and sit outside the browser same-origin boundary; they are unaffected prerequisites, not targets.

## 3. Done conditions

- **DC-1:** The intended mutation gate order and its threat-model rationale are documented at the pipeline owner where the mutation wrapper lives.
- **DC-2:** The observe-only cross-origin telemetry has been reviewed and the review positively confirms zero legitimate cross-origin callers before any enforcement flip, or the split trigger is recorded as fired.
- **DC-3:** Explicit Origin or Referer mismatches on browser mutations are rejected with a 403 emitted through the 3.10.2.1 problem mapper, while same-origin mutations still succeed and cron/service routes are unaffected.
- **DC-4:** The classification of missing-provenance requests (no Origin and no Referer) is a settled, documented decision, and observed behavior matches that decision.
- **DC-5:** The same-origin coverage inventory gains discovery so a newly added mutating route carrying no declared classification fails the coverage test.
- **DC-6:** Backlog LGI-03 is completed and closed, and no other security-tranche backlog item is absorbed by this session.
- **DC-7:** The AF-008 auth-surface watch trigger is rechecked in the same change and its resulting status is recorded.

## 4. In scope

- **IS-1:** The documented mutation gate order and threat-model rationale co-located with the mutation pipeline owner.
- **IS-2:** Graduating the same-origin check from observe-only to enforcement for explicit Origin/Referer mismatches on browser mutations, delivered through the 3.10.2.1 mapper.
- **IS-3:** The review of the observe-only telemetry period and its recorded conclusion about legitimate cross-origin callers.
- **IS-4:** The classification decision for missing-provenance requests and the observable behavior it prescribes.
- **IS-5:** The same-origin coverage inventory discovery extension plus negative and same-origin integration coverage.
- **IS-6:** The AF-008 auth-surface watch recheck performed within this change.
- **IS-7:** Completing and closing backlog LGI-03 exactly as its entry is written.

## 5. Out of scope

- **OOS-1:** LGI-02 step-up auth and the other security-tranche items (LGI-01, LGI-04, LGI-05, LGI-06, LGI-07, LGI-09, LGI-10, LGI-11, LGI-12); each keeps its own backlog trigger.
- **OOS-2:** A full Fetch-Metadata or content-type provenance policy if the telemetry review argues for its deferral; the include-or-defer decision is recorded either way.
- **OOS-3:** Any change to cron or service caller auth, and any move of those routes onto the browser same-origin boundary.
- **OOS-4:** The typed failure set and RFC 9457 mapper themselves (owned by 3.10.2.1) and any status-code semantics beyond the same-origin 403.

## 6. Hard constraints

- **HC-1:** The observe-only telemetry review must positively confirm zero legitimate cross-origin callers before the enforcement flip; a discovered caller fires the split trigger rather than a silent allowlist.
- **HC-2:** The 403 is produced only through the 3.10.2.1 problem mapper; no new plain-text or ad-hoc error response is introduced.
- **HC-3:** Cron and service routes stay on separate caller auth and must not begin returning same-origin 403s.
- **HC-4:** This is the movement's only intentional production behavior change and an independent rollback boundary; it ships as its own bundle and is never merged together with an additive refactor, registry, or unrelated slice.
- **HC-5:** The change touches auth-surface files; the AF-008 watch trigger, which fires at four or more auth-surface files and stands at three, must be rechecked in the same change and must not cross its threshold silently.
- **HC-6:** No application behavior changes anywhere except the same-origin enforcement flip and its 403 response.

## 7. Decisions the session plan must resolve

- **PD-1:** The enforcement rollout shape — a direct flip versus a short reject-log-then-enforce window.
- **PD-2:** How missing-provenance requests (no Origin and no Referer) are classified — rejected, allowed, or provenance-policy gated — consistent with the confirmed telemetry.
- **PD-3:** Whether a Fetch-Metadata or content-type provenance policy is included now or deferred, with the decision recorded either way.
- **PD-4:** The exact discovery mechanism by which the same-origin coverage inventory forces a newly added mutating route to declare a classification.
- **PD-5:** The documentation home and threat-model framing co-located with the mutation pipeline owner.

- **AC-3:** Proves DC-3: an integration test shows a cross-origin browser mutation returns 403 through the problem mapper while a same-origin mutation succeeds, and cron/service routes are unaffected.

- **AC-5:** Proves DC-5: a seeded newly added mutating route with no declared classification fails the same-origin coverage test, and the declared 22-route set passes.

Neutral baseline direction: the change adds one documentation block and one
enforcement decision rather than new measured production surface, so no metric
growth is expected and the strict data-only baseline's frozen version-start

files, planning must recheck the AF-008 watch trigger — four or more
auth-surface files, currently three — against the current baseline and confirm
the change stays below that threshold before close-out.
