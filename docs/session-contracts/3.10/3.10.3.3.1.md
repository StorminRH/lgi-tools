## 1. Objective

Ship the architecture movement's final slice: a zone/dependency map that is
derived from the live Fallow configuration rather than drawn, and a stylized
version of that map published on the public devlog. The map presents the
finished Phase 1 zone graph before any presentation-system work begins, and its
single visible surface is why this outcome is its own sub-version.

## 2. Current context and dependencies

- **DEP-1:** Sub-version 3.10.1.1 delivered full boundary coverage — every production source file classifies into a named `.fallowrc.json` zone with declared allowed dependencies — so that configuration is the generator's single authoritative source of zones and edges.
- **DEP-2:** Sub-version 3.10.1.2 must be complete before this session begins; its shared→composition split and blocking dependency-cycle checks will make the zone graph this map shows complete and one-directional (acyclic), with no reverse or sanctioned-cycle edge left to render.
- **DEP-3:** This outcome is sequenced as the architecture movement's final slice; it renders the finished graph and precedes the presentation-system work, which it does not touch.
- **DEP-4:** The public devlog already renders assembled content within established rendering rules and the terminal/EVE visual identity using theme tokens; this session extends one devlog section within those existing rules.

## 3. Done conditions

- **DC-1:** A generator that reads `.fallowrc.json` zones and rules emits a deterministic Mermaid source of the zone-level dependency graph — zones as nodes, allowed edges as arrows, first-match and exception edges visibly distinct — using only the standard library or the existing toolchain.
- **DC-2:** The generated Mermaid source is committed as the tracked intermediate, and a drift test fails whenever the committed source no longer matches the current configuration.
- **DC-3:** A stylized rendering of that zone graph appears in the appropriate public devlog section in the established terminal/EVE visual identity, styled from theme tokens only, with no raw hex and no ad-hoc palette.
- **DC-4:** The devlog page renders the visualization within existing rendering rules, raw-HTML sinks stay lint-restricted with no new exception, and one devlog prose passage introduces the map.
- **DC-5:** No application behavior changes beyond the new devlog map and its introducing passage, and no other devlog section is restyled or otherwise altered.
- **DC-6:** ux-check evidence on the changed devlog route and the operator's approved local browser review are both recorded before the sub-version PR opens.
- **DC-7:** The final sub-version PR carries truthful completed 3.10.3.3 lifecycle and changelog state and remains unmerged with green CI and current-head Greptile 5/5.

## 4. In scope

- **IS-1:** The deterministic zone-graph generator and its accompanying drift test.
- **IS-2:** The committed Mermaid source of the zone-level dependency graph as the tracked intermediate.
- **IS-3:** The public devlog visualization component or asset that renders the stylized zone graph and its theme-token styling.

## 5. Out of scope

- **OOS-1:** File-level or symbol-level dependency graphs; this map is zone-level only, and Codegraph already owns fine-grained structure.
- **OOS-2:** A live or interactive graph explorer of any kind.
- **OOS-3:** Restyling, reordering, or otherwise altering any other devlog section.
- **OOS-4:** New heavy client dependencies; a client-side Mermaid runtime is presumed rejected in favor of build-time SVG or a small owned component unless the session plan proves otherwise.
- **OOS-5:** Changing the zones, rules, or boundary definitions in `.fallowrc.json` or the architecture ownership map, which are owned by sub-versions 3.10.1.1 and 3.10.1.2.

## 6. Hard constraints

- **HC-1:** The diagram is derived, never drawn — the committed Mermaid source is generated from `.fallowrc.json`, and no hand-authored zone diagram is introduced.
- **HC-2:** The generator depends only on the standard library or the existing toolchain; no new generation dependency is added.
- **HC-3:** The generated Mermaid source is deterministic; repeated runs against unchanged configuration produce byte-identical output.
- **HC-4:** The devlog visual uses theme tokens only — no raw hex and no ad-hoc palette — and stays within the established terminal/EVE identity.
- **HC-5:** The devlog page renders the visualization within existing rendering rules; raw-HTML sinks remain lint-restricted and no new raw-HTML lint exception is added.
- **HC-6:** No application behavior changes except the new devlog map and its introducing passage.
- **HC-7:** The map shows the completed, one-directional zone graph delivered by 3.10.1.1 and 3.10.1.2, with first-match and exception edges kept visibly distinct.

## 7. Decisions the session plan must resolve

- **PD-1:** The rendering path — build-time Mermaid→SVG with tokenized styling versus a small owned visx/SVG component reading the generator's JSON — decided against the raw-HTML lint rules and bundle cost.
- **PD-2:** Where the generated artifacts live — a `content/devlog` asset versus component-owned files.
- **PD-3:** Whether the drift test also publishes the Mermaid source into the devlog for copy-paste.
- **PD-4:** How first-match and exception edges are made visibly distinct in both the generated source and the rendered visual.
- **PD-5:** Which devlog section hosts the map and the exact introducing prose passage.
