## Session 3.10.2.1.2 — Full route and caller migration to closed-union endpoint contracts

**Sub-version:** 3.10.2.1
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.2.1
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one PR per session
**Roadmap coverage:** §3.10.2.1 — typed error contract and RFC 9457 problem mapper, extended by the operator-folded end-to-end type-safety migration
**Internal phases:** 1. Migrate the route guards, the mutation pipeline, and the shared parsers/limiters to typed failures mapped at the delivery boundary; 2. Redeclare every first-party endpoint contract with its full status map and migrate all route responses through the endpoint-bound helper; 3. Migrate every browser caller to the closed result union and land the raw-`Response` guard gate with the route-level status matrix
**Split triggers:** A status-code semantic change requires a product decision; or a guard or route migration surfaces behavior that is not purely error-shape-preserving

## 1. Objective

Every first-party programmatic route and browser caller consumes the typed
primitives the previous session created. The route guards and the mutation
pipeline emit typed failures translated by the single RFC 9457 mapper, every
endpoint contract declares its complete status map, every route builds its
responses through the endpoint-bound helper, and every browser caller narrows
the closed status-discriminated result union. Who is admitted and who is
rejected does not change anywhere; only error body shapes change.

## 2. Current context and dependencies

- **DEP-1:** The previous session shipped the shared primitives on this branch: the eight-category typed failure set, the single problem mapper with correlation ids, the v2 endpoint-contract model with per-status codecs, the closed-union `apiFetch` overload, the endpoint-bound `apiResponse` helper, the typed-failure cores beside the legacy wrappers, and the feedback endpoint as the migrated exemplar.
- **DEP-2:** The unmigrated surface at planning time: 51 route handlers, 38 v1 endpoint objects across 13 contract modules, roughly 35 `apiFetch` call sites, roughly 68 in-route `satisfies` response pins, and the shared guard emitters in `src/platform/auth/route-guards.ts`, `src/lib/service-auth.ts`, `src/transport/route-body.ts`, `src/lib/rate-limit.ts`, and the cron shell in `src/composition/pipelines/cron-gate.ts`.
- **DEP-3:** The typed failure set is fixed at exactly validation, unauthenticated, forbidden, not-found, conflict, rate-limited, dependency-unavailable, and unexpected; the problem body carries `type`, `title`, `status`, a stable application `code`, and a correlation id.
- **DEP-4:** The Better Auth catch-all route is library-owned and stays outside first-party contract migration.
- **DEP-5:** This session unblocks the seam-cleanup session that deletes the v1 path, and the same-origin enforcement flip that reuses the mapper's 403.

## 3. Done conditions

- **DC-1:** The route guards, bearer-secret guards, body parsers, rate-limit guard, and mutation pipeline return typed failures that reach the wire only through the single mapper, and their admission decisions are identical to before.
- **DC-2:** Every first-party programmatic endpoint has a v2 contract declaring its method, path, request encoding, and every response status with its body codec, owned by its slice's contract module.
- **DC-3:** Every route-built response in first-party routes is constructed through the endpoint-bound response helper; route-local response pins are removed where the helper supersedes them.
- **DC-4:** Every browser `apiFetch` caller consumes the closed status-discriminated union; no caller reads an opaque `Response` from the failure arm.
- **DC-5:** Representative 400/401/403/404/409/429/5xx paths are proven at route level for stable problem shape and non-leakage.
- **DC-6:** A mechanical gate rejects a raw-`Response` return from a guard, with a seeded red fixture proving the gate can fail.
- **DC-7:** Wire statuses, admission behavior, ordering of guards and parsers, and cron and service semantics are unchanged; only error body shapes change, and the migration keeps the repository compiling at every checkpoint commit.

## 4. In scope

- **IS-1:** Migration of the shared guards, parsers, rate limiter, cron shell, and mutation pipeline onto the typed-failure cores.
- **IS-2:** Redeclaration of all first-party endpoint contracts as v2 with complete status maps.
- **IS-3:** Migration of every first-party route response onto the endpoint-bound helper.
- **IS-4:** Migration of every browser `apiFetch` call site onto the closed union.
- **IS-5:** The raw-`Response` guard gate and the route-level representative status matrix with non-leakage assertions.
- **IS-6:** Deletion of legacy wrapper seams whose last consumer migrated within this session's scope, where removal does not require the v1 endpoint model itself.

## 5. Out of scope

- **OOS-1:** Deleting the v1 `ApiEndpoint` model and `apiFetch` overloads; the final cleanup session owns old-path deletion.
- **OOS-2:** Convex-to-Next and other server-to-server transport, Better Auth hand-pinned seams, and query/path parameter builders.
- **OOS-3:** The same-origin enforcement flip from observe-only to rejecting, and any status-code semantic change.
- **OOS-4:** Client error-display redesign beyond mechanical result-union adaptation.

## 6. Hard constraints

- **HC-1:** Admission decisions, guard ordering, wire statuses, and cron and service semantics are preserved exactly; only the authorized error-shape translation changes responses.
- **HC-2:** Exactly one mapper owns failure-to-HTTP translation; no route or guard constructs a problem body by hand.
- **HC-3:** Deep application and data code touched by the migration returns typed failures, never an HTTP `Response`.
- **HC-4:** No `any`, `as unknown as`, broad assertions, or duplicated wire interfaces anywhere in the migration.
- **HC-5:** The repository compiles and `pnpm verify` passes at each planned checkpoint commit; no big-bang breakage window.

## 7. Decisions the session plan must resolve

- **PD-1:** Migration batching: the endpoint-family order and checkpoint boundaries that keep every commit green.
- **PD-2:** The 429 body transition: how existing consumers of the current rate-limit JSON body move to the problem body without user-visible regression.
- **PD-3:** Cron route responses: whether the cron shell's JSON outcome body becomes a declared v2 contract or a documented framework-owned delegation.
- **PD-4:** The guard gate's mechanism: AST-based lint, test-suite classifier, or both, and the exact surfaces it covers.
- **PD-5:** How the same-origin coverage test's assertions track problem bodies without pre-empting the later enforcement flip.

- **AC-2:** Proves DC-2 and DC-3: a mechanical sweep shows every first-party route associated with a v2 contract and every constructed response flowing through the endpoint-bound helper.
