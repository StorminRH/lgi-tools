## 1. Objective

Make Fallow the complete mechanical boundary owner of the source tree: every
production source file classifies into a named architectural zone with declared
allowed dependencies, deny-by-default, with no unzoned area remaining. This is
the first Phase 1 slice because it finishes the boundary map the two later
loophole-closing slices build on, and it lands after Phase 0 so its ownership
table and guide sentences write into the consolidated, single-owner docs.

## 2. Current context and dependencies

- **DEP-1:** Phase 0 (3.10.0.x) is shipped; the consolidated map and guide docs and the single-owner anti-duplication rules are the destinations this slice's ownership table and import-direction sentences must land in, which is why §3.10.1.1 is sequenced after Phase 0.
- **DEP-2:** `.fallowrc.json` declares six zones today (`auth-surface`, `ui`, `features`, `data`, `lib`, `shared`); roughly 204 production files are unzoned across `src/app`, `src/db`, `src/config`, `src/purge`, `src/search`, `src/page-settings`, `src/esi-datasets`, and `convex/`, plus the loose top-level `src/proxy.ts` and `src/instrumentation*` files and the nine co-located `src/components/*.ts` files the `.tsx`-only shared glob misses. Planning confirms these areas and counts against live source.
- **DEP-3:** The `boundary-violation` rule is already error-level, but the violation surface the new deny-by-default zones expose is unknown until the config first runs; this is an unbounded-discovery spike, so the ledger escape hatch and the split trigger both remain in force.
- **DEP-4:** No architecture-arc peer is a prerequisite; 3.10.1.2 (shared→composition split, cycle-rule flips, `server-only` rails) is a later sub-version that depends on this one, not the reverse.

## 3. Done conditions

- **DC-1:** Every currently unzoned production area and every loose top-level and co-located source file classifies into a named zone with an explicit allow-list, deny-by-default, so no production source file remains unclassified.
- **DC-2:** Each new zone's allow-list expresses the same allowed-dependency direction recorded in the consolidated import-direction facts, and the config grants no dependency the map or guides forbid.
- **DC-3:** The architecture ownership-map table (area / owns / may depend on / must not own), covering every zone, is published under `docs/`.
- **DC-4:** Every boundary violation the new zones surface is triaged; Floss-sized violations are resolved in-slice as deletions or import moves, and every remaining violation is ledgered so no real debt is hidden.
- **DC-5:** Boundary enforcement rejects a cross-zone import that a declared allow-list forbids and is green on the reconciled tree.
- **DC-6:** No application behavior changes, and the affected map and guide sentences are reconciled to the delivered full coverage.

## 4. In scope

- **IS-1:** `.fallowrc.json` zone and rule additions that give every currently unzoned production area and every loose top-level and co-located source file a named zone with an explicit allow-list.
- **IS-2:** The architecture ownership-map table as a `docs/` artifact.
- **IS-3:** Small conformance fixes limited to deletions or import moves for Floss-sized violations, plus ledger entries for the violations that exceed that bound.
- **IS-4:** Reconciling the affected consolidated import-direction sentences to the delivered zone coverage.

## 6. Hard constraints

- **HC-1:** No application behavior changes; every conformance fix is a deletion or an import move only.
- **HC-2:** Coverage is deny-by-default: no production source file may remain unzoned, and a named zone without an explicit allow-list is not acceptable.
- **HC-3:** Each zone allow-list must agree with the recorded import-direction facts; the config may not grant a dependency the map or guides forbid.
- **HC-4:** The ledger escape hatch stays available for any surfaced violation larger than Floss-sized; unbounded discovery may not force an unreviewable in-slice fix, and a ledger entry may not conceal real debt.
- **HC-5:** Extend the existing Fallow boundary rail rather than adding parallel boundary machinery, except where the §7 `convex/` decision resolves to a dedicated import test given its separate bundling.

- **PD-2:** Whether `convex/` boundary rules live in Fallow or in a dedicated import test, given Convex's separate bundling.
- **PD-3:** The classification of the loose top-level `src/instrumentation*` and `src/proxy.ts` files.
- **PD-4:** The classification of the nine co-located `src/components/*.ts` files the current `.tsx`-only shared glob misses.
