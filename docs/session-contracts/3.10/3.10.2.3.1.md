## Session 3.10.2.3.1 — Data ownership, transaction and RLS registry

**Sub-version:** 3.10.2.3
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.2.3
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.2.3 — data ownership, transaction and RLS registry
**Internal phases:** 1. Extend the dataset-declaration census with per-table owning slice, permitted cross-owner read contracts, database-enforced write invariants, and a transaction-boundary statement; 2. **Split triggers:** A proposed constraint would reject writes that are currently valid, requiring a data-cleanup or product decision before it can ship

## 1. Objective

Give every persisted table a named owning slice, a stated transaction boundary,
and a recorded authorization decision, and turn the invariants that are genuinely
exposed to concurrency into database-enforced constraints. This belongs in
sub-version 3.10.2.3 because it is the data-layer production-flow contract: the
request path already flows through good gates, but data ownership, cross-owner
access, and race-sensitive invariants are only conventions. The outcome extends
the existing dataset-declaration rail, not a new parallel governance document.

## 2. Current context and dependencies

- **DEP-1:** The registry extends the live dataset-declaration census in `src/db/dataset-declarations.test.ts`, which reflects the current schema dynamically over its present table set; this session extends that single rail rather than creating a second document.
- **DEP-2:** The per-data-class authorization decision is recorded against the real role model documented in the 3.9.4.1 privilege runbook (`docs/security/db-privilege-runbook.md`).
- **DEP-3:** Sequenced after Phase 0 so ownership and rendering guidance already has single owners; this slice has no dependency on the other Phase 2 slices and is parallel-safe with them.
- **DEP-4:** The standing tenant boundary is application-authorization-only with the Neon Data API disabled; no table uses row-level security today, and that remains the default this session may only depart from where a recorded decision justifies it.

## 3. Done conditions

- **DC-1:** Every census table carries a named owning slice, its permitted cross-owner read contracts, the write invariants the database enforces for it, and an explicit transaction-boundary statement.
- **DC-2:** Each data class carries a recorded authorization decision — application-authorization-only or application plus row-level security — resolved against the live role model, with row-level security enabled nowhere the decision does not justify it.
- **DC-3:** Every invariant judged genuinely race-exposed today is enforced by a database constraint backed by real-Postgres coverage, and no already-valid write is rejected by any constraint added.
- **DC-4:** An undeclared cross-owner write fails the census test.

## 4. In scope

- **IS-1:** The per-table ownership, cross-owner read-contract, database-enforced write-invariant, and transaction-boundary declarations, delivered through the existing dataset-declaration census inputs or a sibling registry the same test reads.
- **IS-2:** The per-data-class authorization decision recorded against the live role model.
- **IS-3:** Database constraints for the identified race-exposed invariants together with their real-Postgres `*.db.test.ts` coverage.
- **IS-4:** The census-test extension that rejects undeclared cross-owner writes, and any dead cross-owner edge the declaration exercise surfaces.

## 5. Out of scope

- **OOS-1:** Enabling row-level security anywhere the per-data-class decision does not justify it; the application-authorization-only tenant boundary and the disabled Neon Data API remain the standing invariant.
- **OOS-2:** Schema redesign, table restructuring, or migration work beyond the constraints the race-exposed invariants require.
- **OOS-3:** Convex data-placement rules, which remain owned by `docs/CONVEX.md`.
- **OOS-4:** The vendor resilience registry and its wrapper/timeout/retry work, which belongs to sub-version 3.10.2.4.

## 6. Hard constraints

- **HC-1:** The registry extends the single existing dataset-declaration rail; no second parallel data-governance document is introduced.
- **HC-2:** Any new database constraint must reject only writes that are already invalid; confirm that no currently-valid write is rejected before shipping the constraint.
- **HC-3:** Row-level security is recorded as a per-data-class decision and is not enabled where that decision does not justify it.
- **HC-4:** No application behavior changes except the database constraints this master-plan section explicitly authorizes.

## 7. Decisions the session plan must resolve

- **PD-1:** The registry home — extending the census test inputs directly versus a sibling registry the same test reads.
- **PD-2:** Which invariants are genuinely race-exposed today and therefore warrant a database constraint rather than an application-level check.
- **PD-3:** The data-class taxonomy the authorization decision is recorded against and how it maps onto the live role model from the 3.9.4.1 privilege runbook.

## 8. Acceptance criteria

- **AC-1:** Proves DC-1: every census table exposes its owning slice, permitted cross-owner read contracts, database-enforced write invariants, and transaction-boundary statement, and the census passes with those declarations complete.
- **AC-2:** Proves DC-2: a recorded authorization decision exists for each data class and resolves against the live role model, with no row-level security enabled outside a justifying decision.
- **AC-3:** Proves DC-3: each identified race-exposed invariant is enforced by a database constraint whose real-Postgres coverage passes, and a check confirms no currently-valid write is rejected.
- **AC-4:** Proves DC-4: a seeded undeclared cross-owner write turns the census test red.

## 9. Verification

- **V-1:** Dataset-declaration census evidence, including the seeded undeclared cross-owner write that fails the census test.
- **V-2:** Real-Postgres `*.db.test.ts` coverage for each added constraint, run through the shared database test harness.
- **V-3:** Confirmation, against the current data shape, that no already-valid write is rejected by any newly added constraint.
- **V-4:** The standard close-out definition-of-done and review gates.

## 10. UX/operator gates

- **G-1:** UX gate is No; this session changes no user-facing behavior or appearance, so no operator browser review is required.

## 11. Baseline/hotspot boundary

Expected direction is neutral-to-improves: the declaration exercise may surface
and delete dead cross-owner edges, and the added constraints and registry fields
do not touch measured production surfaces. No metric refresh is required and the
strict data-only baseline and its frozen version-start values are unaffected.
Planning verifies the current baseline before recording the update.

## 12. Close-out behavior

This is the sole session in sub-version 3.10.2.3. It commits on the shared
sub-version branch and completes the sub-version through standard close-out; its
execution marker may become `Complete` once the done conditions are proven, under
the ordinary Greptile and CI gate of record. The slice is kept separate from
3.10.2.4 because combining data-governance and vendor-resilience work would
produce a genuinely unreviewable diff mixing two independent enforcement systems
and two independent production-risk changes — a Postgres schema and constraint
migration versus rewrapping live external calls. The session plan supplies the
exact branch, commands, and release records.
