## Session 3.10.2.1.1 — Typed error contract and RFC 9457 problem mapper

**Sub-version:** 3.10.2.1
**Master plan:** `docs/VERSION_3_10_PLAN.md` §3.10.2.1
**UX gate:** No
**Execution profile:** Frontier autonomous coding agent
**Delivery unit:** One agent session, one shared sub-version branch, one sub-version PR
**Roadmap coverage:** §3.10.2.1 — typed error contract and RFC 9457 problem mapper
**Internal phases:** 1. Define the typed failure set and the single RFC 9457-compatible problem mapper; 2. **Split triggers:** A status-code semantic change requires a product decision; or a guard migration surfaces behavior that is not purely error-shape-preserving

## 1. Objective

Give the request path one error vocabulary. Application failures become typed
values, and a single delivery-boundary translation renders them as a stable,
safe HTTP problem shape. This is the first Phase 2 production-flow contract: it
replaces improvised plain-text error responses with typed failures and one
RFC 9457-compatible mapper, changing only the emitted error shape and never the
admission decision behind it.

## 2. Current context and dependencies

- **DEP-1:** This bundle is parallel-safe with Phase 1 and sequenced after Phase Zero's consolidated ownership; it carries no cross-session prerequisite within the architecture arcs.
- **DEP-2:** Verified greenfield state: there is no typed failure set and no RFC 9457 / problem+json mapper today. The route guards return bare plain-text `Response('Unauthorized'|'Forbidden')`, and the existing route-body and ESI error helpers are ad-hoc utilities, not a shared HTTP error vocabulary.
- **DEP-3:** The typed failure set is a settled product fact fixed by the roadmap: exactly validation, unauthenticated, forbidden, not-found, conflict, rate-limited, dependency-unavailable, and unexpected.
- **DEP-4:** The RFC 9457-compatible body is a settled product fact: it carries `type`, `title`, `status`, a stable application `code`, and a correlation id.
- **DEP-5:** This bundle unblocks 3.10.2.2, whose same-origin `403` reuses this mapper, and 3.10.3.1, whose capability telemetry reuses this bundle's stable error `code`.

## 3. Done conditions

- **DC-1:** A single typed failure set exists covering exactly the eight named categories and is consumable by deep application and data code.
- **DC-2:** One RFC 9457-compatible mapper translates any typed failure at the delivery boundary into a stable body carrying `type`, `title`, `status`, a stable application `code`, and a correlation id.
- **DC-3:** The route guards and the mutation pipeline emit mapped failures instead of plain-text `Response` objects, while their admission decisions — who is allowed and who is rejected — remain identical.
- **DC-4:** Deep application and data code returns typed failures and never constructs an HTTP `Response` directly.
- **DC-5:** Representative 400/401/403/404/409/429/5xx paths produce the stable problem shape and leak no internal detail such as stack traces, internal identifiers, or raw dependency messages.
- **DC-6:** `apiFetch` is aware of the problem body so clients receive the typed problem rather than opaque text, with no redesign of client error display.
- **DC-7:** Migration is incremental and behavior-preserving: routes not touched by the guards or the pipeline are left to convert opportunistically, no status-code semantics change, and a raw-`Response` return from a guard is mechanically rejected.

## 4. In scope

- **IS-1:** The typed failure set covering the eight named categories.
- **IS-2:** The single RFC 9457-compatible problem mapper and its stable body fields.
- **IS-3:** Migration of the route guards and the mutation pipeline to emit mapped failures.
- **IS-4:** `apiFetch` awareness of the problem body.
- **IS-5:** Contract tests across the representative status set for shape and non-leakage, plus the mechanical gate that rejects a raw-`Response` return from a guard.
- **IS-6:** Opportunistic conversion of the deep application and data code that the guard and pipeline migration directly touches.

## 5. Out of scope

- **OOS-1:** Mass-rewriting route handlers that already behave correctly; conversion of untouched routes is a later opportunistic follow-on, not this bundle.
- **OOS-2:** Client UX redesign of error display.
- **OOS-3:** Changing any status-code semantics; any such change requires a product decision and a test documenting the old and new contract.
- **OOS-4:** The same-origin enforcement flip from observe-only to a rejecting `403` and the declared mutation-pipeline order, both owned by 3.10.2.2.
- **OOS-5:** The data-ownership/RLS registry (3.10.2.3), the vendor resilience registry (3.10.2.4), and the capability telemetry and SLI work (3.10.3.1).

## 6. Hard constraints

- **HC-1:** No application behavior changes except the error-shape translation this roadmap section authorizes; admission decisions are preserved exactly.
- **HC-2:** Exactly one mapper owns the delivery-boundary translation; no second HTTP error vocabulary or parallel mapper is introduced.
- **HC-3:** The problem body is RFC 9457-compatible and stable and must not leak internal implementation detail.
- **HC-4:** The typed failure set is exactly the eight roadmap-named categories — no fewer and no speculative extras.
- **HC-5:** Deep application and data code returns typed failures and never an HTTP `Response`.
- **HC-6:** Migration remains incremental; the bundle does not rewrite already-correct handlers en masse.

## 7. Decisions the session plan must resolve

- **PD-1:** The physical home of the failure set — a `src/lib` leaf versus a contract-adjacent location — without creating a redundant error vocabulary.
- **PD-2:** The correlation-id source — reuse of the existing telemetry id versus a newly minted id.
- **PD-3:** How the same-origin coverage test assertions track the new problem bodies without pre-empting the 3.10.2.2 enforcement flip.
- **PD-4:** The representative route or path chosen to exercise each of 400/401/403/404/409/429/5xx for the shape and non-leakage proofs.
- **PD-5:** The exact mechanical gate that fails a raw-`Response` return from a guard and the surfaces that gate covers.

## 8. Acceptance criteria

- **AC-1:** Proves DC-1 and DC-2: the typed failure set enumerates exactly the eight named categories, and each category maps through the single mapper to a complete body carrying `type`, `title`, `status`, a stable application `code`, and a correlation id.
- **AC-2:** Proves DC-3: the route guards and the mutation pipeline return mapped problem bodies for unauthenticated, forbidden, and pipeline failures while the same requests are admitted or rejected identically to before.
- **AC-3:** Proves DC-5: representative 400/401/403/404/409/429/5xx responses carry the stable problem shape and contain no stack trace, internal identifier, or raw dependency message.
- **AC-4:** Proves DC-4 and DC-7: deep application and data code returns typed failures rather than `Response`, a seeded raw-`Response` return from a guard fails its gate, routes outside the guard and pipeline path are unchanged, and no status-code semantics shift.
- **AC-5:** Proves DC-6: `apiFetch` surfaces the typed problem body to clients without any client-side error-display redesign.

## 11. Baseline/hotspot boundary

Neutral-to-improves. The bundle deletes ad-hoc error construction and replaces
it with one typed failure set and one mapper; the existing ad-hoc route-body and
ESI error helpers remain in place except where the guard and pipeline migration
directly touches them, and neither becomes the shared vocabulary. No production
surface grows in a way that warrants a metric refresh, and the strict data-only
code-health baseline and its frozen version-start values are preserved.
Planning verifies proximity against the current baseline before committing.
