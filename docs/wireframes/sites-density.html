Wireframe Notes — sites-density (Resource-Scoped Hovers)

Updated per feedback: hover affordances apply only to ore and gas cards. Combat, relic, and data cards stay static — no :hover rule, no card-glow, no resource preview. The two top panels show the full ore/gas vocabulary in its hovered state; the two bottom panels show the wave-driven cards at rest, which is also how they render under a pointer.

Why this split. Ore and gas cards live or die on the resource list — that's the entire question the player is asking ("what's in this site, is it worth running?"). The header ISK answers roughly; the hover preview answers precisely, without forcing an expand-and-scroll. Combat / relic / data cards answer their value question in the header already (est. loot is a single number); their detail-tier content (waves, EWAR, hackable containers) doesn't fit a 3-row preview the way ore values do, so the hover would be busy without being useful.

The two affordances on resource cards.

  • Card-glow — box-shadow: 0 0 0 1px rgba(61,214,140,0.08) + a 1px green-tinged border shift. Indicates the card is interactive (clickable to expand or to deep-link).
  • Resource preview — 240px-wide panel anchored to the right edge of the card showing top-3 resources by ISK + the total. Slides in on the same hover trigger; in production it'd be absolutely-positioned and overlay neighbouring cards. (In this wireframe the parent card is shifted left via margin-right: 260px to make room — purely a layout cheat for the static sketch.)

Implementation note. The split is one CSS class: .card.resource vs .card.wave-driven. SiteCard.tsx already knows the site type via the isWaveDriven = isCombat || isHackSite flag (line 46 of the current component) — emit the class from there. No new prop, no new logic; the renderer just selects which hover bucket the card sits in.

Dropped from the previous version: the "wave totals in header" panel (no longer needed — combat cards don't get new affordances) and the "freshness chip" panel (separate question; not part of this density slice). Both ideas can resurface as their own version-2.9.x slot if they earn it later; they're not on the table for the implementation that follows from this wireframe.
