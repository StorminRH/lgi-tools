Wireframe Notes — nav-search-inline-push (chosen direction)

Mechanic. Search bar lives between the wordmark and the tool strip. At rest it's a 280px input with a placeholder and a ⌘K hint. On focus (click or keyboard), the field grows to 440px and the tool strip is pushed right; the tools shrink from full labels (Wormhole Sites) to 2-letter abbreviations (WH) so the right slot stays put. On blur, everything restores.

Wordmark is static. No blinking cursor block in the logo — the bracket-stamp [LGI].tools sits at rest with no animation. The search input uses the browser's native text caret when focused; the only styled visual is the green > prompt at the input's left edge. One thing happening at a time.

Right-slot guarantee (load-bearing). The nav-right cluster (admin chip · portrait · name · characterId) never gets pushed or overlapped. CSS contract: .nav-right { flex-shrink: 0 }; .nav-tools { flex-shrink: 1 } absorbs the squeeze first via the full-label → acronym swap; the search bar's expansion stops at whatever room remains after the right slot and the acronym tools have claimed their space. On narrow viewports the search bar reaches its expansion ceiling earlier rather than the right slot being pushed.

Pros.

  • Tool links remain visible the whole time — discoverability never disappears.
  • Search is always present as a visible affordance, not hidden behind a keyboard shortcut.
  • Acronym fallback (WH/IP/WR) is a natural future-proof: tooltips on the truncated tools recover the full label.
  • Admin/login chip is permanent — every state transition preserves the user's awareness of who they're logged in as.

Scaling beyond 3 tools. At 5+ tools the acronyms start to collide (WH vs WR vs WC ambiguity is the canary). When that point lands, the natural move is to group related tools rather than keep adding top-level entries: a Wormholes ▾ dropdown that contains Sites + Roll Calc + future wormhole-adjacent tools, an Industry ▾ dropdown for Planner + Blueprint Calc, etc. The nav stays at 3-5 visible entries forever; the depth grows under dropdowns. No need to design that today — the inline-push mechanic itself doesn't change.

Spotlight-like categorization is shown in the dropdown: Sites, Recent, Commands. Tools, Resources, and Help are additional sections that'd appear for matching queries. The scope chips in the dropdown footer let the user restrict the search to one category if results are noisy. Long-term, this is the surface where every searchable entity in the platform (sites, ore + gas types, sleepers, commands, blueprints, killmails…) plugs in — one query string, one result list, multiple data sources behind the scenes.
