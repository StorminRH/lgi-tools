Wireframe Notes — nav-side-rail (Alt)

Layout. 56px-wide vertical rail on the left. Top: bracket-mark only (no .tools wordmark — vertical text would read poorly). Middle: icon-only tool list using 2-letter monogram glyphs (WH / IN / BP) with hover tooltips. The Industry item is drawn in its hover state — icon shifts to --isk, faint green outer glow, tooltip slides in. Bottom: compact freshness dot + portrait square.

Page header bar. Because the rail can no longer hold the tool name in a horizontal line, the main area gains a thin 48px header bar identifying the active tool, plus a corner slot for the login session. This is new chrome the top-bar option doesn't need.

Pros: wins horizontal space, looks unmistakably "professional tool" rather than "blog with nav." Rail stays out of the way on a wide monitor and never crowds the right side of the content.

Cons: brand legibility takes a hit — the full LGI.tools wordmark only appears on the landing page, not next to the tool you're using. The 2-letter monograms work for three tools but get strained at 6-8 (BP vs BC vs B?). And the page-header bar duplicates information the rail could otherwise convey, eating some of the space the rail just saved.

Recommended only if the user wants the horizontal real estate back for wide tool surfaces (the eventual Industry Helper's bill-of-materials table, say). For a content-density tool like /sites, the top-bar option is a cleaner fit.
