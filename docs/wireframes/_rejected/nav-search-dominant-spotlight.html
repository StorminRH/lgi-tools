Wireframe Notes — nav-search-dominant-spotlight

Mechanic. Search is the dominant element of the nav at all times — it claims roughly 60% of the nav width even when no one's interacting with it. Tool links collapse to 48px-wide icon-buttons (acronym + micro-label) to keep navigation visible without competing for space. Nav is 52px (8px taller than the other two) to accommodate the always-visible scope chips inside the search field.

The thesis. If search is going to cover every feature and item in the site — tools, all 69 sites, all SDE ore + gas + salvage types, commands, recent history, admin actions — then it deserves to be the primary navigation, not a secondary affordance. Tool icons are still discoverable but they're shortcuts for "open the live tool," not "navigate the platform."

Scope chips are persistent. Unlike the overlay-takeover variant where chips appear only when active, here they're always visible. ⌘1-5 keyboard shortcuts switch scope without needing to mouse. The selected scope changes what the dropdown shows AND filters results in real time as you type.

Cross-reference dropdown. The example shows the power: searching bistot with resources scope returns the type itself AND every site containing that resource. This is the "build lots of functionality into it" angle — the same search can be a type lookup, a site filter, a command palette, and a recent-history jumper. Future scopes (sleepers, drops, fits, killmails) all slot into the same UI.

Pros.

  • Strongest commitment to search as a first-class platform feature.
  • Scope chips make the breadth of searchability self-documenting — a new visitor sees "ah, I can search resources AND commands AND sites."
  • Tool icons stay clickable for users who want the classic nav model, but they're explicitly secondary.
  • 52px nav is the only one with room for inline scope chips AND keyboard hints — no compromise.

Cons / open questions.

  • 52px nav is 8px taller than the other options. Confirm the vertical real-estate cost is acceptable.
  • Tool icons (WH/IP/WR) need to be self-explanatory or the micro-label has to do work. Acronyms collide as more tools land — by tool #6, "WR" vs "WC" is ambiguous.
  • Persistent scope chips are visual noise when the user isn't searching. Counter-argument: they advertise functionality, so the noise is also the discoverability.
  • Login chip drops the character name (no room) — only portrait + characterId pill survive. Confirm that's OK post-2.8.1 (the name was originally added so users could verify which alt they're logged in as).

The bistot → sites containing this resource cross-reference uses the existing siteResources.typeId column (set in Phase 2.6's resource-aliases pass) — no new schema or queries needed beyond a reverse-lookup. Scopes map directly to data sources the codebase already has: tools = the landing-grid tile registry, sites = listSiteDetails(), resources = eve_types SDE table filtered to ore/gas/salvage categories, commands = a static registry like USAGE_ACTIONS mapped to handlers.
