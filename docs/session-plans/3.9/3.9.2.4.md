## Context

The api-contract convention (17 `api-contract.ts` files, `apiFetch`, `route-body.ts`) is
convention-complete but not mechanically complete. The existing audit,
`src/app/api/api-contracts.test.ts`, asserts only that every route imports *some*
`api-contract` module. Nothing checks that a body-consuming route actually validates via a
contract Zod schema, the "no-user-input marker" that AGENTS.md claims the audit expects
**does not exist mechanically** (it is free prose in 7 files, in inconsistent forms), and
nothing prevents a future inline-literal `apiFetch` argument. ~15 routes today have
neither schema consumption nor any marker. This session expands the existing audit into a
completeness gate — the contract's combine-bias explicitly prefers this over a new sibling
mechanism.

## 1. Goal

A route handler under `src/app/api` cannot ship without either consuming a Zod schema from
its slice's `api-contract.ts` (via `route-body.ts`, directly or through the mutation
pipeline) or carrying a formal, closed-vocabulary input marker — and every `apiFetch` call
site must reference a declared endpoint object, enforced by lint.

## 5. Interface designs

**No new production exports.** The entire gate lives in the extended test file, the marker
vocabulary, and one lint selector. Public surfaces and their draft contracts:

**(a) The formal input marker** — closed vocabulary, exact sibling of `// authz:`:

```
// input: none   — the handler reads no user-supplied body, query, or path input
// input: query  — user input is query/path params only, validated as typed values
                   in-handler (JSON/form bodies remain the schema's job)
```

Rule enforced per route (XOR): the route source either references
`parseJsonBody`/`parseFormBody` (schema consumption — marker forbidden, it would lie) or
carries exactly one valid `// input:` marker. Vocabulary is closed; an unknown class is
red. Bodyless-`runMutationRoute` routes (e.g. `sessions/revoke`) take `// input: none`.

**(b) Extended test prologue** (draft interface comment for `api-contracts.test.ts`):

```
// Mechanical API-contract completeness gate (3.9.2.4) — expands the 3.4.T
// presence check. For every src/app/api/**/route.* file this asserts:
//   1. it imports from its owning slice's api-contract module (unchanged);
//   2. it either consumes a contract Zod schema through route-body.ts
//      (parseJsonBody/parseFormBody — directly or via the mutation pipeline's
//      parse: option) or declares exactly one closed-vocabulary input marker:
//        // input: none | query
//      — never both: a marker on a schema-consuming route would lie;
//   3. if it builds a JSON payload in-route (.json( appears), at least one
//      `satisfies` pins the payload to an identifier imported from the
//      contract module (cron routes delegate to defineCronRoute<T>, redirect
//      routes return no JSON — both exempt by construction).
// The checks are pure functions over route source, unit-tested below with
// seeded-violation fixtures (the gate's own red evidence). Semantic drift
// remains pnpm typecheck's job, as before.
```

The classifier functions (`hasSchemaConsumption`, `findInputMarkers`,
`checkResponsePinning` — names indicative) stay **private to the test file**: they have no
production caller (P4), and exporting them would create a test-only production surface.

**(c) Lint selector** — appended inside the existing `apiFetchSelectors` const:

```
// An inline object literal as apiFetch's first argument bypasses the declared-
// endpoint convention (it typechecks against ApiEndpoint). Endpoints are
// declared once in the owning slice's api-contract.ts and passed by name.
{ selector: "CallExpression[callee.name='apiFetch'] > ObjectExpression:first-child", ... }
```

Passing a typed identifier (incl. `use-live-dataset.ts`'s parameter and the test-local
consts in `api-client.test.ts`) stays legal — only in-place literals are banned.

## 6. Design alternatives

- **Gate mechanism — chosen:** extend `api-contracts.test.ts` in place (combine-bias;
  runs inside `pnpm test` → `verify` → CI; fixture-testable pure functions).
  **Rejected:** a new sibling registry test with a static route→classification manifest
  (`same-origin-coverage.test.ts` style). One more file, one more static list to
  reconcile, and the classification is derivable from route source — a manifest would
  duplicate knowledge the source already states (P2).
- **Marker shape — chosen:** closed two-class `// input: none | query` marker line,
  validated exactly-one, mirroring `authz-markers.test.ts`. **Rejected:** enforcing the
  existing free prose ("No user input …") — 7 files carry it in 4 shapes, half inside
  TSDoc blocks; regexing prose is fragile and the prose stays valuable as rationale
  *beside* the token. Also rejected: single-token `// body: none` — accurate but hides
  the query-input story the closed vocabulary can state.
- **Cron/webhook family — resolved:** cron/service routes keep their contract import and
  `defineCronRoute<T>` response pinning and take `// input: none` like any other
  schema-less route. No family-level exemption: their bearer auth is already declared by
  `// authz: cron|service`, and an input marker states a different fact (no user input),
  so the uniform rule needs no special case (P5 — the special case is defined away).
