## 5. Interface designs

**No new production exports.** The new surface is lint configuration plus one
test file; the decision each piece owns:

- Three shared pattern consts in `eslint.config.mjs` (mirroring the existing
  selector-family consts, with the same replace-semantics rationale). Draft
  comments (config `//` commentary — the comment standard's `/** */` interface
  form applies to `src/`/`convex/` exports, none of which are added):

  ```js
  // UI-library import rail (3.9.2.9, PL-012): feature and app code consume
  // Base UI and sonner only through the wrap-once library in
  // src/components/ui/. Factored like the selector families above because
  // flat-config rule options REPLACE per matching file — every block that
  // re-states no-restricted-imports must re-list the bans it keeps.
  const baseUiImportPatterns = [
    {
      group: ["@base-ui/react", "@base-ui/react/*"],
      message:
        "Base UI is consumed only through the shared wrappers in @/components/ui — import the primitive (Dialog, Select, Tooltip, …), not the package. See CONTRIBUTING.md (Component system).",
    },
  ];
  // The pre-1.0 package name; never a valid dependency in this repo.
  const deprecatedBaseUiImportPatterns = [
    {
      group: ["@base-ui-components/react", "@base-ui-components/react/*"],
      message:
        "@base-ui-components/react is the deprecated Base UI package — the repo uses @base-ui/react, and only through the shared wrappers in @/components/ui.",
    },
  ];
  const sonnerImportPatterns = [
    {
      group: ["sonner"],
      message:
        "sonner is consumed only through @/components/ui/toast (the sole Toaster owner) — import its toast helpers instead. See CONTRIBUTING.md (Component system).",
    },
  ];
  ```

- A new `ui-import-rail.test.mjs` in `scripts/` — durable seeded-red suite. Owns the
  decision "what proves the rail's scope": ESLint-API `lintText` fixtures with
  virtual paths, asserting on `ruleId === 'no-restricted-imports'` findings
  only (fixtures use side-effect imports, so unrelated rules like jsdoc never
  fire).

Interface questions (Step 3): each const owns one nameable boundary decision;
callers (the config blocks) need only spread them; ownership stays in the one
config home ESLint already owns; no speculative surface (the consts have 5+
real spread sites in this PR); no caller-facing failure modes.

## 6. Design alternatives

- **Rule mechanism — chosen:** `no-restricted-imports` `patterns` threaded
  through the existing import-rail blocks, with `ignores` + dedicated
  re-state blocks for the wrappers (the exact `eve-image.tsx`/`esi-datasets`
  idiom). **Rejected:** `no-restricted-syntax` ImportDeclaration selectors —
  they would have to be re-listed across ~10 existing syntax-exemption blocks
  (versus 3–5 import blocks) and duplicate what the import rule natively does.
- **Exemption scope — chosen:** explicit enumeration — one block listing the
  14 Base UI wrapper files (sonner still banned there), one block for
  `toast.tsx` (Base UI still banned there). Adding a future wrapper is a
  deliberate, reviewable config edit, and both exclusivities stay mechanical.
  **Rejected:** exempting `src/components/ui/**` wholesale — explicitly ruled
  out by the contract; it would let any ui/ module (cn.ts, tones.ts, card.tsx…)
  import either package and would not preserve toast.tsx's sonner exclusivity.
- **Seeded-red evidence — chosen:** a durable vitest suite driving the real
  `eslint.config.mjs` through the ESLint API with virtual paths (mechanism
  proven live). **Rejected:** transient seeded files with captured lint
  output — matches earlier sessions' probes but is not durable; a future
  scope/glob regression would go unnoticed. The contract asks for a *durable*
  demonstration.

## 7. Ordered implementation steps

1. **eslint.config.mjs — add the rail.**
   - Add the three pattern consts (above) beside the selector families.
   - Base `src/**/*.{ts,tsx,mts}` import block: spread all three into
     `patterns`; add the 14 wrapper files + `toast.tsx` to its `ignores`.
   - New dedicated block `files: [<14 wrapper files>]`: re-state the
     `next/image` path ban, `**/staleness` pattern, deprecated-package and
     sonner patterns (Base UI allowed).
   - New dedicated block `files: ["src/components/ui/toast.tsx"]`: re-state
     `next/image`, `**/staleness`, deprecated-package and Base UI patterns
     (sonner allowed).
   - `src/lib/esi-datasets/**` and `src/app/api/cron/**` blocks: spread all
     three patterns (replace semantics — they would otherwise silently lose
     the new bans).
2. **scripts/ui-import-rail.test.mjs — durable seeded-red suite.**
   One `ESLint({ cwd: <repo root> })` instance per suite; `lintText` fixtures
   (side-effect imports, virtual paths), asserting presence/absence of
   `no-restricted-imports` findings whose message names the package:
   - RED: a virtual fixture path under `src/features/industry-planner/`
     importing `@base-ui/react/dialog`; same path importing `sonner`; same
     path importing `@base-ui-components/react`; `src/components/ui/toast.tsx`
     importing `@base-ui/react/dialog`; `src/components/ui/dialog.tsx`
     importing `sonner`; `src/components/ui/card.tsx` (non-wrapper ui file)
     importing `@base-ui/react/dialog`.
   - GREEN: `src/components/ui/dialog.tsx` importing `@base-ui/react/dialog`;
     `src/components/ui/toast.tsx` importing `sonner`.
   Update the `vitest.config.ts` include-comment to mention the lint-rail
   fixture suite alongside the script-helper tests.
