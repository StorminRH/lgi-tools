## 5. Interface designs

**`EveTokenOkResponse`** — narrowed to the one field its consumer reads.
Decision owned/hidden (P2): what the token-vend wire exposes. Draft comment:

```ts
/**
 * 200 — pinned with `satisfies` in the route; type-imported by convex/ in
 * 3.4.3. Carries ONLY the short-lived access token: the consumer's ESI reads
 * own their freshness windows, so no expiry/identity/scope metadata rides
 * this wire (PL-013).
 */
export interface EveTokenOkResponse {
  accessToken: string;
}
```

**`FreshTokenResult`** — ok-variant narrowed to what its two callers (route,
`vendTokenFor`) read. Existing comment stands:

```ts
/**
 * Token-vending verdict: a usable access token, a confirmed reauthentication requirement, or a
 * retryable upstream failure.
 */
export type FreshTokenResult =
  | { kind: 'ok'; accessToken: string }
  | { kind: 'not_found' }
  | { kind: 'reauth_required' }
  | { kind: 'upstream_error' };
```

Knock-on private simplifications inside `eve-token-service.ts` (no new
exports; P1 — the surface shrinks, the implementation stays rich):

- delete `parseScopes` and the module's scope comment block;
- `readCachedToken(row, characterId, scopes)` → `readCachedToken(row)`
  (`characterId` was only threaded into the result; the expiry guard logic is
  unchanged);
- `reflectStoredToken` and the fresh-refresh return stop carrying
  `expiresAt`/`characterId`/`scopes` (expiry **guards** remain — only the
  returned payload narrows);
- `loadAccountRow` drops the now-unread `scope` column from its select
  (`revokeCharacterToken` and all CAS logic never read it).

No new modules, hooks, or exports; every remaining field has a live caller
(P4); no caller-facing error case changes (P5).

## 6. Design alternatives

- **Chosen: producer-deep deletion.** Narrow the wire contract, the service
  result, and the `scope` column read together, deleting the divergent parser
  at its source.
- **Rejected: wire-only narrowing** (route stops echoing fields but
  `FreshTokenResult`/`parseScopes` survive). Keeps the exact defect PL-013
  targets — a wrong parser producing data nobody reads — one layer down;
  Fallow would also flag the newly-unread surface. Rationale seed: the
  deletion must reach the producer, or the dead knowledge merely moves.
- **Rejected: parser unification** (replace `parseScopes` with
  `tokenizeScopes` from `scope-health.ts`). Fixes the divergence but
  preserves a zero-consumer contract field — speculative surface P4 bans.
