## Interface designs

New module `db-test-harness.ts` under `src/db/test-support/` (promotion rename
of `db-coverage-harness.ts`; the old file is deleted when its last consumer
migrates — no compatibility façade). Public surface (6 exports, replacing the
6 old ones; `LOCAL_DB_URL`, `schemaUrl`, `canReachDb`, `setupDisposableSchema`,
`addForeignKey`, `dropDisposableSchema` all become module-private):

```ts
/**
 * Configuration for one real-Postgres suite. `schema` must be unique per test
 * file (schemas are dropped and recreated wholesale). `tables` are cloned from
 * the migrated local `public` schema via LIKE … INCLUDING ALL — parents first,
 * because `resetBetweenTests: 'delete'` wipes in reverse declaration order.
 * FKs are never copied by LIKE; list the load-bearing ones in `foreignKeys`.
 */
export interface DbTestHarnessOptions {
  schema: string;
  tables: readonly string[];
  foreignKeys?: readonly DbForeignKey[];
  /**
   * Steer the request-path `db` proxy (`@/db`) into the disposable schema by
   * stubbing LOCAL_DB_DRIVER/DATABASE_URL for the suite; teardown closes the
   * proxy's lazily created pool. Required whenever the code under test imports
   * the `db` singleton rather than accepting a database argument.
   */
  steerDbProxy?: boolean;
  /** Extra vi.stubEnv pairs the code under test reads (restored in teardown). */
  env?: Readonly<Record<string, string>>;
  /**
   * Per-test wipe of every cloned table: 'delete' removes rows in reverse
   * declaration order (child tables last in `tables` are cleared first);
   * 'truncate' issues one TRUNCATE … CASCADE. Omit for suites that manage
   * their own rows.
   */
  resetBetweenTests?: 'delete' | 'truncate';
}

/** One foreign key to re-add to the cloned tables (LIKE never copies FKs). */
export interface DbForeignKey {
  table: string; column: string; refTable: string; refColumn: string;
  onDelete: 'cascade';
}

/**
 * Live handles for a suite whose harness reached Postgres. `sql` and `db` are
 * getters that become valid once the harness's own beforeAll has run; reading
 * them earlier (or in a skipped suite) throws with a pointed message, so a
 * suite gated on `reachable` can never touch a half-built client.
 */
export interface DbTestHarness {
  /** False when no Postgres answered; gate the suite with describe.skipIf(!reachable). */
  readonly reachable: boolean;
  /** Admin postgres-js client steered into the disposable schema. */
  readonly sql: Sql;
  /** Drizzle instance over `sql`, for seeds and direct assertions. */
  readonly db: PostgresJsDatabase;
}

/**
 * Own one real-Postgres suite's entire DB lifecycle. Call once at test-file
 * top level (top-level await): probes reachability against
 * DATABASE_URL ?? the local Docker default, then registers the suite's
 * beforeAll (env stubbing, schema clone, FKs), optional beforeEach reset, and
 * afterAll (proxy pool close, schema drop, client close, env restore) —
 * every hook no-ops when the DB is unreachable. The disposable schema clones
 * the migrated local `public` schema; run `pnpm db:migrate` first. A suite
 * needs only this call, a describe.skipIf(!harness.reachable) gate, and its
 * seeds/assertions.
 */
export async function createDbTestHarness(
  options: DbTestHarnessOptions,
): Promise<DbTestHarness>;

/**
 * Insert one Better Auth `user` row with valid defaults (name/email derived
 * from `id`). Overrides win field-by-field; pass `activeCharacterId` or a
 * synthetic email through them when a scenario needs it.
 */
export async function seedUser(
  db: PostgresJsDatabase, id: string,
  overrides?: Partial<typeof user.$inferInsert>,
): Promise<void>;

/**
 * Insert one linked EVE `account` credential row for an existing user:
 * providerId 'eve', accountId String(characterId), timestamps now. Token
 * columns, ownerHash, and invalid-grant counters ride in `overrides` for the
 * suites that assert on them.
 */
export async function seedEveAccount(
  db: PostgresJsDatabase,
  base: { id: string; characterId: number; userId: string },
  overrides?: Partial<typeof account.$inferInsert>,
): Promise<void>;

/**
 * Insert one `characters` profile row with valid defaults (name/portrait
 * derived from the id); affiliation, role, and preference fields ride in
 * `overrides`.
 */
export async function seedCharacter(
  db: PostgresJsDatabase, characterId: number,
  overrides?: Partial<typeof characters.$inferInsert>,
): Promise<void>;
```

Step 3 answers:

1. **Decision owned/hidden (P2):** "how a test reaches a real Postgres safely"
   — probe semantics, schema steering, proxy teardown order, env hygiene,
   pool size (normalized to `max: 4`), reset ordering. Previously leaked into
   14 files.
2. **Surface vs implementation (P1):** one factory + one options object + two
   handles vs ~120 lines of lifecycle per suite. Callers learn 4 concepts
   (options, reachable-gate, `sql`/`db`, seed builders).
3. **Ownership:** `src/db/test-support/` — unzoned by design (documented in the
   existing module prologue); all slices may import it, as they already do.
4. **Real callers (P4):** every option value has ≥1 existing consumer
   (steerDbProxy: 10; env: 3; foreignKeys: 3; reset 'delete': 5,
   'truncate': 1); seed builders have 5–7/6/5 consumers.
5. **Errors defined away (P5):** unreachable DB → hooks no-op + `reachable`
   gate (never rejects); stale schema from a crashed run → drop-first
   self-heal (existing); forgotten proxy teardown/env restore → harness-owned;
   premature handle access → pointed throw.
6. **Non-signature knowledge (P7):** carried in the comments above — the
   migrated-`public` precondition, parents-first table ordering, why FKs need
   explicit listing.

Suites keep exactly: `const harness = await createDbTestHarness({…})`,
`describe.skipIf(!harness.reachable)`, their seeds (via builders or local
inserts), and assertions.

## Design alternatives

- **Chosen — lifecycle-owning factory:** `createDbTestHarness` registers
  vitest `beforeAll`/`beforeEach`/`afterAll` itself and returns lazy handles.
  Hides the whole duplicated preamble/teardown; a new suite is seeds +
  assertions, which is the contract's objective.
- **Rejected — expanded à-la-carte toolkit** (more helpers, suites keep
  assembling): smaller diff but preserves the observed duplication — the
  10-suite teardown block and env hygiene stay caller-owned, so the next
  campaign re-copies it; fails the contract's "stops hand-assembling setup".
- **Sub-decision:** seed builders are standalone functions taking `db`
  (chosen) rather than harness methods — only the 6 auth-family suites use
  them, so binding them to every harness instance would widen the common
  surface (P4/P8). Dataset-row seeds (usage_logs, corp_access_audit, GSC
  trio) stay suite-local: their inserted values are the assertion subjects,
  so a defaults-owning builder would hide exactly what each test asserts
  (contract §7 fixture-breadth decision: promote the identity trio only).

## Ordered implementation steps

Contract §7 commit-granularity decision: **one behavior-preserving commit per
suite (P9 default)** — each is small, mechanically similar, and independently
green; family order A → B → C so the simplest consumers prove the factory
first.

1. **Harness commit:** add `db-test-harness.ts` (factory + fixtures, internals
   absorbed from `db-coverage-harness.ts`) and migrate the first Pattern A
   suite (`esi-snapshot-retention.db.test.ts`) onto it in the same commit so
   the new exports are born with a consumer.
2. **Commits 2–4 (family A):** table-retention, universe, esi-refresh-jobs
   (its raw `process.env` writes move onto harness `env` stubbing — same
   values, now restored).
3. **Commits 5–8 (family B):** eve-data, gsc, telemetry, owned-structures
   `queries.db.test.ts` with `steerDbProxy: true`.
4. **Commits 9–14 (family C):** linked-characters, admin-users,
   owner-transfer, affiliation-store, eve-token-service.concurrency, then
   account-purge (`resetBetweenTests`, `foreignKeys`, `env` as each suite
   already uses); the account-purge commit also **deletes
   `db-coverage-harness.ts`** (last consumer gone; zero old-path call sites).
5. **Rail commit:** new flat-config block in `eslint.config.mjs` scoped
   `files: ["src/**/*.db.test.ts"]` re-listing the base selector families
   (`cspSelectors`, `hexColorSelectors`, `rgbaColorSelectors`,
   `apiFetchSelectors` — replace semantics are documented in-file) plus two
   new named selector consts: `CallExpression[callee.name='postgres']` and a
   postgres connection-string literal/template ban, each with a message
   pointing to the promoted harness module under `src/db/test-support/`. The
   harness module
   itself is not matched by the glob, so its `LOCAL_DB_URL` constant and
   `postgres()` calls stay legal.
6. **Agent-guide commit:** add the DB-testing bullet to `AGENTS.md` Testing
   policy naming the harness as the standard (draft: real-Postgres suites are
   `*.db.test.ts` on `createDbTestHarness`, which owns skip-gating, the
   disposable-schema clone of the migrated local `public` schema, proxy
   steering, seeds, and teardown; direct `postgres()`/connection strings in
   suites are lint-banned). Run `python3 .agent-local/check_agent_drift.py`
   and resolve findings (revision bump only if the gate demands it).
7. **Changelog + `APP_VERSION` → 3.9.2.1** in `content/changelog/v3.9.md`.
8. Close-out per contract §12 (single-session sub-version: full close-out).
