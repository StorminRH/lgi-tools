## Context

The three close-out rituals that still rely on read-back-and-hope — release
consistency, PR-body privacy, and the archive transition — get mechanical
verification. Folded in per operator direction: a hotfix for the post-merge
"shipped" reconciliation quirk. The operator selected **close-out,
mechanized**: authorship stays at close-out immediately after merge (evidence
in-context), and the quirk is fixed by machine-verifying the state before the
PR and after the reconciliation commit, plus stating explicitly that the
reconciliation branch is named by the post-merge resolver rerun — no more
agent "opting".

Key facts reconciled against live state:

- `.agent-local/` is **tracked** (since 3.9.1.4). The contract's proposal to
  keep the privacy pattern list "in the gitignored script itself" rests on a
  stale premise — a tracked instance list would itself be the leak. Resolved
  below with runtime-derived identifiers plus an untracked extension file.
- The "merged-but-unreconciled" state is file-identical to normal mid-branch
  state (`APP_VERSION` == newest changelog heading == first nonterminal
  roadmap row). One checker therefore validates **two named signatures**, and
  call sites pin which one they require.

## 5. Interface designs

### 5a. `checker_common.run_checker` extension (`.agent-local/checker_common.py`)

Add optional `add_arguments: Callable[[argparse.ArgumentParser], None] | None
= None`. When provided, `collect` is invoked as `collect(root, args)`;
existing one-arg checkers are untouched. Draft docstring addition:

```python
def run_checker(collect, argv=None, add_arguments=None):
    """Run one checker CLI and block under --check only when errors are present.

    add_arguments, when given, registers script-specific flags on the shared
    parser and switches collect to the two-argument form collect(root, args).
    The CLI contract (--root/--check/--pretty, errors/warnings JSON) stays
    owned here; scripts never build a sibling parser.
    """
```

- Decision owned/hidden (P2): the checker CLI contract keeps exactly one
  owner.
- Callers: the two new arg-taking scripts (real consumers in this PR — P4).

### 5b. `check_release_consistency.py` (new, tracked)

```python
"""Cross-check the release identity triplet before a PR and after reconciliation.

Owns one decision: whether APP_VERSION (src/config/app-version.ts), the newest
entry heading in the active version's content/changelog/vX.Y.md, and the active
roadmap's delivery rows tell one consistent story. Exactly two signatures are
valid — 'pre-pr' (the triplet names the first nonterminal row, all prior rows
terminal) and 'reconciled' (the triplet names the latest terminal row);
--expect {pre-pr,reconciled} pins which one a call site requires. Any other
combination, or APP_VERSION != newest changelog heading, is an error.
Read-only; roadmap parsing is imported from resolve_development_state.
"""
```

- Flags: shared set + `--expect {pre-pr,reconciled}` (optional; absent =
  either signature passes).
- Parsing: `APP_VERSION` via one regex over `src/config/app-version.ts`;
  roadmap via resolver imports; changelog newest `### v<version> — date`
  heading in the active version's master file. No new markdown parser (P2 —
  the resolver owns roadmap parsing).
- Errors (each anchored to file:line): APP_VERSION ≠ newest changelog heading;
  triplet matches neither signature; changelog entry missing for the version
  APP_VERSION names; roadmap rows out of terminal-prefix order; `--expect`
  given and the other signature observed.
- Call sites: PR_REVIEW Step 1 (`--check --expect pre-pr` before
  `gh pr create`), post-reconciliation (`--check --expect reconciled`),
  `start-session` at session start and the drift gate (no `--expect` — both
  signatures are legitimate rest states). This is the hotfix's mechanical
  detection.

### 5c. `scrub_pr_body.py` (new, tracked)

```python
"""Block personal identifiers from reaching a public PR title or body.

Owns one decision: whether a candidate PR body-file (and optional --title
string) contains operator-identifying or credential-shaped content per the
PR_REVIEW privacy rule. Three pattern sources: (1) generic shape regexes
tracked here (emails, home-directory paths, machine hostnames, credential-
shaped tokens); (2) operator identifiers derived at runtime — git user.name /
user.email and its local-part, $USER, $HOME leaf, platform hostname — so no
personal instance list is ever tracked; (3) the untracked extension file
.agent-local/pr-privacy-local-patterns.txt (one regex per line, # comments,
absent = fine). Every match is an error anchored to the body-file line;
publish is blocked under --check. Read-only.
"""
```

- Flags: shared set + `--body-file PATH` (required) + `--title STR`
  (optional).
- The documented extension point is the pattern registry constant plus the
  local file. `.gitignore` gains
  `.agent-local/pr-privacy-local-patterns.txt`; the session creates the real
  local file (untracked) with the operator's identifiers; AGENTS.md's
  deliberately-ignored local-state list names it.
- Errors define the failure away for callers (P5): close-out runs one command;
  no judgment call about what counts as personal.

### 5d. `verify_archive.py` (new, tracked)

```python
"""Verify the version-archive transition of DEVELOPMENT_LIFECYCLE §7.

Owns one decision: whether the active version may be archived and whether the
copied bundle is faithful. --phase pre checks preconditions 1-4 (every roadmap
row terminal; audit plan 'Audit status: Complete'; every Floss/Campaign finding
Verified with no new actionable finding in the current cycle; the baseline
names the audited version and exact Audited ref). --phase post additionally
requires the destination bundle under --archive-root/versions/X.Y/ to contain
the plan, contract set, session plans, and audit plan byte-identical (sha256)
to the active copies, reporting each missing or differing file. --archive-root
defaults to the documented sibling '../LGI Tools Document Archive'.
Verification only; the copy and any deletion stay with the skill/operator.
"""
```

- Flags: shared set + `--phase {pre,post}` (default `pre`) + `--archive-root
  PATH` (default derived from repo root). **Contract decision resolved:**
  configured default with an override flag — the path is stable and
  documented; requiring it at every call site pushes complexity up (P5).
- Version under test derived from the active roadmap; findings/audit parsing
  imported from the resolver.
- Preconditions 6–7 (SCRATCHPAD/pointer updates, resolver + drift run) stay
  procedural — they are actions, not verifiable file states; the docs keep
  them as named human/skill steps.

## 6. Design alternatives (design-it-twice records)

- **Privacy patterns:** chosen = tracked shape regexes + runtime-derived
  operator identifiers + untracked extension file. Rejected: a tracked
  instance list (the list itself becomes the published leak — the contract's
  gitignored premise is stale); an untracked script (escapes review and drift
  parity that 3.9.1.4 deliberately established).
- **Release consistency:** chosen = one checker with two named signatures +
  `--expect`. Rejected: separate pre-PR and post-merge scripts — duplicate
  parsing, two rituals, and the drift gate could run neither safely.
- **Archive root:** chosen = documented default + `--archive-root` override.
  Rejected: required argument — every caller re-states a stable fact.
- **Reconciliation timing (operator-decided):** chosen = keep at close-out,
  mechanized. Rejected: move to fresh session start — the unreconciled state
  is file-indistinguishable from mid-branch work, so detection would need
  git/GitHub evidence in the resolver, and the shipped-ledger entry would be
  reconstructed lossily.
- **CLI args:** chosen = extend `run_checker` with optional `add_arguments`.
  Rejected: per-script bespoke parsers — the CLI contract would grow a second
  owner (P2 violation).
