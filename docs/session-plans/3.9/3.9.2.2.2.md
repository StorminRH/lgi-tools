## Goal

The `drain-esi-refresh-jobs` cron becomes the proving `idle-silent` consumer:
an Upstash Redis pending-work signal lets its healthy no-op runs touch zero
Neon, `runCronJob` is deleted, and the 3.9.2.2 sub-version closes through the

## Context

The drain wakes Neon 96×/day on an idle deploy: `runCronJob` opens the direct
connection for the advisory lock every run,
`maybeAlertPublicEsiBudgetExhaustion()` runs a Neon telemetry count every run
(~always zero), and the drain queries the queue even when empty. Session .1
built the `defineCronRoute` shell; this session adds the shell's idle stage,
the Redis pending-work and exhaustion signals, the daily Neon-backed heal
slot, and the drain's declaration migration.

## Interface designs (draft interface comments)

**1. New `upstash.ts` under `src/lib/` (prep refactor — third consumer of
the dual-env decision):**

```ts
/**
 * Resolves the Upstash Redis REST credentials from either provisioning path —
 * `KV_REST_API_*` (Vercel marketplace, preferred) or `UPSTASH_REDIS_REST_*`
 * (direct signup) — or null when neither pair is configured. One owner for the
 * dual-name decision; callers own their unconfigured behavior (the ESI gate
 * fails closed in production, the pending-work signal degrades to unknown).
 */
export function resolveUpstashRest(): { url: string; token: string } | null;
```

Adopted by `scoreboard/index.ts`, `rate-limit.ts` (both behavior-preserving),
and the new signal module.

**2. New `pending-signal.ts` under `src/data/esi-refresh-jobs/` (key
`lgi:esi-refresh:next-due`; value = epoch-ms of the earliest live job's
`nextAttemptAt`):**

```ts
/**
 * Lowers the pending-work signal to `dueAt` if it is earlier than the stored
 * value (atomic write-if-lower), so concurrent enqueues can only pull the next
 * wake time forward. Called by the queue insert with Neon already awake; a
 * missing Redis configuration or write failure is swallowed — the daily heal
 * run bounds the damage of a lost signal at ~24h.
 */
export function advancePendingWorkSignal(dueAt: Date): Promise<void>;

/**
 * Replaces the signal with the post-drain residual truth: the earliest live
 * `nextAttemptAt`, or clears it when the queue holds no live jobs. Runs at the
 * end of every drain that reached Neon, which also self-heals a stale or lost
 * signal.
 */
export function writeBackPendingWorkSignal(earliest: Date | null): Promise<void>;

/**
 * Answers the drain's pre-Neon idle question from Redis alone: `due` when the
 * stored next-due time has arrived, `idle` when the queue is empty or its
 * earliest retry is still in the future (a retry due in hours must not wake
 * Neon every 15 minutes), `unknown` when Redis is unconfigured or unreachable
 * — callers treat unknown as proceed.
 */
export function readPendingWorkSignal(now: Date): Promise<'due' | 'idle' | 'unknown'>;
```

**3. `src/data/esi-refresh-jobs/queries.ts` (one new export):**

```ts
/**
 * Post-drain residual: how many live jobs are already due and the earliest
 * `nextAttemptAt` across every live (non-terminal) job, or null when none
 * remain. Feeds the pending-work signal write-back and the drain's telemetry.
 */
export function getEsiRefreshQueueResidual(
  now?: Date, database?: AnyPgDb,
): Promise<{ dueCount: number; earliestNextAttemptAt: Date | null }>;
```

**4. New `exhaustion-marker.ts` under `src/lib/esi/`:**

```ts
/**
 * Fire-and-forget short-TTL marker (≈35 min, two alert windows plus slack)
 * recording that the public ESI budget refused a request just now. Set at the
 * refusal origin so the drain cron can ask "did anything happen?" without a
 * Neon telemetry query. Failures are swallowed; the marker is an optimization
 * hint, never a record.
 */
export function markRecentBudgetExhaustion(): void;

/**
 * Whether a budget refusal happened within the marker TTL: `false` lets the
 * drain skip the Neon alert-window count entirely; `unknown` (Redis
 * unconfigured/unreachable) means proceed to Neon, matching the Redis-optional
 * convention.
 */
export function hasRecentBudgetExhaustion(): Promise<boolean | 'unknown'>;
```

`markRecentBudgetExhaustion()` is called once in `enforceBudget` at the throw
path (single origin, all four refusal reasons).

**5. Shell (`src/db/cron-gate.ts`) — one new optional declaration field:**

```ts
/**
 * Pre-Neon idle stage for idle-silent declarations. The shell runs `probe`
 * after auth and before preLock, the advisory lock, and work — the probe must
 * not touch Neon. An idle verdict finishes the run with `body` and the
 * boundary line (durable telemetry still governed by the recording policy, so
 * a noteworthy idle run stays durable-silent); a non-idle verdict proceeds to
 * the normal stages. Unknown probe state must map to `{ idle: false }`.
 */
idle?: {
  probe: () => Promise<{ idle: true; telemetry?: Record<string, unknown> } | { idle: false }>;
  body: (durationMs: number) => Body;
};
```

Ordering becomes: auth → **idle probe** → preLock → advisory lock → work →
telemetry. `runCronJob` and its tests are deleted once the drain migrates.

**6. Drain declaration (`drain-esi-refresh-jobs/declaration.ts` + thin
route, sweeper pattern):** `wakeClass: 'idle-silent'`,
`record: { policy: 'noteworthy' }`, existing advisory lock key, new action
`'cron_esi_refresh_jobs'`. Probe: heal window → proceed; exhaustion marker
true/unknown → proceed; signal due/unknown → proceed; else idle. Work:
marker-gated budget alert (inside `maybeAlertPublicEsiBudgetExhaustion`,
which gains a marker precheck returning a new
`{ status: 'no-recent-exhaustion' }` variant), then `drainEsiRefreshJobs()`,
which now ends with residual query + `writeBackPendingWorkSignal`. A pure,
tested `isDailyHealWindow(now)` (UTC hour 12, minute < 15 — **the 12:00 UTC
run is the daily heal slot**, riding the post-11:20–11:50-batch warmth rather
than a cold midnight wake) lives beside the declaration.

**7. Enqueue-side write:** inside `enqueueEsiRefreshJob` after the
insert/coalesce — awaited, error-swallowed
`advancePendingWorkSignal(nextAttemptAt)`.

## Design alternatives (design-it-twice)

- **Signal shape — selected: earliest-`nextAttemptAt` timestamp.** Rejected:
  flag/count — neither can express "pending but not yet due", so a single
  retry due in hours would wake Neon every 15 minutes until it matured (the
  contract's thrash case). The timestamp answers idle-vs-due from Redis
  alone.
- **Signal write owner — selected: inside `enqueueEsiRefreshJob`.** The
  insert owns "job entered the queue ⇒ signal reflects it"; no future call
  site can forget it (P2/P5). Rejected: writes at the `enqueueBudgetDeferral`
  callback — leaks the must-signal rule to every present and future caller.
- **Probe wiring — selected: declaration-owned probe behind a shell `idle`
  slot.** The shell keeps the ordering guarantee while staying
  Redis-ignorant; the sweeper legitimately stays probe-free. Rejected: shell
  reads the signal itself — binds `src/db` to one data slice's Redis concern
  and forces probe semantics on every idle-silent route.
- **Env resolution — selected: shared `lib/upstash.ts`** (three real
  consumers; one owner for the dual-name decision). Rejected: a third local
  copy of the same 6-line decision.
