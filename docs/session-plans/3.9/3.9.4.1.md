## 5. LGI-08 — diagnosis, role design, drill

### 5.1 Diagnosis target: a disposable Neon child branch — never production

`neon branches create` a fresh child of the production branch; child branches
copy the parent's databases, roles, and data, so role attributes, ownerships,
and grants are faithfully inspectable there. All `psql` diagnosis runs against
that branch. **Branch creation and deletion are shared-state actions requiring
point-of-action operator approval** (not mere disclosure); cleanup is an
**explicit deletion step** — `neon.ts` never auto-applies, so no TTL is relied
on. Which role identity each production URL actually uses is established by
(a) an operator-mode, redaction-safe read of just the username from the
connection settings (Ryan confirms; values never enter the transcript) and
(b) `current_user`/`session_user` probes on the child branch's matching
endpoints. No production SQL, ever.

### 5.2 Deliverable: actor/credential/capability matrix

| Actor path | Credential today | Needs |
| --- | --- | --- |
| Request path (neon-http, pooled) | `DATABASE_URL` | DML on app tables + sequences |
| Cron/direct (advisory locks, unpooled) | `DATABASE_URL_UNPOOLED` | DML + session advisory locks + **TRUNCATE on the enumerated regenerable SDE tables only** |
| Migrations (build-time) | `DATABASE_URL` | DDL (owner) |
| Build bootstraps (`ingest-sde-if-empty`, `backfill-users-if-empty`) | `DATABASE_URL_UNPOOLED` | DML + SDE TRUNCATE + session advisory locks |
| Operator maintenance | operator | full |

For each user/character-keyed table: what the runtime principal can read/write
across tenants today (answer: everything — one shared identity, no RLS), and
the concrete blast radius of a runtime-credential leak before vs. after the
role split.

**Explicit verified non-path (Ryan's 2026-07-19 question):** the risk model
leads by confirming that **no client-reachable database surface exists** — the
browser never queries Neon directly; `neon.ts` declares no Data API service
and the diagnosis verifies none is enabled on the project; all DB access is
server-side behind Better Auth + Zod + Drizzle parameterization. The
Supabase-style "user reaches the DB through frontend queries" attack path is
recorded as verified-absent, and keeping it absent (never enabling the Data
API without an RLS decision) is a standing runbook invariant. The residual
risks addressed are therefore (a) backend-mistake cross-tenant leaks — the
deferred RLS campaign's territory — and (b) credential blast radius — this
session's role split.

### 5.3 Honest threat claim

Because migrations run in the Vercel build, a `DATABASE_MIGRATION_URL` project
env var remains readable by deployed functions; the split therefore does
**not** protect against full application-runtime compromise (an attacker
reading `process.env` gets the owner URL). The risk model states this
explicitly. What the split *does* buy: the high-exposure runtime **DB
credential** (used per-request over HTTP on two paths) becomes
least-privilege — containing credential-leak scenarios, accidental/app-logic
DDL, and SQL-injection escalation through the request path. Isolating the
owner credential from the deployed runtime entirely (separately authenticated
migration job outside `vercel-build`) is recorded in the runbook as the
stronger future option, reconciled against the Vercel-only
production-build policy.

### 5.4 Role/grant ownership — selected design

**Hybrid: migration-owned privilege role + operator-owned login credential.**

- One drizzle migration creates a **`NOLOGIN` privilege role** (e.g.
  `lgi_runtime`) and owns its grants: `USAGE ON SCHEMA public`,
  `SELECT/INSERT/UPDATE/DELETE` on application tables, `USAGE` on sequences,
  **table-scoped `TRUNCATE` on the enumerated regenerable SDE tables** (exact
  target list + `CASCADE` closure named in the migration), and
  `ALTER DEFAULT PRIVILEGES` with **separate future-table and future-sequence
  clauses, pinned to the verified migration-creator role** (default ACLs apply
  only to objects created by that exact role — verified in the drill). The
  `DO` block is idempotent but **fail-closed**: if `lgi_runtime` already
  exists with unsafe attributes (`LOGIN`, `CREATEROLE`, `BYPASSRLS`,
  superuser membership), the migration errors rather than silently accepting
  it.
- The operator creates the **LOGIN credential via SQL** (as the owner:
  `CREATE ROLE … LOGIN NOSUPERUSER NOCREATEDB NOCREATEROLE NOBYPASSRLS IN
  ROLE lgi_runtime`), with the password set through a mechanism that never
  enters git, the transcript, or shell history. **Not** `neon roles create`:
  Neon Console/API/CLI-created roles automatically receive `neon_superuser`
  (broad read/write + `CREATEROLE` + `BYPASSRLS`), which membership in
  `lgi_runtime` would not remove — that route would silently defeat the
  split. Custody, rotation, and cutover/rollback live in the runbook.
- `neon.ts` gains only a pointer comment.

### 5.5 Drill and probe matrix on the child branch

The DB test harness cannot run under the runtime role (it creates/drops
schemas by design — `db-test-harness.ts:210`), so validation is a dedicated
probe script run twice (owner URL = control, runtime URL = subject):

**Identity/attribute assertions first:** `pg_roles` attributes,
`pg_auth_members` memberships (no superuser/`neon_superuser` inheritance),
`information_schema.role_table_grants`, `pg_default_acl`, and
`current_user`/`session_user` on both (pooled/unpooled) endpoints.
**Positive (must succeed as runtime role):** representative request-path reads
(sites catalogue, planner queries), auth-data read/write, sequence nextval,
session advisory lock acquire/release on the unpooled endpoint, the SDE
refresh pipeline including its enumerated TRUNCATE set, and both build
bootstraps **with their write branches forced** (seeded empty-table
conditions), not their no-op branches. A **newly owner-created table and
identity sequence** must be readable/writable — proving the default ACLs
actually fire. Migrations succeed under the owner URL.
**Negative (must fail as runtime role):** `CREATE/ALTER/DROP` table,
`TRUNCATE` on any non-SDE table (e.g. an auth table), `GRANT`/`REVOKE`,
`CREATE ROLE`/`SET ROLE` to owner, schema/database-level DDL, and disabling
or bypassing RLS attributes. **`CREATE TEMP TABLE` is tested and documented,
not assumed to fail:** PostgreSQL grants database `TEMPORARY` to `PUBLIC` by
default; the drill decides (revoke vs. accepted-and-recorded).
Timed readouts for both runs are the runbook evidence. **Honest claim scope:**
the drill proves the *probed DB-privilege surface*, not the entire deployed
build/request surface; a child-backed Vercel preview is recorded as the
optional stronger check (operator-gated), not required.

### 5.6 Cutover status — explicit

Two distinct production moments:

- **Privilege-policy deployment happens at merge automatically:** the grants
  migration runs inside the production `vercel-build` like every migration.
  It is additive and inert pre-cutover (a `NOLOGIN` role + grants; no
  existing credential changes), which is why it may ship in the PR — disclosed
  there as the deliberate privilege-policy-deploy moment.
- **Credential cutover is NOT a done condition and never automatic:** creating
  the LOGIN role in production and repointing Vercel's
  `DATABASE_URL`/`DATABASE_URL_UNPOOLED` (+ new `DATABASE_MIGRATION_URL` →
  owner) is Ryan's explicit, point-of-action go-ahead, with the runbook's
  rollback (repoint env vars back). If green-lit within the session window,
  the post-cutover close-out smoke doubles as verification.

### 5.7 RLS: deferred, with an honest rationale

RLS policies are deferred **not** because the driver is incapable — the
neon-http adapter's `batch()` runs real non-interactive transactions that
could carry `set_config`-style context — but because identity propagation
(wrapping every protected query with verified per-request identity) is an
app-wide design campaign touching every query call site. That campaign, with
the report's §G/H as seed and batch-transactions recorded as a viable
mechanism, routes to the backlog for a future version alongside the
"defense-in-depth vs. formal tenant boundary" decision.

## 8. Interface designs

**`resolveMigrationUrl(env)` — new pure export** (home: a new migrate-url
module in `src/db/`, imported by `migrate.ts`):

1. *Decision owned:* "which credential runs DDL" — one home; the runtime
   proxy never consults it. (P2)
2. *Surface:* one pure `(env) => string` function; simpler than the policy it
   encodes. (P1)
3. *Zone:* `src/db/`; imports nothing above `lib`. Legal.
4. *Callers:* `migrate.ts` + its test. Real, present. (P4)
5. *Failures absorbed:* unset **or empty** `DATABASE_MIGRATION_URL` → fall
   back to `DATABASE_URL` (empty-as-missing semantics, matching the registry
   category where the key is registered); missing both → throw with a clear
   message. Local/dev and un-migrated envs unaffected. (P5)
6. *Comment-carried invariant:* runtime `DATABASE_URL` must be the
   least-privilege role; DDL uses the owner credential here. (P7)

Draft interface comment:
```ts
/**
 * Chooses the connection string for schema migrations (DDL). Prefers
 * DATABASE_MIGRATION_URL — the schema-owner role — and treats unset or empty
 * as missing, falling back to DATABASE_URL so local/dev and single-role
 * environments behave exactly as before the role split. The request path
 * (src/db/index.ts) must keep DATABASE_URL on the least-privilege runtime
 * role; this is the only sanctioned owner-credential read in src/.
 */
```

The grants migration and probe script are execution artifacts, not exported
code surfaces; the migration carries a prologue comment naming the role model

## 9. Design alternatives

**Role/grant ownership:**
- **Selected — migration-owned `NOLOGIN` privilege role + operator login
  credential** (§5.4): grants versioned/idempotent/drift-free (default
  privileges cover future tables), credential custody operator-side.
- **Rejected — runbook-only operator SQL:** no drift detection, future tables
  silently uncovered, grant history invisible to review. (Also rejected, and
  verified impossible: `neon.ts` declaration — `@neondatabase/config` 0.4.2
  has no role/grant fields; and coupling role state to `neon config apply`
  would be wrong blast-radius coupling even if it existed.)

**Verification structure:**
- **Selected — parallel finding-cluster workers → main-session
  reconciliation:** fast, verdicts stay with the session.
- **Rejected — one sequential deep worker:** slower, single-pass trust.
