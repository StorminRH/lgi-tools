## 5. Interface designs

**(a) `src/features/wormhole-sites/dev-sample.ts` — one export.**

```ts
/**
 * Dev-only representative catalogue sample for fast /sites visual iteration.
 * Returns the reduced site list when sample mode is active, or null when it is
 * not. Activation requires BOTH `NODE_ENV === 'development'` — true only under
 * `next dev`; production builds and the Vercel runtime run with
 * `NODE_ENV=production`, where bundlers inline the literal and the branch is
 * statically dead — AND the explicit `LGI_SITES_SAMPLE=1` opt-in from the env
 * registry. The mode is impossible to enable in production by construction.
 * Selection is input-order-independent and deterministic: candidates are
 * canonicalized by ascending id, and one site is kept for every
 * (site type × wormhole class) pair present in the full catalogue — classes
 * via siteClassSet, class-less sites contributing a (type, none) pair — so
 * every filter combination the /sites rail can express against the full
 * catalogue stays non-empty in the sample. The returned array preserves the
 * caller's input order. Callers own the visible labeling of the mode.
 */
export function selectDevSampleSites(sites: SiteDetail[]): SiteDetail[] | null;
```

Step 3 answers (stable since v2): the module owns both the activation and
representativeness decisions (P2); one nullable-return function (P1); zone
`features/wormhole-sites`, feature → lib import direction legal; single real
caller `page.tsx`, no speculative parameters (P4); no caller-visible error
states (P5); production-impossibility + coverage rule carried in the comment
(P7).

**Representativeness:** pairwise (type × class) coverage — marginal per-axis
coverage would leave AND-across-axes filter combinations falsely empty.
**Pinned current expectation (round-3 live-catalogue verification): 25 distinct
pair keys over the 69-site catalogue, deterministic first-by-id union = 20
sites; all twelve class-less entries are ore sites.** The suite reports actual
counts; tests lock the general invariant, not the constant.

**(b) `src/lib/env.ts` + `.env.example`** — `LGI_SITES_SAMPLE: verbatim`
(`=== '1'` call site) + commented example line documenting dev-only semantics.

**(c) `src/app/sites/page.tsx`** — page-internal only:

```ts
const loadAllSites = cache(async () => {
  const raw = await listSiteDetails({});
  const sample = selectDevSampleSites(raw);
  const sites = await overlayLivePrices(sample ?? raw);
  return { sites, fullCount: raw.length, sampled: sample !== null };
});
```

When `sampled`, `SitesContent` renders above the filter layout — the marker on
a route-owned wrapper, the primitive untouched:

```tsx
<div data-dev-sample={`${sites.length}/${fullCount}`} className="mb-4">
  <Banner tone="warn">
    DEV SAMPLE MODE — showing {sites.length} of {fullCount} sites (LGI_SITES_SAMPLE=1)
  </Banner>
</div>
```

`Banner` is `'use client'` but *rendered* only in dev; production `/sites`
HTML/RSC payload never references it (client components enter the payload only
when rendered; the §2 interpretation covers the artifact residual).

**(d) Profiler/suite measurement contract (rounds 2–3 reconciled against the
live layout):** `SitesFilterLayout` mounts **either** cards **or** table — one
surface in the DOM at a time — and the profiler's anonymous request carries no
view cookie, so its drained body renders the default **cards** view only.
Profiler measurements therefore exercise cards; the Playwright probe (run in
both modes) owns both-surface evidence. The drained document is hybrid
HTML + RSC flight data, and the *unrendered* table node is serialized in the
flight scripts — so counts must be **tag-scoped in rendered markup, excluding
`<script>` content**, never raw substring hits. `profile-parse.mjs` helpers:

- `extractCardCount(html)` — tag-scoped `[data-lazy-details]` element count in
  rendered (non-`<script>`) markup;
- `extractDevSampleMarker(html)` — `{present, shown, total}` from
  `data-dev-sample="N/M"` in rendered markup;
- `assessModeExpectation({cardCount, marker, sampleEnv, expectedFullCount})` —
  the pinned verdict with an **independent full-catalogue oracle**
  (`expectedFullCount: 69`, pinned by the suite and recorded in each result):
  sample runs must show marker present, `shown < total`,
  `total === expectedFullCount`, `cardCount === shown`; non-sample runs must
  show marker absent and `cardCount === expectedFullCount`. Marker-absent
  *reduced* counts fail. Recorded as `modeCheck: pass | fail`; **a `fail` is a
  failed suite outcome, not an informal inspection.**
- suite aggregation helpers — medians/ranges over the six runs, the §7 step 10
  completion-gate verdicts (`modeCheck` roll-up, `< 60 s` cold criterion,
  no-regression bound).

**(e) `scripts/profile-sites-suite.mjs`** — the six-run orchestrator (§3):
counterbalanced label order `after-1, sample-1, sample-2, after-2, after-3,
sample-3` (cancels monotonic temporal drift; alternation alone always measures
sample second), `rm -rf .next` before each run, env injection for sample runs,
per-run precondition capture, aggregation via the tested helpers, one suite
JSON on stdout plus the gitignored file (the .1 output conventions; labels fit
the existing `^[a-z0-9-]{1,32}$` grammar; the per-run profiler is unchanged in
cadence/thresholds/teardown).

**(f) `package.json`** — `"dev:sample": "LGI_SITES_SAMPLE=1 next dev"`,
`"profile:sites-suite": "node scripts/profile-sites-suite.mjs"`.

## 6. Design alternatives (design-it-twice records)

1. **Sample seam — chosen: post-query filter in the page composition.**
   *Rejected:* sampling inside `listSiteDetails` — feeds the AF-003 measured
   surface, makes `'use cache'` content env-dependent, forces every consumer to
   know the mode. Rounds 2–3 concurred the seam is defensible; efficacy is
   measured, not assumed (§7 step 10).
2. **Gating — chosen: static `NODE_ENV === 'development'` short-circuit + a
   registered explicit opt-in env var.** *Rejected:* a separate dev-only route
   (surface + classification burden + drift); a UI/cookie toggle (reachable
   from production surfaces).
3. **Sample data shape — chosen: deterministic pairwise-representative subset
   of the real catalogue.** *Rejected:* synthetic fixtures (P2 leak, drift);
   marginal per-axis coverage (falsely empty filter combinations).
4. **Surface shape — chosen: one export returning `SiteDetail[] | null`.**
   *Rejected:* separate `isActive()` + `select()` exports (wider surface, P1).
5. **Marker placement — chosen: route-owned wrapper div; `Banner` untouched.**
   *Rejected:* widening `Banner` with rest-prop forwarding — a shared-primitive
   interface change (role/semantics override risk) for one dev-only caller.
6. **Production-absence proof — chosen: enablement-impossibility (layered §8
   evidence) with the residual stated.** *Rejected:* Vercel post-build artifact
   assertion — post-merge-only failure mode, unverifiable pre-merge under the
   no-local-prod-build rule, minifier-dependent. **Declined by Ryan at
   approval; one-time production check instead (§8).**
7. **Measurement execution — chosen: a bounded suite orchestrator over the
   unchanged per-run profiler.** *Rejected:* six manual invocations with
   hand-computed medians — the readout controls backlog deletion, so its
   aggregation and gates must be mechanical and reproducible (also the standing
   looping-work-becomes-a-script rule).
