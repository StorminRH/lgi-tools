## Design decision: cover through existing seams ŌĆö no production-logic edits

Editing a production file is what would expose its functions to CI's
origin/main-pinned Fallow gate, and every target already has a public seam that
yields honest coverage without touching the flagged modules:

- `runIngest(db, opts)` accepts an **injected database** (`ingest.ts:90`). Mocking
  `./source` (`downloadSdeJsonl`ŌåÆfixture paths, `cleanupSdeJsonl`ŌåÆnoop) and
  `./universe` (`parseUniverse`ŌåÆempty six-array dataset, `emitUniverseNeon`ŌåÆa
  **zero-count summary object**, not a bare noop) lets a DB-harness test drive the
  **real** `streamInsert` + **real** mapRow arrows + **real** inserts over tiny
  fixture JSONL. Covers all three ingest findings at once with real behavior.
- `getSkillsForUserOnView` composes `nameIds` through the **real**
  `getLiveDatasetOnView`, which invokes `nameIds(rows)` then `getTypeNames(...)`
  (`live-dataset-view.ts:82-87`). A unit test mocks only the **leaves** (character
  enumeration, the cached-data/sync-state reads, `getTypeNames`, and
  `next/server.after` so the write-behind records-but-doesn't-run) and asserts the
  **public** `characters`/`names` result plus the exact id set `getTypeNames`
  receives ŌĆö real behavior, not a captured internal callback. Postgres-independent.

The contract explicitly prefers harness coverage for the ingest functions and
permits extraction "only if that makes a function honestly testable"
(`3.9.5.2.md:47,61`). Since honest seams exist, **nothing is extracted** ŌĆö no
`stream-insert.ts`, no `skill-name-ids.ts`, no exported mapRows. This removes the
P9 "characterize-before-restructure" problem entirely and keeps the session to
**test files + required release identity/changelog + the approved lifecycle plan**.

**Why this is CI-safe.** CI (`.github/workflows/test.yml`) runs `pnpm test:coverage`
with **no Postgres service** and pins Fallow to the **PR base (origin/main)**, not
the version-start SHA. Because the four flagged production modules remain
byte-unchanged, CI never attributes any target function as introduced ŌĆö regardless
of Postgres. Coverage for the five DB-bound findings is supplied **locally**, where
Postgres is reachable, for the two gates the contract requires (version-start
291ee78 + close-out origin/main); the GitHub CI run is the proof for the no-Postgres
environment. `nameIds` gets a Postgres-independent unit test that runs everywhere.

## 6. Design alternatives (selected vs rejected)

- **Chosen: cover through existing seams, no production-logic edits** (above).
- **Rejected: extract `streamInsert`+mapRows into `stream-insert.ts` + exported
  functions, and `nameIds` into a `skill-name-ids.ts` leaf.** It manufactures the
  CI-attribution problem, forces extraction-before-characterization (P9), and adds
  metric-driven fragmentation (P1/P8/P10) ŌĆö all avoidable since honest seams exist.
- **Rejected for `nameIds`: mock `getLiveDatasetOnView` and capture/invoke the
  `nameIds` callback directly.** That proves an internal callback contains a loop,
  not the public `getSkillsForUserOnView` behavior ŌĆö it tests implementation
  structure (AGENTS.md testing policy). Driving the real seam and asserting the
  public result is the honest behavioral lock.
- **Rejected for ingest: a unit test with a stubbed `flush`.** The `runIngest` seam
  tests the real inserts and all mappers with the same effort and no export.
