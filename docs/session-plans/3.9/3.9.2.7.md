## 5. Interface designs

The ledger is a document, not code — no new exports, so no TSDoc drafts. Its
"interface" is the schema future consumers parse, and Step 3's questions are
answered for it as an artifact:

- **Decision it owns (P2):** what counts as one primitive, which rail
  enforces it, and its lifecycle verdict — the single home of primitive↔rail
  coverage. Future audits diff it; 3.9.3.2 and 3.9.3.5 append their rows to
  it; the version-close checklist reads its dispositions.
- **Public surface (P1):** exactly two parts with fixed schemas, stated in
  the document preamble so future overwrites preserve them:
  - **Part 1 — primitive map:** one table per area (UI/design, API/backend,
    data, infra/platform, agent tools/workflow, auth/trust) with columns
    `| Primitive | Owning module | Decision it hides | Rail |`. `Rail` names
    the concrete mechanism (lint selector family / restricted-import rule,
    Fallow zone/boundary, or registry-gate test path) or the explicit value
    `none` — an unrailed primitive is a visible finding class, not a blank.
  - **Part 2 — verdict table:** stable `PL-NNN` rows,
    `| ID | Area | Class | Evidence | Proposed end-state | Est. size | Status |`.
    `Class` ∈ create/combine/delete/expand/keep (the arc's lifecycle verbs).
    `Status` ∈ `Delivered` | `Proposed` | `Approved (3.9.2.N)` |
    `Backlog` | `Rejected (reason)`. IDs are permanent; rows are never
    renumbered.
- **Granularity rule (contract decision 7, resolved):** **one row per
  decision an agent must get right.** A family collapses to one row when its
  members share one decision and one rail (the 57 `src/components/ui/*`
  components = one "UI component library" row: the decision is "consume
  `@/components/ui`, never raw Base UI/sonner/`next/image`"; picking Button
  vs Input is need-driven, not policy). A member gets its own row when it
  hides a distinct decision or carries a distinct rail (e.g. `EveImage`,
  `tones.ts`, `cn.ts` merge-config). Non-UI areas mostly stay per-surface
  because each surface hides a distinct decision. Expected result: **~40–50
  map rows** instead of ~115.
- **Failure modes defined away (P5):** `none` as an explicit rail value;
  fixed Status vocabulary (no free-text states); one file (no index to
  drift); overwrite-in-place semantics stated in the preamble (the
  baseline's pattern — never append history).
- **P7:** each verdict row's Evidence cell carries the concrete consumer or
  observed repetition (the arc's standing evidence rule); judged Keeps carry
  the written distinction.

## 6. Design alternatives

- **Granularity — selected:** decision-level rows (above). **Rejected:**
  literal one-row-per-exported-surface — ~115 rows, 57 of them repeating the
  same decision/rail text; the document's interface would be larger than the
  knowledge it hides (P1 applied to the artifact), and every future UI
  primitive addition would demand a content-free ledger diff.
- **Delivered-verdict representation — selected:** one verdict row per arc
  slice (PL-001…PL-006), each naming the primitives it delivered/expanded
  and its rail story; AF-009's closure framing lives in the 3.9.2.2 row;
  per-primitive detail stays in the map. **Rejected:** one verdict row per
  touched primitive — duplicates the map, and delivery evidence (PR, rail
  demonstration) is naturally per-slice, not per-primitive.
