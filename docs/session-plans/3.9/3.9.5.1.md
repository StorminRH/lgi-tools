## 5. Interface designs (Step 3 answers + draft interface comments)

AF-014 relocates existing symbols; the design question is **ownership**, not new
surface. After the move:

**`src/lib/eve-image.ts` ‚Äî the single owner of "which server-backed sizes exist"
(P2). Draft/relocated interface comments:**

```ts
/** Canonical ordered ladder of server-backed EVE image renditions, in pixels.
 *  The single owner of which square sizes the EVE image server supports; every
 *  size type, cap, and snap decision derives from this array. */
export const EVE_IMAGE_SIZES = [32, 64, 128, 256, 512, 1024] as const;

/** A supported square EVE image rendition in pixels, drawn from EVE_IMAGE_SIZES.
 *  Requests are snapped upward to one of these. */
export type EveImageSize = (typeof EVE_IMAGE_SIZES)[number];

/** Closed set of EVE image-server endpoint families (portraits, logos, and the
 *  type icon/render/blueprint renditions) that share the size ladder. */
export type EveImageFamily = /* 7 families, moved verbatim */;

/** Rounds a requested pixel width up to the nearest rendition the family
 *  supports, capped at the ladder's maximum (derived from EVE_IMAGE_SIZES; no
 *  separate MAX constant). */
export function snapEveImageSize(family: EveImageFamily, requestedWidth: number): EveImageSize;

/** Builds the canonical EVE character-portrait URL for a supported square pixel
 *  size. (Size type now spans the full server ladder, 32‚Äì1024.) */
export function characterPortraitUrl(characterId: number, size: EveImageSize = 64): string;
/** ‚Ä¶corporationLogoUrl mirrors this. */
```

- **What decision does lib own and hide? (P2)** The size vocabulary ‚Äî which
  renditions exist, which families support them, and how a request snaps. This
  is lower-layer server-capability knowledge, independent of `next/image`.
- **Public surface much simpler than implementation? (P1)** Yes ‚Äî a ladder, a
  size type, a family type, one pure snap function, two URL builders.
- **Slice/import rules? (P3/boundaries)** Lib imports only lib. Consumers
  (`src/components/eve-image.tsx`, `type-icon.tsx`) import downward ‚Äî allowed.
- **Real callers now? (P4)** Every symbol has a live caller today (URL builders,
  `EveImage`/`eveImageUrl`, `type-icon`). The **1024 widening** of the lib
  builders' size type is **not speculative**: portraits/logos already resolve
  1024 via `FAMILY_SIZES`, and unifying to one ladder-derived type *requires* the
  lib type to include 1024 or the component path would regress narrower.
- **Failures defined away? (P5)** `snapEveImageSize` returns a valid rendition
  for any width (caps at the ladder max) ‚Äî no throw, no caller handling.
- **Rationale the signature can't carry? (P7)** The "one owner of server
  capability; component consumes" note lands as the lib module prologue.

**`src/components/eve-image.tsx` ‚Äî keeps only the `next/image` adapter/rendering.**
`eveImageUrl` (the loader that consumes `ImageLoaderProps`) and `EveImage` stay;
they now import `snapEveImageSize`/`EveImageFamily` from lib. Signatures and
their existing interface comments are unchanged.

**`saved-plans-view.ts` (AF-015):** the render verdict becomes file-local **and
renamed** for full name-distinctness (Ryan's decision):

/** View-render verdict for the saved-plans surface: still loading (blank), an

The controller `interface SavedPlansState` in `use-saved-plans.ts` is

## 6. Design alternatives (selected + rejected, one sentence each)

- **AF-014 ownership.** *Selected:* lib owns the ladder/family/snap facts; the
  component consumes them. *Rejected:* keep the ladder in the component and let
  lib import it ‚Äî forbidden (lib‚Üícomponent) and it ties server-capability facts
  to `next/image`.

  the `next/image` loader adapter, consuming `ImageLoaderProps`). *Rejected:*
  move it to lib too ‚Äî would couple lib to `next/image` and contradicts the
  "component keeps the adapter" done-condition.

  existing EVE image *variant*-literal rail already guards rendition strings; a
  new size rail is speculative (P4).

  `SavedPlansViewState`. *Rejected:* unexport only ‚Äî leaves the same name in two
  files, a residual P7 smell.
- **AF-011 README.** *Selected (contract rec):* reframe the row as CI/Vercel-only
  rather than delete it ‚Äî the script exists and forkers may build; `pnpm verify`
  stays the local DoD. *Rejected:* remove the row ‚Äî loses honest information.
- **AF-012/P2 exemplars.** *Selected:* live, stable, constitution-endorsed files

  would shift under the same change.
