## 5. Interface designs

**New module: `src/lib/esi-datasets/freshness.ts`** (the leaf's runtime face; imports only `./entries` and `./types`).

Decision it owns and hides (P2): *how a dataset's declared freshness model + effective TTL become a runtime verdict*, including the shared null-handling rule ("absence of a stamp/boundary always reads stale"). Callers know their dataset name and their stamp; they never see a TTL constant, a model discriminant, or window arithmetic.

Draft interface comments (P7 — written before code):

```ts
/**
 * The names of registry datasets with a static staleness window — the
 * caller-ttl and row-stale-after models, whose effective TTL is the
 * entry's verified upstream cache time or its recorded override.
 */
export type StaticWindowDatasetName = /* derived from typeof ESI_DATASET_ENTRIES */;

/**
 * A dataset's bound runtime staleness gate. `isStale` is pure and
 * clock-injected: true when the owner was never refreshed (null) or the
 * refresh is older than the entry-derived window. `ttlMs` exposes the
 * same window as a number for write-time `stale_after` stamping and SQL
 * cutoff arithmetic — both faces read one registry source.
 */
export interface FreshnessGate {
  readonly ttlMs: number;
  isStale(refreshedAt: Date | null, now: Date): boolean;
}

/**
 * Returns the staleness gate bound to a statically windowed dataset's
 * registry entry. The registry is the only TTL source: deleting a
 * per-feature TTL constant and calling this instead cannot change the
 * window, because the entries were characterized against those constants
 * before the collapse.
 */
export function freshnessGate(name: StaticWindowDatasetName): FreshnessGate;

/**
 * Row-boundary staleness verdict for expires-boundary datasets: stale
 * when the persisted boundary has passed or no boundary row exists. The
 * boundary is row-carried (derived from the upstream response), so this
 * verdict takes no TTL; pure and clock-injected.
 */
export function isBoundaryStale(staleAfter: Date | undefined, now: Date): boolean;
```

Interface answers (Step 3):
- **P1:** callers use one factory call + one method; the implementation hides entry lookup, model legality (type-level), override-vs-verified resolution, and null-handling. Surface far simpler than implementation.
- **Slice/zone:** `lib`; every slice may import it; rail 3 makes it the *only* legal verdict source.
- **P4 real callers now:** six descriptor bindings, four membership predicates, the eve-characters route filter, `refreshStaleAffiliationsForUser`, `affiliation-store` SQL cutoff, prices `ingest.ts`/`refresh-on-view.ts` write stamps, history `staleIds` filter. No speculative exports — `freshnessGate` is type-restricted to static-window datasets instead of returning `ttlMs: number | null` for all (P5: the "no window" case is defined out of existence at the type level).
- **P5:** `isStale` absorbs the null/never-synced case; `isBoundaryStale` absorbs the missing-meta-row case; no caller handles a TTL-missing error.
- **P7 non-obvious knowledge:** the module prologue records the trigger-layer verdict (§6b) — why the three on-view *trigger* implementations above this leaf deliberately stay separate.

**Changed surfaces (all narrowing):**

## 6. Design alternatives

**a) Verdict signature (contract decision 1).**
- **Selected: per-dataset bound gates** — `freshnessGate(name)` returning `{ ttlMs, isStale }`. Caller-knowledge count: one concept (the bound gate), name passed once at module scope, call sites unchanged in shape; `affiliation-store`'s SQL cutoff and the prices write stamp read `ttlMs` from the same binding that serves the verdict — one source per dataset.
- **Rejected: one generic `isStale(entry, refreshedAt, now)`** — every call site must also learn entry lookup and thread the entry through descriptor closures and predicates (more concepts per caller, P1), and the TTL-as-number callers would still need a second accessor. One sentence for the owning comment: *bound gates were chosen because every existing call site already speaks `(refreshedAt, now)` and P1 counts entry-threading as caller knowledge.*

**b) Trigger layer (contract decision 2 — the judged tier).**
Two decompositions compared for the on-view trigger family:

| Dimension | market-prices | market-history | owner-sync engine |
| --- | --- | --- | --- |
| Freshness gate | none — always live; 30s `'use cache: remote'` coalescing | row `stale_after` = response `Expires` boundary | per-slice `isStale` before token vend |
| Budget policy | degrade to DB seed, never retry | short-circuit, back-fill stored | durable deferred queue + worker + dead-letter |
| Retry | nightly cron sweep | next stale view | `esi_refresh_jobs` queue |
| Write path | `after()` write-behind, binary outcome | `after()` write-behind, tri-state | awaited descriptor `save`/`stampFresh` |
| Concurrency | bounded fan-out + cross-instance cache | bounded fan-out, gate dedupes | serial passes, parallel owners, token-serialization |
| Fallback source | ESI→Fuzzwork | ESI-only | ESI-only, per-owner token |

- **Selected: Keep-with-written-distinction.** The three implementations differ on every decision-owning dimension; the only shared shape is the temporal skeleton "check → fetch → persist", and structuring a module around execution order is the P2 anti-pattern. A unified primitive would be a shallow parameterized shell (flag per row of the table — special-case creep, P4/P8). The distinction is recorded in the `freshness.ts` prologue (condensed) and this plan (full).
- **Rejected: shared trigger primitive.** Scope fence — unification is campaign-shaped if it (i) touches more than two slices' trigger paths or (ii) changes budget/retry semantics or Next cache primitives. Both trip: it spans prices + history + engine + deferred queue (~1,000 LOC) and would have to reconcile three budget policies. Per the contract, no absorption; and because the verdict is "Keep is right," not "unify later," **no Proposed row for the trigger layer** goes to the 3.9.2.7 audit. (The pricing feature's staleness-*consumption* migration remains the contract's named Proposed-row candidate for 3.9.2.7 and is reported at close-out.)
