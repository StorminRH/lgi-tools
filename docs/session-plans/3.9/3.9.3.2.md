## Context

Every surface that shows an EVE item image currently decides *which rendition*
to request (item `icon`, 3D `render`, blueprint `bp`) at its own call site. The
render door is already centralized (`TypeIcon`/`EveImage`, 3.8.2.6), but the
*choice* layer is duplicated and drifted: `HeroCard` inlines a renderable
ternary, `node-card-view` inlines an `'icon'` default, `nodeIcon`/
`isRenderableCategory` sit in planner-private `industry-styles.ts`, and the
`/industry` landing rows show typed monogram initials instead of images at all.
This session creates the one decision owner (contract
`docs/session-contracts/3.9/3.9.3.2.md`, backlog "EVE images", Ryan-directed
2026-07-10), migrates every call site to it, upgrades the landing rows to
product icons, and adds a lint rail so the drift cannot recur.

## 1. Goal

Every EVE type-image call site states *intent* and receives a resolved
`{typeId, variant}` from one shared resolver in `src/data/eve-data/`; the
`/industry` landing rows upgrade from monogram initials to product icons with
monogram fallback.

## 5. Interface designs

New module **`type-images.ts`** (new, in `src/data/eve-data/`; + co-located
`type-images.test.ts`), imported directly (eve-data has no barrel). Draft
interface comments â€” the complete public surface:

```ts
/**
 * Closed rendition vocabulary served by CCP's type-image endpoint. Owned here so every
 * rendition decision has one home; render plumbing (TypeIcon) consumes it, and call
 * sites only ever receive it inside a resolved EveImageDescriptor.
 */
export type TypeIconVariant = 'icon' | 'render' | 'bp' | 'bpc';

/**
 * A fully resolved EVE type-image choice â€” which id to show and which rendition to
 * request. Produced only by this module's intent resolvers; hand it to TypeIcon as-is.
 */
export type EveImageDescriptor = { typeId: number; variant: TypeIconVariant };

/**
 * Show-the-item intent: search rows, recents/template/saved rows, raw ledgers, and
 * node defaults show the item's own inventory `icon`. 404s stay TypeIcon's monogram
 * fallback, so callers never pre-check existence.
 */
export function itemImage(typeId: number): EveImageDescriptor;

/**
 * Hero intent: the large product hero upgrades to the 3D `render` only when the
 * product's SDE category is renderable (isRenderableCategory, computed where the
 * category name lives â€” at query time â€” and threaded as this flag). Anything else
 * keeps `icon`, never issuing a /render that would 400.
 */
export function heroImage(typeId: number, renderable: boolean): EveImageDescriptor;

/**
 * Show-what-you-run intent (the promoted nodeIcon): a buildable/reaction node shows
 * the producing blueprint/formula `bp`; a raw leaf with no producing type keeps its
 * own `icon`. `bpc` stays deliberately unused: ownership is conveyed by frame tone.
 */
export function nodeImage(
  producingBlueprintTypeId: number | undefined,
  typeId: number,
): EveImageDescriptor;

/**
 * Industry-job intent: a job row shows the product's `icon` when ESI reported a
 * product, else the blueprint being run as `bp` â€” never a bare blueprint `icon`
 * request (blueprints serve no icon rendition).
 */
export function jobImage(
  productTypeId: number | undefined,
  blueprintTypeId: number,
): EveImageDescriptor;

/**
 * Returns whether an EVE category's products serve the large `render` rendition
 * (verified: Ship, Drone, Structure). Every other category degrades to `icon`.
 */
export function isRenderableCategory(categoryName: string): boolean;
```

Step-3 answers: the module owns and hides *the rendition rule per intent* plus
the variant vocabulary (P2, one home). Surface = 2 types + 5 small functions;
implementation absorbs the CCP-endpoint quirks (blueprints serve no icon,
/render 400s) so callers can't hit them (P5). Every export has a real caller in
this PR (P4): `itemImage` Ã— 6 surfaces, `heroImage` Ã— HeroCard, `nodeImage` Ã—
CockpitBuildPlan, `jobImage` Ã— personal + corp job rows,
`isRenderableCategory` Ã— planner `queries.ts:205`. Renderable categories stay a
module-private `RENDERABLE_CATEGORIES` set (P2); the hero declares
renderability via the query-time-computed boolean because the category name
only exists at data time. Rationale the signatures can't express (why `bp` for
both blueprints and formulas, why `bpc` unused, the 400/404 verification) lives
in the interface comments above (P7).

## 6. Design alternatives

- **Chosen:** small named per-intent functions sharing one descriptor type.
  **Rejected:** a single `resolveEveImage(intent: DiscriminatedUnion)` â€” it
  makes every caller learn the full union and its tag strings while hiding
  nothing more (ceremony without added depth).
- **Rejected:** an `intent` prop on `TypeIcon` itself â€” it would widen the
  render door with domain knowledge the contract explicitly keeps out (render
  door consumed as-is), and put a data-zone decision in a shared component.
- **Rejected:** vocabulary home in `src/lib/` â€” eve-data already owns SDE
  category/blueprint semantics; lib is domain-neutral.
