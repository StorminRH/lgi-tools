**(b) `scripts/profile-sites-dev.mjs`** — bounded profiling orchestrator, reused unchanged for `.2`'s "after" readout. Command contract (pinned; the module prologue restates it):

- **CLI:** `pnpm --silent profile:sites-dev --label <label>` (`--silent` because pnpm banners would corrupt the stdout JSON). Label grammar `^[a-z0-9-]{1,32}$` (validated; it becomes a path component). Documented labels: `as-found`, `clean` (this session); `after`, `sample` (session `.2`).
- **Output contract:** **exactly one JSON document on stdout on every exit path** — success, environment refusal, and threshold abort all emit the document with a `status` field (`ok | refused | aborted`) and whatever was measured; diagnostics and child dev-server output go to stderr only. Exit codes: 0 ok, 1 aborted, 2 refused. Also written to `docs/ux-check/profiles/<label>-<timestamp>.json` (gitignored).
- **Preflight refusal (exit 2, before spawning):** port 3000 occupied (never kills a foreign server); local Postgres unreachable; **insufficient memory headroom** — macOS memory pressure not normal or free+inactive memory below 6 GB (the archived melt began with <1.2 GB free against a ~3.4 GB dev baseline; refusing beats aborting mid-melt).
- **Lifecycle:** spawns `next dev -H 127.0.0.1 -p 3000` with `detached: true` (own process group; PGID recorded); waits for Ready (bounded 180 s). **Teardown is guaranteed on every path:** the same idempotent cleanup (SIGTERM to the PGID → ≤10 s poll → SIGKILL → verify port 3000 closed) runs from `finally`, from idempotent `SIGINT`/`SIGTERM`/`SIGHUP` handlers, and from uncaught-exception/unhandled-rejection handlers (SIGKILL of the profiler itself is the one unhandleable case, stated in the prologue).
- **Sampling:** every 500 ms, PGID-wide `ps -g <pgid> -o pid=,rss=,pcpu=,comm=` aggregated (total RSS/%CPU, per-`next-server` child breakdown); `sysctl vm.swapusage` every 5 s.
- **Measurement order:** Ready → 30 s idle window (idle RSS/CPU baseline, zero requests) → prewarm `GET /` (bounded 60 s; removes the control route's own compile from the later probe) → cold `GET /sites` (AbortSignal 120 s) with the responsiveness probe `GET /` (AbortSignal 15 s) launched at **t+750 ms while `/sites` is still in flight**; if `/sites` completes first the result records `not overlapped` rather than claiming a responsiveness verdict → two warm `GET /sites` (AbortSignal 30 s each).
- **Request completion definition:** HTTP status validated **and the body fully drained**, then a bounded wait (≤10 s) for the matching Next dev-log line, correlated by route + close in time; the raw matched line is retained in the JSON. (`fetch` resolves at headers; `/sites` streams a Suspense shell — headers alone would fake a fast cold time.)
- **Abort thresholds (exit 1, teardown still runs):** swap growth > 1 GB from run start; aggregate PGID RSS > 4.5 GB (the archived melt level; healthy June peak was 2.9 GB); ≥ 800% total CPU sustained 20 s during the idle window. Threshold logic (equality boundaries, sustained-window reset, missing-sample handling, abort priority) lives in `profile-parse.mjs` and is unit-tested.
- **Attribution rule:** the dev-log `application-code` bucket attributes cost to the app bucket only; internal causes (query vs overlay vs render) are reported as *candidates* unless separately instrumented.

**(c) `scripts/profile-parse.mjs`** — import-safe pure helpers with co-located behavioral tests: dev-log request-line parsing (Next 16's `next.js` / `proxy.ts` / `application-code` labels, verified against the installed `next/dist/server/dev/log-requests.js`), duration parsing across `µs|ms|s|min`, `ps`/`vm.swapusage` parsing, label validation, threshold/abort logic, sentinel counting, result shaping.

**(d) `docs/ux-check/probes/sites-lazy-detail.mjs`** — durable probe, run by name via `node docs/ux-check/run-probes.mjs sites-lazy-detail`, registered in the README index. Seeds `expand` mode via localStorage (the anonymous-preference mechanism, per the `dialog-open` precedent) to pin the tested mode. Structural gates (desktop + mobile):

1. literal equality: outer `summary` count === card count;
2. **all** `[data-lazy-details]` wrappers have zero element children pre-open;
3. opening the first card mounts content in exactly that wrapper (all others still empty);
4–6. switch to Table via the `Sites view` control and repeat 1–3 against `details.sites-table-row`.

Sentinel strings ("Wave Spawns" / "No Sleeper presence" — detail-body-only today, but copy-dependent) stay secondary evidence inside the profiler JSON, not probe gates.

## 6. Design alternatives (design-it-twice records)

1. **Profiling vehicle — chosen: a self-contained bounded script** (§5b–c) per the standing script rule (looping, multi-call, file-producing) and `.2`'s reuse of the identical protocol. *Rejected:* ad-hoc shell/curl timing — not reproducible (no pinned cadence/timeouts/PID selection/completion definition), no machine-readable output, no guaranteed teardown against the documented melt hazard. Also rejected: in-source `console.time` instrumentation — dirties the tree; the dev-log split already buckets the cost.
2. **Lazy-render lock — chosen: one committed `data-lazy-details` hook + a both-surfaces structural probe + sentinels as secondary evidence.** *Rejected:* cards-only probe on a structural CSS selector — leaves the table caller unlocked and couples to sibling order; jsdom component test — repo has no `.test.tsx` by policy, and the native-`toggle` seam is only faithful in a real browser.
3. **Reconciliation shape — chosen: plan-only** (contract unamended; this plan records the interpretation). *Rejected:* in-session contract/master amendment — a `plan-version` ownership violation (flagged by both first review rounds); formal `plan-version` pass — two extra session-terminal outcomes for a truth reconciliation `.2` planning and a later plan-version pass already own.
