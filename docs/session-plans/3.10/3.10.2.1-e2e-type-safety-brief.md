# End-to-End Type-Safety Migration — Planning and Execution Brief

Use this as the task brief for LGI.tools. Begin through the repository's normal `start-session` workflow. Resolve the current lifecycle state and read every artifact required by `AGENTS.md` before planning or changing code. Work from the current branch and current installed dependency versions; do not rely on remembered Next.js, Zod, Better Auth, Convex, or TypeScript APIs.

This is a cross-cutting architecture migration. Enter Plan mode first, inventory the complete surface, produce the canonical session plan required by the repo, and pause for approval before implementation. The approved plan becomes the execution authority; do not maintain a second implementation plan beside it.

## Bottom line

GOAL: Complete LGI.tools' first-party end-to-end type-safety chain. Every programmatically consumed internal endpoint must have one authoritative contract that owns its method, path or typed path builder, request encoding and schema, every declared HTTP status, and every response body schema. Clients must receive a closed result union in which `status` discriminates the exact success or error body. Routes must build responses through a typed response helper tied to that same endpoint contract. Remove duplicated URL strings, untyped `Response` pass-throughs, `res.json() as Type` casts, hand-copied wire interfaces, and dependency-owned REST shapes where an official typed API exists.

DONE means a request or response change at any first-party boundary either propagates automatically or fails typecheck/tests at every incompatible producer and consumer. No old and new contract paths remain in parallel. `pnpm verify` and the repo's full close-out verification pass with zero new waivers, exceptions, Fallow baselines, or architecture-boundary violations.

This is not a Prisma or tRPC migration. Keep Drizzle, Postgres, Next.js route handlers, Convex, and the existing feature/data ownership model. Strengthen the contract system already centered on `src/lib/api-client.ts` and the owning slices' `api-contract.ts` modules.

## Verify live before designing

Before relying on an API, verify it against the installed version and current official documentation:

- TypeScript's literal-key, conditional-type, and discriminated-union behavior.
- Zod 4.4.x input/output typing, object behavior, safe parsing, and error handling.
- Next.js 16.2.x route-handler `Response` behavior using the installed Next docs under `node_modules/next/dist/docs/`.
- Better Auth 1.6.x typed client APIs for OAuth sign-in, sign-out, token/JWT access, and any other currently hand-pinned REST call.
- Convex 1.42.x generated function references and the supported way to share typed contracts with Convex actions.

Record the relevant conclusions in the plan. If an official typed Better Auth API cannot replace a hand-pinned route, prove why from the installed package/docs and design a mechanically verified adapter rather than silently retaining an asserted interface.

## Current seams that must be addressed

Treat this list as a starting point, not a complete inventory. Search the repository mechanically before finalizing the plan.

### Browser-to-Next API seam

`src/lib/api-client.ts` currently couples method, path, request schema, and one success schema, but its failure branch returns `{ ok: false; status: number; response: Response }`. That leaves error bodies opaque and lets each caller parse them independently.

Replace this with a contract model that declares every supported response status and its wire codec/schema. `apiFetch` must infer a closed union from the endpoint itself. A representative result should narrow like this in ordinary control flow:

```ts
const result = await apiFetch(exampleEndpoint, { body });

if (result.ok) {
  // result.status is the endpoint's declared success literal.
  // result.data is exactly the schema for that status.
} else {
  switch (result.status) {
    case 400:
      // result.error is exactly the declared 400 body.
      break;
    case 409:
      // result.error is exactly the declared 409 body.
      break;
  }
}
```

The final naming may differ, but the following properties are required:

- `status` is a literal discriminant, not `number`, for every declared outcome.
- Success and error bodies cannot be read through the wrong status arm.
- JSON, text, and empty bodies are represented deliberately; do not force plain-text or 204 routes through fake JSON.
- Requestless endpoints cannot accept a body; body endpoints require the schema-derived input.
- Unexpected statuses, invalid JSON, and response-schema drift can never be returned as trusted typed application data. Give transport failures and contract/protocol failures an explicit, tested policy. Do not collapse them into a declared domain error.
- Abort/network behavior remains deliberate and documented.
- There is no development/production semantic split where invalid production data is trusted while development merely logs it. Choose and document one safe policy. Preserve wire bytes and avoid accidental Zod normalization unless the contract intentionally calls for it.

### Route-to-contract response seam

Add a typed route-response helper in the lowest legal shared layer. It must accept an endpoint contract, one of that endpoint's declared status literals, and only the body valid for that status. It must serialize using the declared JSON/text/empty codec.

Representative intent:

```ts
return apiResponse(exampleEndpoint, 409, {
  error: 'reauth_required',
});
```

The helper must make these fail at typecheck:

- A status not declared by the endpoint.
- A success body returned with an error status.
- An error body returned with the wrong error status.
- A body returned for an empty response.
- An endpoint response created from an unrelated contract.

Update route handlers to use the helper for every declared response they construct. Remove route-local `satisfies` response pins once the helper becomes the stronger owner; do not retain ceremonial duplication.

Existing helpers such as `runMutationRoute`, `parseJsonBody`, authorization guards, same-origin handling, rate-limit guards, cron shells, and service-auth guards currently pass opaque `Response` objects through some paths. Redesign their boundaries as needed so a route cannot escape its endpoint's declared response union. Prefer typed failure values that the owning route maps through `apiResponse`, or a generic helper parameterized by the endpoint. Do not weaken auth ordering, rate-limit ordering, same-origin behavior, or existing response semantics while doing this.

### String-based route seams

The actual URL must remain a string on the wire, but it must have one authoritative owner.

- Inventory every first-party `/api/...` literal, absolute internal URL concatenation, query-string construction, dynamic path construction, method declaration, and endpoint descriptor.
- Browser callers must consume endpoint contracts, never reproduce an internal URL or method.
- Convex/server-to-server callers must consume the same endpoint contracts through an appropriate typed transport that can apply a trusted base URL, timeout, and service-auth headers. Replace `${siteUrl}/api/internal/...` duplication and every `await res.json() as SomeResponse` cast.
- Query parameters and dynamic path parameters must be schema-derived and serialized through a typed contract path/query builder. Callers must not hand-maintain parameter names or encoding.
- Add or extend a mechanical route-presence gate so each declared first-party endpoint's method and concrete/static route shape match an actual Next route export. Dynamic route templates must be checked deliberately.
- Preserve one source of truth: do not introduce a separate central route registry that duplicates every slice-owned endpoint. If a generated manifest is the cleanest solution, generation and drift checking must be deterministic, committed only if repo conventions require it, and enforced in verification.

### Hand-maintained dependency and cross-runtime seams

- Remove Better Auth request/response schemas that merely copy a library-owned REST shape when an official typed Better Auth client API can own the operation. Restructure command/action composition if necessary to respect the existing data/feature/app import graph; do not add a Fallow exception just to reach the auth client.
- If any Better Auth HTTP seam truly must remain, bind it to exported library types or an executable contract test against the installed version. No unverified `as`, copied interface, or comment-only "check this on upgrade" is acceptable.
- Replace Convex-to-Next response assertions in `convex/lib/characterSync.ts` and any equivalent files with the shared typed transport plus runtime contract validation.
- Keep Convex's generated `api` references for client-to-Convex calls. Do not wrap them in a second home-grown RPC layer.
- External ESI and third-party HTTP responses are not end-to-end controllable. They must remain `unknown` until runtime validation/projection succeeds; never make them appear safe through a type assertion.

### Completeness and enforcement seams

Upgrade `src/app/api/api-contracts.test.ts` and the ESLint rails so the migration cannot regress. At minimum, mechanically enforce:

- Every non-library-owned Next API route is associated with its owning endpoint contract.
- Every programmatically consumed endpoint declares request encoding and all response statuses.
- Every route-built response uses the typed response helper or a narrowly documented framework/library-owned delegation.
- Raw same-origin API `fetch` calls, raw internal absolute-URL construction, `res.json() as ...`, and route-local response casts are rejected.
- No endpoint contract has an orphaned path/method, and no programmatically consumed route is missing an endpoint contract.
- Endpoint errors use closed literal codes where the vocabulary is known; avoid generic `{ error: string }` for a closed domain taxonomy.
- All remaining allowlists are small, named, justified, and tested against accidental growth. The Better Auth catch-all may remain library-owned; ordinary first-party routes may not use that label as an escape hatch.

Do not rely only on regex tests if a TypeScript/AST-based check is needed for correctness. Keep any new enforcement understandable and deterministic.

## Implementation constraints

- Preserve current HTTP methods, paths, status codes, cache behavior, auth/authz, rate limits, same-origin behavior, cron ordering, service-secret handling, and user-visible behavior unless the approved plan explicitly identifies and tests a necessary correction.
- Keep contracts in the owning feature/data slice and keep contract modules safe for both client and server imports.
- Preserve the repository import-direction rules. Composition belongs in `src/app`, `src/search`, or another established composition layer—not through a new cross-slice dependency.
- Do not add Prisma, tRPC, another router, another validation library, or another HTTP client.
- Do not use `any`, `as unknown as`, broad type assertions, non-null assertions, or duplicated interfaces to force the migration through.
- Do not retain deprecated compatibility overloads after every caller has migrated.
- Do not create a second API client beside `apiFetch`; evolve the shared transport coherently. A server/service transport may share a lower-level executor when browser-only concerns differ.
- Keep public exported surfaces documented to the repo's TSDoc standard.
- Respect Fallow complexity limits; use small type utilities and testable pure helpers instead of a single type-level/runtime monolith.
- Do not run a production build locally. Follow the verification commands in `AGENTS.md`.

## Tests required

Add compile-time and runtime tests that prove the architecture rather than merely covering lines.

### Type-level tests

Use the repo's established Vitest/TypeScript conventions to prove:

- Request input is inferred from the request schema.
- Each success status narrows to its exact data body.
- Each error status narrows to its exact error body.
- Bodies cannot cross status arms.
- Undeclared statuses fail compilation.
- Empty-body endpoints reject bodies.
- Query/path builders require exactly their declared parameters.
- The route-response helper is tied to the endpoint passed to it.

Use explicit negative type tests only where the repo permits them, with a concise explanation of the invariant.

### Runtime tests

Prove:

- JSON, text, and empty success/error bodies serialize and parse correctly.
- Declared statuses return the expected discriminated result.
- Undeclared statuses and schema-drifted bodies take the chosen protocol-failure path in every environment.
- Network rejection and abort behavior remain correct.
- The base-URL/service transport preserves timeout and auth headers without duplicating endpoint paths.
- Guard, parser, rate-limit, auth, and handler failures still preserve existing ordering and status/body behavior.
- Better Auth calls use the verified official typed path or the fallback adapter fails loudly when its installed contract changes.
- Route/contract completeness tests contain seeded red fixtures proving each new gate can fail.

Add representative direct route tests for complex multi-status endpoints, especially `/api/internal/eve-token`, JSON-body validation failures, one auth-gated account mutation, one rate-limited mutation, one cron route, one query/path route, and one 204 route. Do not duplicate identical tests across every route when the shared helper plus mechanical completeness gate proves the invariant.

## Migration order

Plan the work in dependency order and keep the repository compiling at deliberate checkpoints:

1. Inventory and settle the contract/result/codec design with type tests.
2. Implement the shared contract definition, typed executor, protocol-failure policy, and typed route-response helper.
3. Adapt parser/guard/mutation/cron/service helpers so failures cannot bypass endpoint contracts.
4. Migrate a small representative vertical slice and prove the ergonomics.
5. Migrate every browser-consumed first-party endpoint and caller.
6. Migrate Convex/server-to-server first-party calls.
7. Replace Better Auth hand-pinned calls with verified official typed APIs or the approved mechanically bound adapter.
8. Migrate query/dynamic-path builders and all remaining route responses.
9. Remove old overloads, opaque response handling, string duplication, casts, obsolete types, comments, and allowlists.
10. Strengthen the completeness/lint gates, run the full inventory again, and prove zero undisposed seams.

If the inventory makes this too large for one safe session, divide it into ordered sessions on one independently shippable branch, with the shared primitive first and the no-bypass enforcement/final cleanup last. Do not open a PR while both contract systems remain valid long-term. The branch is complete only when the old path is gone.

## Definition of done

The final report must provide evidence for all of the following:

- One endpoint contract is the authoritative source for method, path/query/path parameters, request encoding, declared statuses, and response bodies.
- `apiFetch` returns closed status-discriminated success and error unions.
- Route responses are constructed through the endpoint-bound typed helper.
- No programmatic first-party caller duplicates an internal route string or asserts a decoded response type.
- Convex-to-Next internal calls use the same contracts with runtime validation, timeout, and service auth preserved.
- Better Auth's former hand-maintained shapes are removed or mechanically tied to the installed library with documented evidence.
- Every route and caller from the planning inventory has a final disposition; repeat searches show no stale seam.
- Type-level tests demonstrate that incorrect request, status, and body combinations do not compile.
- Runtime tests demonstrate declared responses, contract failures, network failures, and empty/text/JSON codecs.
- The route completeness and lint gates include seeded failures and prevent a new bypass.
- `pnpm typecheck` passes using a fresh non-incremental run where appropriate.
- `pnpm lint`, `pnpm test`, `pnpm fallow`, `pnpm assert:routes-present`, and `pnpm verify` pass.
- Close-out coverage-backed Fallow passes against `origin/main` with dead code 0, architecture issues 0, introduced duplication 0, and no new waiver/baseline entry.
- No production build was run locally.

End with a concise before/after contract example, the final inventory counts, verification output, and any truly external or library-owned boundary that necessarily remains outside first-party end-to-end inference. Do not call the work complete while an untyped first-party escape hatch remains.
