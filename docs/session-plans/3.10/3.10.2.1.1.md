## Why now

Phase 2 of v3.10 begins with the production-flow error contract, and the
operator has folded the full end-to-end type-safety migration into this
sub-version. Everything downstream — the .2 route/caller migration, the .3
seam cleanup, 3.10.2.2's rejecting 403, and 3.10.3.1's telemetry codes —
consumes the primitives this session creates. Building them first, additively,
with one real slice proving the ergonomics, means the bulk migrations become
mechanical rather than exploratory.

## Scope (the destination)

After this session the repository contains one authoritative typed chain for
first-party endpoints, used by exactly one production endpoint (feedback) and
by nothing else yet:

- **Typed failures (`src/lib/failure.ts`).** `AppFailure` is a closed
  eight-category discriminated value carrying a stable application `code`
  (defaulting to the category), an optional request-safe `detail`, optional
  `retryAfterSeconds`, an optional `status` override only where a category
  legitimately spans statuses (`dependency_unavailable`: 502 upstream-failed
  vs 503 unavailable/unconfigured), and an optional non-serialized `cause`
  for logs. Deep code constructs failures through per-category constructors.
- **One mapper (`src/lib/problem.ts`).** `problemResponse(failure)` mints a
  correlation id (`crypto.randomUUID()`), derives the HTTP status from the
  category map (400/401/403/404/409/429/503-or-502/500), builds the RFC 9457
  body, sets `content-type: application/problem+json` and `Retry-After` for
  `rate_limited`, and never serializes `cause`. `problemBodySchema` (Zod) is
  the client-side parse contract. This is the only failure→HTTP translation
  in the codebase (HC-2).
- **Endpoint contracts v2 (`src/transport/endpoint.ts`).** `defineEndpoint`
  declares method, path, request schema (or none), and a per-status response
  map where each status carries a codec: `jsonBody(schema)` (non-transforming),
  `problem(...codes)` (problem body with an optional closed literal `code`
  union), `emptyBody()` (204/205-style), or `textBody()`. Literal statuses are
  preserved through the type system.
- **Closed-union `apiFetch` (`src/transport/api-client.ts`).** New overload
  for v2 endpoints returning a closed union: `{ ok: true; status: S; data }`
  per declared success status; `{ ok: false; kind: 'api'; status: S; error }`
  per declared error status with the exact body type (problem bodies with
  narrowed `code` where declared); `{ ok: false; kind: 'protocol'; status;
  detail }` for undeclared statuses, invalid JSON, wrong content shape, or
  schema drift — identical policy in every environment, never trusted as
  data; `{ ok: false; kind: 'network'; aborted; cause }` for fetch
  rejection/abort. The v1 overloads remain untouched for the 35 unmigrated
  call sites (deleted in .3). No second client is created.
- **Endpoint-bound route responses (`src/transport/api-response.ts`).**
  `apiResponse(endpoint, status, body)` compiles only for the endpoint's
  declared statuses with exactly that status's body type (an `AppFailure` for
  problem statuses — serialized through the single mapper; a schema-typed
  value for JSON; no body argument for empty), and serializes through the
  declared codec.
- **Typed-failure cores.** `readJsonBody(request, schema)` in
  `src/transport/route-body.ts` and `checkRateLimit(request, options)` in
  `src/lib/rate-limit.ts` return `{ ok: true, data } | { ok: false, failure:
  AppFailure }`. The existing `parseJsonBody`/`rateLimitGuard` wrappers are
  re-expressed on the cores with byte-identical wire output for all current
  callers.
- **The feedback slice, migrated end-to-end.** `feedbackEndpoint` becomes a
  v2 contract declaring 204 empty, 400 problem (`invalid_json`,
  `invalid_body`, `message_empty`, `path_invalid`), 429 problem
  (`rate_limited`), 502 problem (`discord_failed`), 503 problem
  (`feedback_unconfigured`); the route builds every response through
  `apiResponse` with unchanged statuses and ordering, and each 400
  failure's `detail` carries today's exact plain-text message (e.g.
  "message must not be empty", "path must start with /") so the modal's
  displayed copy is unchanged; `FeedbackModal` and its `feedback-view.ts`
  message mapper consume the closed union with no visible UX change.

Everything else — 51 routes, 38 v1 endpoints, all guards, the mutation
pipeline, cron shell, Convex, Better Auth — is untouched. Contract §§3–6 are
satisfied for the .1 share described in the restructure note without widening
it; the moved obligations are listed under Scope coverage.

## Resolved implementation decisions

- **Contract PD-1 — failure-set home: `src/lib` (`failure.ts` + `problem.ts`),
  endpoint/transport surfaces in `src/transport`.** `src/lib/rate-limit.ts`
  and `src/lib/service-auth.ts` must eventually construct failures, and the
  lib zone may import only lib, so the failure set and mapper cannot live in
  `src/transport`; `platform/esi` and `convex` (later sessions) can also
  reach lib. The endpoint model, executor, and `apiResponse` are transport
  concerns and transport imports lib. **Rejected:** contract-adjacent homes
  (per-slice) — they would fragment the single vocabulary; a transport-owned
  mapper — unreachable from lib guards in .2.
- **Contract PD-2 — correlation id: minted per problem response via
  `crypto.randomUUID()` (unflagged global WebCrypto since Node 19; this
  stack runs Node 20+ on Vercel and Node 24 locally, and the mint is
  server-side only).** Verified there is no existing request/correlation id anywhere
  in `src/` telemetry to reuse. The id appears in the body (and is available
  to .2 for guard-side telemetry). **Rejected:** reusing a telemetry row id —
  none exists at request scope; adding request-id middleware — out of scope
  and unneeded for the contract's traceability goal.
- **Contract PD-3 — same-origin coverage tracking: no change in .1.**
  `src/app/api/same-origin-coverage.test.ts` pins route inventory and guard
  presence, not body shapes; .1 touches neither the guards nor any
  pipeline/mutation route, and the feedback route's
  `requireSameOrigin(request);` line is preserved verbatim so the 22-route
  inventory pin holds. The problem-body tracking question becomes real in .2
  and is answered there. **Rejected:** pre-emptively rewriting its
  assertions — nothing it asserts changes in .1.
- **Contract PD-4 — representative path: the feedback endpoint.** One route
  exercises validation (400 × invalid JSON, schema-invalid, two
  post-sanitise emptiness/shape checks), `rate_limited` (429),
  `dependency_unavailable` in both flavors (503 unconfigured, 502 Discord
  failed), and an empty-codec 204 success — the widest real status spread of
  any single non-auth route, with public authz (no session fixtures) and
  low product risk. The remaining statuses (401/403/404/409) are proven at
  mapper level in .1 (V-1) and at route level in .2's matrix.
  **Rejected:** `preferences` (only 400/401/200) and `saved-plans` family
  (needs session fixtures; conflict semantics owned by .2's guard work).
- **Contract PD-5 — the raw-`Response` guard gate: deferred to .2 with the
  guard migration.** A gate rejecting `Response`-returning guards cannot pass
  while the guards still legitimately return `Response` (they migrate
  in .2). .1's mechanical protection is narrower: the api-contracts gate
  classifier learns to accept the v2 pattern (`apiResponse` + `readJsonBody`)
  so the migrated feedback route stays green, with a seeded-fixture test
  proving the classifier still fails a contractless route. **Rejected:**
  landing a disabled/allowlisted guard gate now — a dead gate with a
  universal allowlist proves nothing and invites drift.
- **Plan — result-union discriminants: outer `ok`, error arms discriminated
  by `kind: 'api' | 'protocol' | 'network'`, declared arms carrying literal
  `status`.** Matches the brief's narrowing example (`result.ok` then
  `switch (result.status)`) while keeping transport/protocol failures
  structurally impossible to confuse with declared domain errors (brief
  requirement). Abort is distinguished on the network arm via
  `aborted: boolean` derived from `AbortError`. **Rejected:** collapsing
  protocol failures into a synthetic 5xx problem — the brief forbids
  presenting them as trusted domain data.
- **Plan — validation policy: v2 responses are schema-validated in every
  environment; drift → protocol failure; on success the raw parsed JSON is
  returned (typed by the non-transforming schema), preserving wire bytes.**
  This removes the dev-logs/prod-trusts split for migrated endpoints, per
  the brief. The v1 path's dev-only drift logging is untouched until those
  endpoints migrate. **Rejected:** returning Zod output — Zod 4 object
  parsing strips undeclared keys, which silently normalizes wire data.
- **Plan — closed `code` literals where the vocabulary is known, enforced at
  runtime too.** `problem(...codes)` narrows the problem body's `code` to the
  declared literal union per endpoint-status AND runtime-checks membership
  during response decoding: a wire `code` outside the declared union is a
  protocol failure, never returned as typed data (the drift-trust rule from
  the brief). `problem()` with no codes declares an open vocabulary and
  falls back to `code: string` with no membership check. The shared
  `problemBodySchema` keeps `code: string` as the structural layer.
  Feedback declares the five codes listed under Scope. **Rejected:** a
  single global code enum — codes are endpoint-vocabulary, not a repo-wide
  closed set (brief allows per-domain vocabularies).
- **Plan — `apiResponse` status authority: the declared status literal wins,
  with one shared serializer.** `src/lib/problem.ts` exports exactly one
  `serializeProblem(body: ProblemBody): Response` owning the
  `application/problem+json` content-type and `Retry-After` emission;
  `problemResponse(failure)` composes mint + `CATEGORY_STATUS` +
  `serializeProblem`, and `apiResponse(endpoint, status, failure)` composes
  mint + the declared status literal + `serializeProblem`. `apiResponse`
  runtime-asserts that the failure's category-derived status (including the
  502/503 override) equals the declared status and throws on mismatch — a
  programmer error surfaced in tests, not a wire behavior. The
  single-producer proof is that `serializeProblem` is the only code setting
  `application/problem+json`. **Rejected:** letting the mapper's
  category-derived status win — it would silently break the declared-literal
  guarantee `apiResponse` exists to give.

### Interfaces and contracts

- `src/lib/failure.ts`:
  ```ts
  /** Closed application-failure categories fixed by roadmap §3.10.2.1. */
  export type FailureCategory =
    | 'validation' | 'unauthenticated' | 'forbidden' | 'not_found'
    | 'conflict' | 'rate_limited' | 'dependency_unavailable' | 'unexpected';

  /** Typed application failure; deep code returns these, never a Response. */
  export interface AppFailure {
    category: FailureCategory;
    /** Stable application code (telemetry/client contract); defaults to the category. */
    code: string;
    /** Request-derived safe detail only; never internal state. */
    detail?: string;
    retryAfterSeconds?: number;
    /** dependency_unavailable only: 502 upstream-failed vs 503 unavailable. */
    status?: 502 | 503;
    /** For server logs; never serialized. */
    cause?: unknown;
  }
  // Per-category constructors, e.g.:
  export function validationFailure(code?: string, detail?: string): AppFailure;
  export function dependencyUnavailable(code: string, status: 502 | 503, cause?: unknown): AppFailure;
  ```
- `src/lib/problem.ts`:
  ```ts
  /** RFC 9457 problem body served by every mapped failure (HC-3 members only). */
  export const problemBodySchema: z.ZodType<ProblemBody, ProblemBody>;
  export interface ProblemBody {
    type: string;      // https://lgi.tools/problems/<category>
    title: string;     // fixed per category
    status: number;
    code: string;
    correlationId: string;
    detail?: string;
    retryAfterSeconds?: number;
  }
  /** Single delivery-boundary translation (HC-2): failure → problem Response. */
  export function problemResponse(failure: AppFailure): Response;
  /** Pure body builder (mapper tests + apiResponse serialization). */
  export function problemBody(failure: AppFailure, correlationId: string): ProblemBody;
  export const CATEGORY_STATUS: Record<FailureCategory, number>;
  ```
- `src/transport/endpoint.ts`:
  ```ts
  /** Declares one endpoint's method, path, request schema, and full response map. */
  export function defineEndpoint<...>(spec): EndpointContract<...>;
  export function jsonBody<T>(schema: z.ZodType<T, T>): JsonCodec<T>;
  // problem('a', 'b') narrows code to 'a' | 'b' with runtime membership
  // checking on decode; problem() with no codes = open vocabulary, code: string.
  export function problem<const C extends readonly string[]>(
    ...codes: C
  ): ProblemCodec<C extends readonly [] ? string : C[number]>;
  export function emptyBody(): EmptyCodec;
  export function textBody(): TextCodec;
  ```
- `src/transport/api-client.ts` (addition):

  ```ts
  /** Calls a v2 endpoint and returns its closed status-discriminated outcome. */
  export function apiFetch<E extends EndpointContract>(

  ): Promise<OutcomeOf<E>>;
  // OutcomeOf<E> =
  //   | { ok: true;  status: S2xx; data: BodyOf<E, S2xx> }
  //   | { ok: false; kind: 'api'; status: SErr; error: BodyOf<E, SErr> }
  //   | { ok: false; kind: 'protocol'; status: number; detail: string }
  //   | { ok: false; kind: 'network'; aborted: boolean; cause: unknown };
  ```
- `src/transport/api-response.ts`:
  ```ts
  /** Builds the endpoint's declared response for exactly one declared status. */
  export function apiResponse<E, S extends DeclaredStatus<E>>(

  ): Response;
  ```
- `src/transport/route-body.ts` addition: `readJsonBody(request, schema)` →
  `Promise<{ ok: true; data: T } | { ok: false; failure: AppFailure;
  zodError?: z.ZodError }>` — failure codes `invalid_json` / `invalid_body`,
  detail = first issue `path: message` (same string the current 400 emits).
  The typed `zodError` field (present only for `invalid_body`) exists so the
  `parseJsonBody` wrapper's `ParseJsonBodyErrors.invalidBody(error)` override
  path — used by eight production routes — is re-expressed on the core with
  no cast and byte-identical output; it is a core-level field, not an
  `AppFailure` member, and is never serialized.
- `src/lib/rate-limit.ts` addition: `checkRateLimit(request, options)` →
  `{ ok: true } | { ok: false; failure: AppFailure }` with
  `retryAfterSeconds`; `rateLimitGuard` keeps its exact current JSON body.
  The production unconfigured-Upstash throw at `rate-limit.ts:96-98`
  propagates through `checkRateLimit` unchanged (routes keep 500ing there;
  converting it to a 503 failure would be an unauthorized status change).

### Control and data flow

Request path (feedback, after migration): `POST /api/feedback` →
`readJsonBody` (failure → `apiResponse(feedbackEndpoint, 400, failure)`) →
`checkRateLimit` (failure → 429) → sanitise checks (failures → 400) →

### Edge and failure behavior

- Undeclared status reaches a v2 caller → `kind:'protocol'` with the raw
  status; never typed data (all environments).
- Declared JSON/problem status with unparseable or schema-drifted body →
  `kind:'protocol'`; wire bytes never surfaced as trusted data.
- Declared problem status whose wire `code` is outside the endpoint's
  declared literal union → `kind:'protocol'` (drift is never returned typed
  as a declared code).
- Declared empty status (204) with a body → tolerated and ignored (`data:
  undefined`); RFC allows entity-less semantics to be transport-checked
  only.
- `fetch` rejection → `kind:'network'`, `aborted: false`; `AbortError` →
  `aborted: true`. Documented on the overload's TSDoc.
- `unexpected` failure with an `Error` cause → problem body contains no
  message/stack; the cause is available to server logs only (non-leakage
  test seeds a recognizable secret string in the cause and asserts its
  absence from the serialized body).
- `rate_limited` → `Retry-After` header and `retryAfterSeconds` member agree.
- `dependency_unavailable` without an explicit status → 503.
