## Resolved implementation decisions

- **Contract PD-1 — `src/app` zone granularity: a single `app` zone.** Measured: API route handlers and RSC pages share one downward allow-list (`features, data, ui, shared, db, page-settings, config, auth-surface, lib`) and produce 0 violations as one zone. There is no import-direction difference to encode as separate rules; the browser/server distinction is 3.10.1.3's concern (`server-only` markers), not a zone split. **Rejected:** an `app/api` vs `app/pages` split — it adds two zones with identical allow-lists and no enforcement benefit.
- **Contract PD-2 — `convex/` in Fallow or a dedicated import test: a Fallow zone.** `convex/**` classifies cleanly (11 files) and its cross-zone imports are enforced correctly by the boundary rail, so no parallel machinery is needed (satisfies HC-5). The separate "no `next/server` in the convex bundle" constraint (memory: [[convex-bundles-lib-esi-no-next-imports]]) is an *external*-import/bundle concern, not an internal zone edge; it is out of this contract's scope and is a candidate for a later forbidden-call rule or the `server-only` slice. **Rejected:** a dedicated convex import test — it would add parallel boundary machinery for internal edges Fallow already covers.
- **Contract PD-3 — classification of `src/instrumentation*` and `src/proxy.ts`: a dedicated `runtime` zone (`src/proxy*.ts`, `src/instrumentation*.ts`).** These are process-level Next.js entry points (middleware/proxy + observability hooks), distinct in role from route/page composition. The two anchored globs capture exactly the four direct `.ts` children (`proxy.ts`, `proxy.test.ts`, `instrumentation.ts`, `instrumentation.node.ts`) and leave any *future* new top-level `src/*.ts` to surface as unclassified rather than silently joining `runtime`; allow-list `config, data, features, lib` matches their measured imports. **Rejected:** folding them into `app` — it conflates process bootstrap with route composition and muddies the `app` zone's meaning.
- **Contract PD-4 — classification of the nine co-located `src/components/*.ts` files: fold into `shared`.** They are the non-JSX siblings (view-models/hooks) of the shared components already in `shared`; extending the glob from `src/components/*.tsx` to also `src/components/*.ts` classifies them with 0 new violations, and `shared`'s existing allow-list already covers what they import. First-match-wins keeps `src/components/ui/**` in `ui` and `src/components/telemetry/**` in `shared` unchanged (both are subdirectories the `*.ts` glob cannot match). **Rejected:** a separate zone — same architectural role as `shared`, so a new zone adds ceremony without a distinct rule.
- **Contract PD-5 — the Floss/ledger disposition bound and how the remainder is recorded: bound = a single-file deletion or a single import repoint with no behavior change and no cross-slice move; larger → a narrow, documented allow-list exception (never a Fallow waiver/baseline).** The one surfaced violation — `convex/onlineStatusSync.ts` importing `parseOnlineBody` and `canSyncOnline` from `features/online-status` — exceeds the bound (relocating shared logic across slices, touching the feature, convex, and tests), so it is recorded as a narrow `convex → features/online-status` allow-list rule with a justifying `// note`, mirroring the existing `data/npc-stats → data/eve-data` exception and CONTRIBUTING's "narrow composition exceptions declared in the authoritative `.fallowrc.json`." This keeps Fallow green with **no waiver/baseline** (AGENTS.md forbids those for boundary violations) and keeps the debt visible in the config note and the ownership-map table. The helpers are intentional, documented cross-boundary reuse (both files state the Convex action imports them and are kept runtime-light for it). Operator confirmed this disposition (Option A) after reviewing the architecture map. **Rejected:** relocating the two pure helpers to a `data`/`lib` leaf — arguably also behavior-preserving, but it expands this config-only slice into a cross-slice change (feature + convex + tests); recorded as future work.

### Control and data flow

No runtime data flow changes. The Fallow analysis gains two checks: `coverage.requireAllFiles` reports a `boundary_coverage_violation` for any source file matching no zone and no `allowUnmatched` glob, and the new `rules` report `boundary_violations` for disallowed cross-zone imports. The repository gate `pnpm fallow` = `fallow audit --fail-on-issues` with `audit.gate: "new-only"` — it analyzes files **changed vs the merge base** and fails on introduced findings, and its dead-code sub-analysis includes both boundary categories. So a newly-introduced unzoned file (itself a changed file) fails the gate, preserving deny-by-default going forward. The one-time **full-tree** zero-unclassified proof runs via `fallow dead-code --boundary-violations` (not part of `pnpm verify`) and is shown at SC-1.

### Edge and failure behavior

- A future new top-level source directory (unzoned) → as a changed file it surfaces a `boundary_coverage_violation` under `pnpm fallow` (`fallow audit`) and fails CI; SC-2 demonstrates this against the real gate. Named contingency: if `fallow audit` does not surface an introduced coverage violation, wire an explicit `fallow dead-code --boundary-violations` coverage assertion into `verify`/CI.
- Co-located test files inside a zoned directory → swept into their zone by the `**` glob; no separate handling, and boundary rules that permit the production edges also permit the test edges (e.g., a `*.db.test.ts` importing `db/test-support`).
- Non-production tooling (`scripts/**`, root config files, `neon.ts`) → matched by anchored `allowUnmatched` entries; each is enumerated and justified in the plan and PR notes, never a silent glob at arbitrary depth.
- The one real cross-boundary edge (`convex` importing `features/online-status`) → resolved by the documented narrow allow-list exception; the clean relocation is named future work in the config note and ownership table.
- `data/npc-stats` has its own rule (overrides the group `data` rule), so it independently receives the `db/config/search/purge` grants; verified to contribute 0 residual violations.
